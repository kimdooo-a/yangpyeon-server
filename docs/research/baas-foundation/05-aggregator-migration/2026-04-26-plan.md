## 2. Multi-tenant 적응 매트릭스

핸들러는 이미 `withTenant` HOF 로 wrap → `app.tenant_id` 자동 SET. **그러나 cron 진입점은 HTTP 요청이 아니므로 별도 진입 메커니즘 필요.**

| spec 항목 | 현재 (단일 사용자) | 멀티테넌트 적응 |
|----------|------|----------------|
| `import { prisma } from "@/lib/prisma"` (모든 모듈) | `prisma.X.findMany(...)` 직접 호출 | `prismaWithTenant.X.findMany(...)` 또는 `withTenantTx(async (tx) => ...)` |
| runner.ts 진입 (cron context) | 무시 — main thread 호출 | `runWithContext({ traceId, tenantId, ... }, () => runAggregatorModule(payload))` |
| `cron/runner.ts` dispatch | `SQL/FUNCTION/WEBHOOK` 만 분기 | `AGGREGATOR` 분기 추가 — `payload.module` 받아서 `runAggregatorModule({ module })` 호출 |
| `enum CronKind` (schema.prisma) | `{ SQL FUNCTION WEBHOOK }` | `+ AGGREGATOR` 값 |
| dedupe `urlHash` unique 제약 | 단일 글로벌 unique | tenant-scoped (`(tenant_id, url_hash)`) — **이미 schema 에 적용됨** ✅ |
| promote.ts `ingestedItemId` unique | 단일 글로벌 unique | tenant-scoped — **이미 schema 에 적용됨** ✅ |
| classify.ts 슬러그 동기화 | `getAvailableCategorySlugs()` 가 정적 배열 | DB `contentCategory` 시드(37개)와 정확히 동기화 — 본 plan 에서 검증만 |

### 2.1 cron AGGREGATOR 분기 신설 (src/lib/cron/runner.ts)

```ts
// (스케치 — 실제 구현 시 spike-baas-002 worker pool 격리 정책 따름)
async function dispatchAggregatorOnMain(
  payload: Partial<CronKindPayload> & Record<string, unknown>,
  tenantId: string,
  started: number,
): Promise<CronRunResult> {
  const moduleName = typeof payload.module === "string" ? payload.module : null;
  if (!moduleName) return failure(started, "payload.module 누락");
  const result = await runWithContext(
    { traceId: crypto.randomUUID(), tenantId, startedAt: started },
    () => runAggregatorModule({ module: moduleName as AggregatorModule, batch: payload.batch as number }),
  );
  return result;
}
```

이미 S69 4 endpoint 에서 검증 완료 (`src/app/api/v1/t/[tenant]/categories/route.ts:23,74,89`). aggregator 라이브러리 코드도 동일 import:

## 7. 서버 재빌드 / 재기동 필요성 평가 ⭐ (사용자 요청 항목)

### 7.1 재빌드 필요 여부

**결론: 필수.** 다음 변경 모두 컴파일된 standalone 번들에 포함되어야 함:

| 변경 항목 | 빌드 영향 | 사유 |
|----------|----------|------|
| 신규 라이브러리 코드 `src/lib/aggregator/*` (10 파일) | ✅ 필요 | TypeScript → JS 컴파일, standalone 번들 포함 |
| `src/lib/cron/runner.ts` 수정 (AGGREGATOR 분기) | ✅ 필요 | cron worker 가 PM2 프로세스 메모리에 캐시됨 |
| ~~`prisma/schema.prisma` 변경~~ | ❌ 불필요 | enum 이미 적용 완료 (T1.6). Prisma client 재생성도 불필요. |
| `package.json` 의존성 3개 추가 | ✅ 필요 | `npm ci` 단계에서 node_modules 재설치 (이미 standalone 빌드 파이프라인에 포함) |
| 환경변수 추가 | ✅ 필요 | 빌드 자체에는 영향 없으나 `~/ypserver/.env` 갱신 → PM2 reload 필요 |
| cron_jobs 시드 (DB row) | ❌ 불필요 | DB 데이터만 — 빌드 무관 |

### 7.2 재기동 필요 여부

**결론: PM2 restart 1회 필수.** 코드 hot-reload 불가능한 영역 변경:

| 영역 | 메모리 캐시 여부 | 재기동 영향 |
|------|----------------|-----------|
| Prisma client (`@/generated/prisma/client`) | ✅ 글로벌 캐시 (proxy) | enum 자체는 이미 인식 중 (T1.6). 단, 새 라이브러리 코드의 새 호출 패턴이 client 캐시 wire-up 되어야 동작 |
| Cron registry (`globalThis.__cronRegistry`) | ✅ 프로세스 글로벌 | restart 시 `loadAll()` 재실행 → 신규 6 job 인식 |
| ALS storage (request-context) | ✅ AsyncLocalStorage | runtime 재시작 시 ALS 상태 초기화 — 동작 없음 |
| 환경변수 | ✅ `process.env` 캐시 | restart 시 `~/ypserver/.env` 재로드 |

**재기동 명령**: `wsl -- bash -lic 'pm2 restart ypserver'` 또는 `/ypserver` 전체 파이프라인 (rsync → build → pack → deploy → PM2 restart) — T8 단계에 통합.

### 7.3 무중단 가능성

- **Cloudflare Tunnel** 측: 24h+ uptime 유지. PM2 restart 동안 ~2~5초 504 (Tunnel keep-alive 가 아니라 origin 502). 정책상 허용 (배포 윈도우).
- **데이터 영향**: PM2 restart 도중 cron tick (60초 주기) 1회 스킵 가능. 매 6h 주기인 fetch 는 영향 없음. classify/promote 30m 주기도 다음 tick 에서 재시도.
- **세션 사용자**: PM2 restart 시 진행 중 HTTP 요청만 502 응답 (대부분 5xx 자동 retry 됨). 영향 ≪10명/시.
