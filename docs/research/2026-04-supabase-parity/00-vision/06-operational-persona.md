## 도입: 왜 페르소나가 필요한가

"양평 부엌 서버 대시보드"는 특정 사용자(1인 운영자, 소규모 팀, 보안 우선 개인)를 위해 최적화된 도구다. 페르소나 정의 없이는 기능 우선순위가 흔들린다. 이 문서는:

1. **주 페르소나** (김도영, 실제 소유자) — 모든 기능 결정의 1차 기준
2. **부 페르소나 A** (소규모 팀 리드, 가상) — 잠재 확장 시나리오
3. **부 페르소나 B** (보안중시 개인, 가상) — 규제 환경 고려
4. **비페르소나** — 지원하지 않는 사용자 명시 (범위 제한)

를 정의하여 Wave 4 아키텍처 청사진 및 기능 요구사항(FR)의 토대를 구성한다.

```
└─ 복원 소요 시간 기록 (목표 RTO 30분)
```

### 1.4 Pain Points (8개 이상)

#### Pain 1: Supabase Cloud 비용 부담
- **증상**: 다른 프로젝트의 Supabase Pro 요금이 매달 $25~100+ 청구됨
- **빈도**: 매월
- **심각도**: ★★★★★ (재정적 지속성 위협)
- **해결 기준**: 자체호스팅으로 Supabase Cloud 기능의 90%+ 대체 → 월 $25 절감
- **관련 카테고리**: 전체 프로젝트 존재 이유

#### Pain 2: 자체호스팅 Supabase 업그레이드 리스크
- **증상**: Supabase OSS를 직접 Self-host하면 major 버전 업그레이드 시 데이터/설정 손실 위험
- **빈도**: 버전 출시마다 (분기 1~2회)
- **심각도**: ★★★★ (데이터 손실 가능성)
- **해결 기준**: 양평 자체 구현은 의존성이 적어 업그레이드 위험 최소화
- **관련 카테고리**: Operations, DB Ops

#### Pain 3: Next.js 16 App Router 통합 부재
- **증상**: 기존 Supabase 클라이언트가 App Router Server Component와 충돌
- **빈도**: 매 기능 개발 시
- **심각도**: ★★★★★ (개발 생산성 직격)
- **해결 기준**: 자체 auth 미들웨어 + Server Component 완전 호환
- **관련 카테고리**: Auth Core, UX Quality

#### Pain 4: MFA 부재 (현재 15점)
- **증상**: 대시보드 로그인이 패스워드만으로 보호됨
- **빈도**: 매일 (로그인 시마다 잠재적 위험)
- **심각도**: ★★★★ (보안 취약점)
- **사용자 인용**: "서버에 DB가 있는데 2FA도 없다는 게 찜찜하다"
- **해결 기준**: TOTP 앱(Google Authenticator) 또는 WebAuthn 지문/패스키 지원
- **관련 카테고리**: Auth Advanced

#### Pain 5: Edge Functions 부재 (현재 45점)
- **증상**: 비즈니스 로직을 Next.js API Route에 모두 집어넣어 서버 부하 발생
- **빈도**: 새 기능 개발 시마다
- **심각도**: ★★★ (확장성 한계)
- **사용자 인용**: "짧은 작업은 격리된 환경에서 실행하고 싶다"
- **해결 기준**: isolated-vm 기반 JS 샌드박스에서 함수 실행, UI에서 배포
- **관련 카테고리**: Edge Functions

#### Pain 6: Realtime 부족 (현재 55점)
- **증상**: 데이터 변경 시 브라우저 새로고침 필요, 실시간 알림 없음
- **빈도**: 매 작업 세션마다
- **심각도**: ★★★ (DX 저하)
- **사용자 인용**: "Table Editor에서 다른 탭이 자동 업데이트됐으면 좋겠다"
- **해결 기준**: wal2json CDC → Realtime Channel → 브라우저 WebSocket 연결
- **관련 카테고리**: Realtime, Table Editor

#### Pain 7: Audit 이벤트 부재
- **증상**: "언제, 누가, 어떤 데이터를 변경했는지" 추적 불가
- **빈도**: 문제 발생 시 (디버깅 필요 시)
- **심각도**: ★★★★ (사후 분석 불가)
- **사용자 인용**: "실수로 삭제한 데이터가 뭔지 알 방법이 없다"
- **해결 기준**: 모든 CRUD 이벤트를 audit_log 테이블에 기록, UI에서 검색
- **관련 카테고리**: Observability, DB Ops, Auth Core

#### Pain 8: AI 통합 가격 투명성 부족
- **증상**: SQL Editor AI 어시스턴트나 자동완성이 얼마나 비용을 쓰는지 알 수 없음
- **빈도**: AI 기능 사용 시마다
- **심각도**: ★★★ (예산 관리 어려움)
- **사용자 인용**: "AI 기능 쓰다가 갑자기 청구서가 왔으면 어떡하지"
- **해결 기준**: 월별 AI API 사용량 + 비용 대시보드, 예산 한도 설정
- **관련 카테고리**: UX Quality, SQL Editor

#### Pain 9 (추가): 다중 도구 컨텍스트 스위칭
- **증상**: pgAdmin ↔ 터미널 ↔ 브라우저 ↔ Cursor를 오가며 인지 부하 증가
- **빈도**: 매 작업 세션
- **심각도**: ★★★★ (생산성 저하)
- **해결 기준**: 단일 대시보드에서 DB, 파일, 함수, 인증 모두 관리
- **관련 카테고리**: 전체 UX

#### Pain 10 (추가): 백업 복원 자신감 부족
- **증상**: wal-g 백업이 실제로 작동하는지, 복원이 30분 내 가능한지 확신 없음
- **빈도**: 백업 드릴 시 (월 1회)
- **심각도**: ★★★★ (재해 복구 준비 미흡)
- **해결 기준**: 복원 드릴 자동화, 마지막 검증 시간/결과를 대시보드에 표시
- **관련 카테고리**: DB Ops

### 1.5 성공 지표 (김도영 기준)

| 지표 | 현재 | 목표 |
|------|------|------|
| 아침 점검 소요 시간 | 20분 (pgAdmin + 터미널 + 브라우저) | 5분 (대시보드 단일 화면) |
| 월 인프라 비용 | $25~100 (Supabase Cloud) | $7/월 (B2 $2 + AI $5) |
| MFA 적용 여부 | 없음 | TOTP 또는 WebAuthn |
| 백업 복원 자신감 | "아마 될 것 같다" | "30분 내 복원 드릴 매월 성공" |
| AI 기능 비용 투명성 | 없음 | 실시간 비용 대시보드 |
| 컨텍스트 스위칭 횟수 | 평균 15회/세션 | 5회 이하 |

### 3.2 사용 시나리오

**시나리오 1: 규제 감사 대비**
```
상황: 개인정보보호법 감사관이 "개인정보 처리 현황" 자료 요청

├─ Auth UI → 사용자 동의 기록 테이블 표시
└─ 대시보드 → 암호화 상태(at-rest) 설정 화면 캡처 제출
```

**시나리오 2: 시크릿 키 갱신**
```
상황: 연간 보안 점검 → JWT 서명 키 갱신 필요
박서연의 흐름:
├─ Observability (Vault) → MASTER_KEY 회전 UI
├─ 기존 DEK 재암호화 진행 표시
├─ JWKS 엔드포인트 자동 갱신 확인
└─ 세션 강제 종료 (키 갱신 후 재로그인 강제)
```

### 3.4 보안중시 페르소나 → MVP 추가 기능

김도영 MVP(80점) 대비 보안중시 MVP에 추가 필요:

4. **KEK 회전 UI**: 키 만료 알림 + 원클릭 회전 (Observability 95점)
5. **세션 강제 종료**: 보안 사고 시 모든 세션 즉시 무효화 (Auth Core 95점)

**분석**:
- **데이터 주권 + 비용 절감 + 1인 운영**: 세 페르소나 공통 최상위 가치 → 핵심 설계 원칙으로 고정
- **팀 협업 기능**: 김도영에게 낮음 → 부 페르소나 기능으로 분류, 80점 이후 구현
- **AI 통합**: 보안중시에 낮음 → BYOK로 해결 (데이터가 AI 서버로 전송되는 범위 최소화)

## 6. 비페르소나 명시 (Anti-Persona)

비페르소나를 명확히 정의함으로써 **범위 크리프(scope creep)**를 방지하고 1인 운영 최적화에 집중한다.

### Anti-Persona 1: 대규모 SaaS 운영자

| 항목 | 내용 |
|------|------|
| 설명 | Multi-tenancy가 필요한 B2B SaaS, 수천 명의 엔드유저 직접 서빙 |
| 대표 요구 | 테넌트별 DB 격리, 수평 확장(Kubernetes), 다중 리전, 99.99% SLA |
| **지원 않는 이유** | 양평 서버는 단일 노드(WSL2). Multi-tenancy 설계가 없음. Kubernetes 운영은 1인 운영 비용 초과 |
| **대안 권고** | Supabase Cloud, PlanetScale, Neon 등 관리형 서비스 사용 |
| 경계선 | "단일 팀의 단일 프로젝트"는 지원. "여러 고객사의 분리된 DB"는 지원 않음 |

### Anti-Persona 2: DBA 전문가 (고급 Postgres 튜닝 필요)

| 항목 | 내용 |
|------|------|
| 설명 | Postgres 인터널 최적화, 파티셔닝, 복잡한 HA 구성이 필요한 전문 DBA |
| 대표 요구 | pg_bouncer 상세 설정, Patroni HA, 복잡한 쿼리 플랜 분석, 고급 인덱스 전략 |
| **지원 않는 이유** | 이 도구는 Postgres 전문가가 아닌 "풀스택 개발자가 DB를 관리할 때" 최적화됨. 고급 DBA 기능은 pgAdmin 병용 가정 |
| **대안 권고** | pgAdmin, DBeaver, Datagrip 병행 사용 (공존 설계, 대체 아님) |
| 경계선 | "쿼리 실행 + 결과 확인"은 지원. "Buffer pool 수동 튜닝 UI"는 지원 않음 |

### Anti-Persona 3: 모바일 앱 전담 엔지니어 (Realtime 필수)

| 항목 | 내용 |
|------|------|
| 설명 | iOS/Android 앱의 실시간 데이터 동기화를 위해 Realtime을 핵심으로 쓰는 모바일 엔지니어 |
| 대표 요구 | 수천 클라이언트 동시 WebSocket 연결, Presence 정확도 <100ms, 오프라인 동기화 |
| **지원 않는 이유** | 양평 서버 Realtime은 관리 대시보드 내 사용(동시 연결 <50 가정). 모바일 대규모 Realtime은 별도 인프라 필요 |
| **대안 권고** | Supabase Cloud Realtime, Liveblocks, Ably 등 전문 서비스 |
| 경계선 | "대시보드 자체의 실시간 업데이트"는 지원. "앱 사용자 수천 명 실시간"은 지원 않음 |

### Anti-Persona 4: 완전 초보 개발자 (코딩 지식 없음)

| 항목 | 내용 |
|------|------|
| 설명 | SQL 문법, REST API, 환경변수 개념이 없는 완전 비개발자 |
| 대표 요구 | 드래그앤드롭 앱 빌더, 노코드 워크플로우, AI가 모든 설정 대신 해줌 |
| **지원 않는 이유** | 양평 대시보드는 최소 "SQL 기초 + Next.js 이해"를 가진 사람을 대상으로 설계됨 |
| **대안 권고** | Supabase Cloud (관리형), Airtable, Notion Database |
| 경계선 | "AI 어시스턴트로 SQL 작성 도움"은 지원. "SQL 없이 DB 관리"는 지원 않음 |

## 7. 페르소나 활용 가이드 (다음 Wave에서)

### 7.1 Wave 4 아키텍처 청사진 적용

각 아키텍처 결정에서 다음 질문을 적용:

1. **"김도영이 이 기능을 혼자 운영할 수 있는가?"** — 운영 복잡도 상한선
2. **"이준혁 팀이 이 기능으로 더 빠르게 협업할 수 있는가?"** — 팀 기능 검증
3. **"박서연이 이 기능으로 규제 감사를 통과할 수 있는가?"** — 보안/규정 준수 최소선

### 7.2 기능 우선순위 결정 규칙

```
IF 기능이 김도영 Pain Point 직접 해결 → P0 (즉시 구현)
IF 기능이 레이어 1 MVP에 속함 → P0~P1
IF 기능이 팀 리드나 보안 페르소나만의 니즈 → P2~P3
IF 기능이 Anti-Persona 니즈 → 구현 안 함 (범위 제외)
```

### 7.3 User Story 작성 기준 (Wave 3 V2에서 사용)

모든 User Story는 다음 형식으로:

```
As [김도영 | 이준혁 | 박서연],
I want to [동작],
So that [가치/목적].

Acceptance Criteria:
- [ ] 조건 1
- [ ] 조건 2
```

Anti-Persona가 주어(As)인 Story는 작성하지 않는다.
