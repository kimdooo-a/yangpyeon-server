## 0. 목적

이 문서는 양평 부엌 서버 대시보드 프로젝트의 **모든 스파이크 실행에 적용되는 공통 프로토콜**을 정의한다.

스파이크 포트폴리오(01-spike-portfolio.md)에 등록된 SP-010~SP-031 전체와, 향후 추가되는 신규 스파이크에 동일하게 적용된다. 프로토콜을 따르면 실험의 재현성, 결과의 추적성, 아키텍처 반영의 일관성이 보장된다.

## 1. 스파이크 실행 라이프사이클 (5단계)

모든 스파이크는 아래 5단계를 순서대로 완료해야 한다. 단계를 건너뛰는 것은 금지한다.

```
[STAGE 1] 기획
  └─ 질문 정의 + 성공/실패 기준 명시 + 공수 산정 + 의존성 확인

[STAGE 2] 환경 셋업
  └─ 격리 환경 구성 + 필요 패키지 설치 + 기준값 측정

[STAGE 3] 실험 실행
  └─ --max-hours 내 계획된 실험 수행 + 원시 데이터 기록

[STAGE 4] 결과 분석
  └─ 측정값 vs 성공 기준 대조 + Go/No-Go 판정 + 영향 범위 분석

[STAGE 5] 결정 및 반영
  └─ ADR 업데이트 + Blueprint 수정 + README 인덱스 갱신 + 후속 액션 기록
```

### 1.1 Stage 1: 기획

**필수 산출물** (실험 착수 전 작성 완료):
- 핵심 질문 1개 (명확한 Yes/No로 답변 가능)
- 가설 1개 ("X를 했을 때 Y가 발생할 것이다")
- 수치화된 성공 기준 (정량 조건)
- 수치화된 실패 기준 (No-Go 판정 근거)
- 예상 공수 (h 단위)
- 의존 스파이크 확인 (선행 결과가 없으면 착수 불가)

**기획 단계에서 금지되는 것**:
- 성공 기준 없이 실험 착수
- "가능하면 좋겠다" 수준의 막연한 목표
- 이미 완료된 스파이크의 동일 질문 재실험

### 1.2 Stage 2: 환경 셋업

**격리 원칙**:
- 운영 환경(`/etc/luckystyle4u/`, PM2 production 프로세스)과 격리된 임시 환경 사용
- 별도 포트(3001+), 별도 DB 파일(`/tmp/spike-XXX-test.db`), 별도 디렉토리
- 환경 셋업 완료 후 기준값(baseline) 측정 — 아무 변경 없이 현재 상태 측정

**기준값 기록 형식**:
```
환경: WSL2 Ubuntu 24.04 / Node 20.x / PM2 5.x
기준 측정: [항목] = [값] (측정 시각: YYYY-MM-DD HH:MM)
```

### 1.3 Stage 3: 실험 실행

**시간 관리**:
- `/kdyspike --max-hours N` 타이머를 시작 전 설정
- 중간 체크포인트: 공수의 50% 소진 시 진행 상황 점검
- 시간 초과 전 10분: 현재까지 결과를 저장하고 중단 여부 결정

**데이터 수집 원칙**:
- 원시 데이터(raw measurements) 전체 보존 (평균값만 저장 금지)
- 최소 3회 반복 측정 (이상값 제거를 위해)
- 환경 노이즈 요인 기록 (CPU 부하, 네트워크 상태 등)

### 1.4 Stage 4: 결과 분석

**판정 순서**:
1. 모든 성공 기준 항목에 대해 측정값 기재
2. 각 항목 충족 여부 체크 (Go/No-Go)
3. 모든 항목 Go → 최종 판정 "Go"
4. 하나라도 No-Go → 최종 판정 "No-Go" (부분 Go는 예외 조건 명시)
5. 판정 근거 1문장 작성 ("SP-XXX: [이유]로 Go/No-Go 판정")

### 1.5 Stage 5: 결정 및 반영

Go/No-Go 판정 후 72시간 이내:
- 해당 ADR의 §결과 또는 §재검토 트리거 업데이트
- Blueprint §관련 섹션 수정 또는 보완
- `spikes/README.md` 마스터 인덱스에 결과 한 줄 추가
- `01-spike-portfolio.md` §기존 스파이크 표 업데이트
- 후속 스파이크 또는 구현 착수 기록

## 2. kdyspike 스킬 통합 프로토콜

### 2.1 마이크로 스파이크 (30분 이내)

단순 설치 가능 여부, API 존재 여부, 빠른 동작 확인에 사용.

```bash
/kdyspike --micro "질문 (Yes/No로 답 가능)"
```

**사용 조건**:
- 단일 측정 (성능 비교 아님)
- 환경 셋업 < 5분
- 실험 자체 < 20분
- 결과 문서 불필요 (채팅 로그로 충분)

**예시**:
```bash
/kdyspike --micro "WSL2 Ubuntu 24.04에서 isolated-vm v6 npm install 성공하는가?"
/kdyspike --micro "PG 18 wal2json 확장이 호환되는가?"
/kdyspike --micro "Prisma Studio headless embed API가 공식 지원되는가?"
```

### 2.2 풀 스파이크 (1h ~ 8h)

성능 측정, 복잡한 통합 검증, 대용량 부하 테스트에 사용.

```bash
/kdyspike --full "상세한 질문 및 범위" \
  --max-hours N \
  --output "docs/research/spikes/spike-XXX-result.md"
```

**파라미터**:
- `--max-hours N`: 엄격한 시간 상한. 초과 시 자동 중단 (결과 저장 후)
- `--output`: 결과 파일 경로. 항상 지정 필수
- `--env`: (선택) 실험 환경 변수 파일 경로

**예시**:
```bash
/kdyspike --full "SeaweedFS 50GB 부하 테스트 — 업로드 성공율 + 무결성 + B2 오프로드" \
  --max-hours 8 \
  --output "docs/research/spikes/spike-016-seaweedfs-50gb-result.md"
```

### 2.3 마이크로 → 풀 에스컬레이션

마이크로 스파이크 결과에서 추가 검증이 필요하면 즉시 풀 스파이크로 에스컬레이션한다.

```
마이크로 결과: "isolated-vm v6 npm install 성공"
→ 에스컬레이션: /kdyspike --full "isolated-vm v6 cold start + 메모리 격리 전체 검증" --max-hours 4
```

동일 질문으로 재실행 금지 — 마이크로에서 이미 답변된 항목은 풀에서 건너뜀.

## 4. Go/No-Go 판정 기준 표준화

### 4.1 판정 공식

```
판정 = Go IF (모든 성공 기준 AND 조건 충족)
판정 = No-Go IF (성공 기준 중 하나라도 미달)
판정 = 부분 Go IF (핵심 기준은 충족 + 보조 기준 일부 미달 + 보완 조건 명시 가능)
```

**부분 Go 허용 조건**:
1. 미달 항목이 핵심 기능에 영향 없음 (보조적 요건)
2. 보완 조건을 구체적으로 명시 가능 ("X 기능만 비활성화 후 Go")
3. 동일 스파이크 재실행 없이 구현 단계에서 해결 가능

**부분 Go 남용 금지**:
- 핵심 기준 미달 시 부분 Go 선언 불가 (No-Go 처리)
- 보완 조건이 "추후 확인"처럼 모호하면 No-Go 처리

### 4.2 수치화 기준 공식

성능 계열 스파이크에서 공통 적용:

```
기준 A (필수): p95 ≤ 목표값
기준 B (강화): p95 ≤ 목표값 × 0.8 (여유 20%)
기준 C (경보): p95 > 목표값 × 1.5 → ADR 재검토 트리거 즉시 발동

오류율 계열:
기준 A (필수): 오류율 = 0%
기준 B (허용): 오류율 < 0.01%
기준 C (경보): 오류율 > 0.1% → No-Go

처리량 계열:
기준 A (필수): 처리량 ≥ 요건값
기준 B (여유): 처리량 ≥ 요건값 × 1.5
기준 C (경보): 처리량 < 요건값 × 0.5 → No-Go + 아키텍처 재검토
```

### 4.3 회색 지대 처리 규칙

| 상황 | 처리 방법 |
|------|---------|
| 조건 N개 중 N-1개 충족 | 미달 항목 심각도 평가 후 부분 Go 또는 No-Go |
| 측정 환경 노이즈로 결과 불안정 | 3회 재측정 평균값 사용 (동일 스파이크 재실행 허용, 1회 한정) |
| 성공 기준 자체가 잘못 설정됨 | 기획 단계 오류 — Stage 1 재수행 후 새 스파이크 번호 등록 |
| 실험 도중 상위 가정 변경 | 실험 중단 + 상위 가정 재확인 후 새 스파이크로 재착수 |

## 5. 대안 경로 기록 규칙

### 5.1 No-Go 판정 시 필수 기록 항목

No-Go 판정 후 72시간 이내 다음을 기록한다:

1. **대안 기술 선정**: 01-spike-portfolio.md §9 대안 매트릭스에서 대안 경로 선택
2. **대안 스파이크 등록**: 대안 기술 검증이 필요하면 새 SP-XXX 번호 발급하여 01-spike-portfolio.md에 추가
3. **ADR 재검토 착수**: 해당 ADR 상태를 "재검토 중"으로 변경 + 재검토 트리거 항목에 "No-Go 판정"으로 발동 기록
4. **Blueprint 경보 표시**: 해당 Blueprint §관련 섹션 상단에 "No-Go 경보: SP-XXX 결과 참조" 주석 추가

### 5.2 No-Go 결과 파일 처리

No-Go 판정된 스파이크의 결과 파일은 **절대 삭제하지 않는다**. 실패도 지식이므로 영구 보존.

```
docs/research/spikes/spike-XXX-result.md
  → 상단에 "판정: No-Go" 명시
  → 대안 경로 섹션 추가
  → spikes/README.md에 "❌ No-Go" 표기
```

동일 기술을 다시 시도할 이유가 생기면 새 스파이크 번호(SP-YYY)를 발급하여 별도 파일로 기록한다.

### 5.3 ADR 재검토 트리거 업데이트 방법

```markdown
# 02-architecture/01-adr-log.md 수정 예시

### ADR-009: Edge Functions 3층 하이브리드

- **재검토 트리거**:
  1. isolated-vm v6 Node 24 ABI 호환 깨짐 (ASM-5 EWI)
     → [SP-012 No-Go 판정 2026-04-25: Node 22에서 ABI 불호환 확인]
     → 대안: workerd 재검토 착수 (SP-032 예정)
```

## 6. 배치 실행 규칙

### 6.1 병렬 실행 가능 그룹

동일 환경 리소스를 공유하지 않는 스파이크는 병렬 실행 가능하다.

**병렬 실행 가능 조건**:
- 서로 다른 포트/DB 파일 사용
- 실험 결과가 서로 의존하지 않음
- 총 CPU/메모리 사용량이 WSL2 제한(CPU 4코어, RAM 6GB) 내

**병렬 실행 금지 조건**:
- 동일 PostgreSQL 인스턴스 대상 (트랜잭션 충돌)
- 동일 파일 시스템 쓰기 (SeaweedFS 동시 부하)
- 선후 의존성 존재

**우선 세트 병렬 그룹 예시**:
```
그룹 A (1주차 동시 실행 가능):
  SP-014 JWKS 캐시  — 포트 3001, PG 연결 없음
  SP-011 argon2id   — 독립 Node 프로세스
  SP-010 PM2 cluster — 포트 3002, 별도 SQLite

그룹 B (그룹 A 완료 후):
  SP-015 Session 인덱스  — SP-011 의존
  SP-012 isolated-vm v6  — 독립 환경

그룹 C (그룹 A 완료 후):
  SP-013 wal2json 슬롯  — SP-010 의존, PG 공유
  [SP-016은 SP-013과 PG 동시 실행 금지 — 순차 실행]
```

### 6.2 환경 격리 원칙

| 격리 대상 | 격리 방법 |
|---------|---------|
| 포트 | 스파이크별 고정 포트 할당 (SP-010: 3010, SP-011: 3011, ...) |
| DB 파일 | `/tmp/spike-XXX.db` 독립 파일 |
| PG 스키마 | `SET search_path = spike_xxx_test` |
| PM2 앱명 | `spike-xxx-app` 독립 PM2 앱 등록 |
| 환경변수 | `.env.spike.XXX` 파일 분리 (`.gitignore`에 포함) |

**실험 후 정리**:
- 임시 파일, 임시 DB 삭제
- 임시 PM2 앱 `pm2 delete spike-xxx-app`
- PG 임시 스키마 DROP
- 단, 결과 파일(`spike-XXX-result.md`)은 보존

## 7. 스파이크 결과의 ADR/Blueprint 반영 흐름

```
스파이크 Go 판정
    │
    ├─[1단계]─→ ADR 갱신
    │             → §결과 항목에 "SP-XXX 검증 완료" 추가
    │             → §재검토 트리거 해소 항목에 날짜 기록
    │             → 상태: Accepted 유지
    │
    ├─[2단계]─→ Blueprint 수정
    │             → 검증된 설정값 / 실측 수치 반영
    │             → "검증 필요(ASM-N)" 주석 제거 → "SP-XXX 검증 완료" 주석으로 교체
    │
    ├─[3단계]─→ spikes/README.md 갱신
    │             → 테이블에 "✅ Go" + 검증일 + 핵심 결론 추가
    │
    └─[4단계]─→ 01-spike-portfolio.md §기존 스파이크 표 갱신
                  → 상태: "신규 계획" → "완료"

스파이크 No-Go 판정
    │
    ├─[1단계]─→ ADR 재검토 착수
    │             → 상태: Accepted → "재검토 중"
    │             → §재검토 트리거에 "No-Go 판정 YYYY-MM-DD" 기록
    │
    ├─[2단계]─→ 대안 ADR 초안 작성 (ADR-019+ 번호 발급)
    │
    ├─[3단계]─→ Blueprint 경보 표시
    │
    └─[4단계]─→ 대안 스파이크 등록 (01-spike-portfolio.md §신규 행 추가)
```

## 8. 실패한 스파이크의 처리

### 8.1 실패는 지식이다

No-Go 판정된 스파이크는 **폐기하지 않는다**. 실패 이유, 실패 조건, 대안 경로가 모두 귀중한 지식이다.

보존 원칙:
- `spike-XXX-result.md` 파일 영구 보존 (삭제 금지)
- `spikes/README.md`에 "❌ No-Go" 표기 유지
- ADR 재검토 트리거 기록에 No-Go 판정 이력 추가

**No-Go 스파이크 파일 상단 표준 표기**:
```markdown
# Spike-XXX: [제목]

> **판정: ❌ No-Go** (YYYY-MM-DD)
> **대안**: SP-YYY [대안 스파이크명] 착수 예정
> **영향 ADR**: ADR-XXX 재검토 중

---
```

### 8.2 No-Go 후 대안 경로 진입 조건

No-Go 판정 즉시 다음 중 하나를 선택:

| 상황 | 대응 |
|------|------|
| 대안 기술이 명확함 | 대안 스파이크(SP-YYY) 즉시 등록 후 Priority에 따라 착수 |
| 대안이 불명확함 | `/kdyinvestigate`로 추가 리서치 후 대안 기술 선정 |
| 기능 자체 불필요 | 해당 ADR 상태를 "Deprecated" 처리 + 기능 범위에서 제외 |
| ADR 전면 재설계 필요 | `kdywave` 호출로 Wave 5+ 재설계 세션 착수 |

## 9. 시간 초과 처리

### 9.1 --max-hours 도달 시 절차

`--max-hours N` 도달 10분 전 자동 경보 → 다음 중 하나 선택:

**선택 A: 중단 (기본)**
```bash
# 현재까지 결과 저장
spike-save --partial --output "spike-XXX-result-partial.md"

# 중간 결과 상태로 파일 헤더 표기
> **상태: 시간 초과 — 부분 결과** (완료: YYYY-MM-DD HH:MM)
> **완료 실험**: 실험 1, 2 (예정: 실험 3, 4)
```

**선택 B: 시간 연장 (1회 한정)**
- 전체 공수의 50% 이하가 추가로 필요한 경우만 허용
- 연장 이유 명시 ("실험 3이 예상보다 데이터 크기가 커서 +2h 필요")
- 연장 후에도 완료 불가 → 선택 A 강제 적용

### 9.2 중간 결과 보존 규칙

시간 초과로 중단된 스파이크의 중간 결과:
- `spike-XXX-result.md` 파일로 저장 (partial 표시)
- 완료된 실험만 판정 — 미완료 실험은 "미측정"으로 표기
- 완료된 실험 결과로 Go/No-Go 판정이 가능하면 판정 기록
- 불가능하면 "판정 보류 — 재착수 필요" 표기 후 새 스파이크 번호 발급 여부 검토

### 9.3 재착수 허용 조건

동일 스파이크 ID로의 재착수는 **원칙적으로 금지**한다. 단, 다음 조건에서만 허용:

1. 시간 초과로 미완료 실험이 있는 경우 (partial 결과)
2. 환경 노이즈로 측정값 신뢰도가 낮은 경우 (3회 반복 측정 조건)
3. 실험 착수 후 상위 가정이 변경된 경우 (이 경우 새 ID 권장)

재착수 시 결과 파일에 버전 표기:
```
spike-XXX-result.md (v1 — 2026-05-01, partial)
spike-XXX-result.md (v2 — 2026-05-08, final)
```
