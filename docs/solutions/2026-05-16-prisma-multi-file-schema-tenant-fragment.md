## 문제

ADR-024 옵션 D (Hybrid plugin 격리) 의 schema 레이어 현실화 필요. 단일 `prisma/schema.prisma` 에 모든 컨슈머 (Almanac/jobboard/...) 의 모델이 섞여 있으면:
- 컨슈머 추가 시 글로벌 schema 수정 필요 (ADR-022 7원칙 #4 "코드 수정 0줄" 위반)
- 컨슈머 패키지 (`packages/tenant-<id>/`) 의 schema 가 글로벌과 동기화 부재
- prisma generate 가 컨슈머 모델만 따로 빌드 불가

가정상 옵션:
- **(a) 빌드 시점 concat 스크립트** — 글로벌 schema + fragment 들을 단일 파일로 합쳐 generate. Cross-file 모델 참조 (Tenant.contentCategories[] backref) 처리 까다로움
- **(b) Prisma multi-file schema** — 디렉토리 내 모든 .prisma 파일 자동 머지. cross-file 참조 native 지원

## 원인

Prisma v6.6 이하에서는 `prismaSchemaFolder` 가 preview feature 였음. v6.7.0 GA — `prisma.config.ts` 에 `schema: "directory"` 명시만으로 활성화. 본 프로젝트는 Prisma 7.6 이라 GA 활용 가능. context7 로 공식 문서 즉시 확인 (블로그 + changelog + reference).

## 해결

### 1. prisma.config.ts 디렉토리 모드 전환

```ts
// 변경 전: schema: "prisma/schema.prisma" (단일 파일)
export default defineConfig({
  schema: "prisma",  // 디렉토리 — 모든 .prisma 파일 자동 머지
  migrations: { path: "prisma/migrations" },
  datasource: { url: process.env["DATABASE_URL"] },
});
```

### 2. scripts/assemble-schema.mjs — fragment 머지 빌드 스크립트

```js
// ASCII-only — Node 24 Windows console 인코딩 함정 회피
import { readdirSync, readFileSync, writeFileSync, existsSync, statSync } from "node:fs";
import { join, resolve, dirname } from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = dirname(fileURLToPath(import.meta.url));
const REPO_ROOT = resolve(__dirname, "..");
const PACKAGES_DIR = join(REPO_ROOT, "packages");
const PRISMA_DIR = join(REPO_ROOT, "prisma");
const TENANT_PREFIX = "tenant-";

const HEADER = `// AUTO-GENERATED by scripts/assemble-schema.mjs - DO NOT EDIT DIRECTLY.
// Source: packages/{tenant}/prisma/fragment.prisma
// This file is gitignored - regenerated on every build.
`;

function findTenantFragments() {
  if (!existsSync(PACKAGES_DIR)) return [];
  return readdirSync(PACKAGES_DIR)
    .filter(e => e.startsWith(TENANT_PREFIX))
    .map(e => ({
      tenantId: e.slice(TENANT_PREFIX.length),
      fragmentPath: join(PACKAGES_DIR, e, "prisma", "fragment.prisma"),
    }))
    .filter(({ fragmentPath }) => existsSync(fragmentPath) && statSync(fragmentPath).isFile());
}

const fragments = findTenantFragments();
for (const { tenantId, fragmentPath } of fragments) {
  const body = readFileSync(fragmentPath, "utf8");
  const destPath = join(PRISMA_DIR, `${tenantId}.prisma`);
  writeFileSync(destPath, `${HEADER}\n${body}`, "utf8");
}
```

### 3. .gitignore 패턴

```gitignore
# scripts/assemble-schema.mjs build artifact
/prisma/*.prisma
!/prisma/schema.prisma
```

### 4. package.json prebuild hook

```json
{
  "scripts": {
    "build": "next build",
    "prebuild": "node scripts/assemble-schema.mjs",
    "prisma:assemble": "node scripts/assemble-schema.mjs",
    "prisma:generate": "node scripts/assemble-schema.mjs && prisma generate"
  }
}
```

### 5. 검증 워크플로우

```bash
npm run prisma:assemble  # fragment → prisma/<id>.prisma 복사
npx prisma validate      # "The schemas at prisma are valid" (복수형 schemas = multi-file 인식)
npx prisma generate      # 모든 .prisma 파일 자동 머지 후 단일 client 생성
```

## 교훈

1. **Single source of truth = packages**: 컨슈머 자산 (모델 정의) 은 `packages/tenant-<id>/prisma/fragment.prisma` 에 한 번만 작성. `prisma/<id>.prisma` 는 빌드 산출물 (gitignored). 이중 보관 시 동기화 부재 위험.
2. **cross-file 모델 참조 자연 동작**: `Tenant.contentCategories ContentCategory[]` 같은 backref 가 schema.prisma (Tenant 정의) 와 almanac.prisma (ContentCategory 정의) 양쪽에 분산되어도 Prisma 가 자동 머지 → relation 해소.
3. **컨슈머 추가 = 코드 수정 0줄**: `packages/tenant-jobboard/prisma/fragment.prisma` 만 추가하면 다음 `prisma generate` 시 자동 통합. 글로벌 schema.prisma, prisma.config.ts, package.json 변경 0. ADR-022 7원칙 #4 코드 사실화.
4. **prebuild hook 자동화**: `next build` 호출 시 `npm` 이 자동으로 `prebuild` 먼저 실행. 운영자가 assemble 호출을 잊어버리는 함정 차단.
5. **.mjs (ESM Node) + ASCII-only**: tsx/ts-node devDependency 회피 가능. 단 Node 24 Windows console 환경에서 .mjs 의 한국어 식별자/주석이 syntax error 유발 (UTF-8 파일이라도 console code page 충돌). 빌드 스크립트는 항상 ASCII-only 작성.
6. **Tenant 모델 잔존**: Tenant 자체 모델 (id/slug/status) 은 schema.prisma 에 유지 — 본 모델이 모든 컨슈머의 FK 대상이므로 base schema 가 그 위치. 컨슈머 fragment 는 비즈니스 모델만.

## 관련 파일

- `prisma.config.ts` — multi-file 모드 활성화
- `scripts/assemble-schema.mjs` — 머지 빌드 스크립트
- `packages/tenant-almanac/prisma/fragment.prisma` — 첫 적용 사례 (5 Content* model + 3 enum)
- `prisma/schema.prisma` — base schema (Tenant + 글로벌 인프라 모델)
- `.gitignore` — build artifact 제외 패턴

## 적용 가능 시나리오

- 신규 컨슈머 plugin 추가 (`packages/tenant-jobboard/` 등)
- 기존 단일 schema 의 도메인 분리 리팩토링
- Multi-tenant BaaS 의 schema 레이어 격리

## 함정

1. **schema 디렉토리에 .prisma 외 파일 혼재 금지** — Prisma 가 .prisma 만 파싱하지만, 디렉토리 자체에 `seeds/` `migrations-draft/` 등이 있어도 무관 (`prisma/migrations/` 가 표준 위치).
2. **컨슈머 fragment 의 enum 도 함께 이전** — Almanac 사례에서 ContentSourceKind 등 3 enum 이 fragment 에 함께. 글로벌 enum (예: CronKind) 은 base schema 잔존.
3. **runtime override 부재**: prisma.config.ts 에 schema 위치 명시 한 후 `npx prisma migrate dev --schema=...` 같은 CLI 옵션 충돌 가능. CI 에서는 config 가 단일 진실 소스.
