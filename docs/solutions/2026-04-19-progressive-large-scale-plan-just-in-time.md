## 문제

대규모 스코프 (47h, 7 세션 규모) 의 구현 플랜을 한 번에 풀 디테일로 작성하려고 하면:

1. **문서가 과대해진다** — 예: Phase 16 의 경우 47h × 세부 단계 = 약 3000+ 줄 예상
2. **후속 세션 결과로 상세가 무효화된다** — 중간 스파이크(SP-018 등) 가 FAIL 하거나 예상과 다른 결과를 내면, 해당 결과에 기반한 이후 세션의 세부 코드/커맨드가 재작성 필요
3. **실행자 집중 분산** — 플랜 전체를 읽어야 다음 단계 이해 가능, 반복 스캔 비용 증가
4. **리뷰 시간 증가** — 풀 디테일 전체에 대한 self-review + user review 부담

**증상**:
- 플랜 작성 후 첫 1~2 세션 실행만 완료됐는데 이후 세션 세부가 이미 낡아 있음
- "플랜 업데이트" 만을 위한 세션이 필요해짐
- 실행자가 "다음 단계 파악"에 플랜 전체를 재탐색

## 원인

플래닝은 **정보가 모인 후에만** 정확할 수 있다는 원칙을 위반한 것:

- 스파이크/실험의 결과를 알기 전에 그 위에 의존하는 코드를 미리 적으면 추정치
- 이전 세션의 실측/결정이 다음 세션의 세부를 바꿀 수 있는데, 이를 예견해 쓸 수 없음
- "완전한 플랜" 에 대한 강박이 오히려 플랜 품질을 떨어뜨림 (early detail > timely detail)

이는 워터폴 vs 애자일의 고전적 문제의 마이크로 버전이다. 큰 스코프를 풀 디테일로 선행 작성하면 waterfall, just-in-time 확장은 iterative planning.

## 해결

**계층적 플랜 작성 원칙**:

1. **다음 1~2 세션만 풀 디테일** — 코드 블록, 테스트 케이스, 실행 명령, 예상 출력 완비
2. **이후 세션은 Outline** — Task 제목 + 핵심 인터페이스 시그니처 + DOD 체크박스만
3. **각 세션 진입 시 확장** — 직전 세션 결과를 반영하여 해당 세션 outline → 풀 디테일 전환
4. **Scope 컨트롤**: Outline 의 정보는 "다음 세션에서 확장할 때 재사용 가능한 수준" 으로 유지 (공수/파일 경로/의존성)

### Phase 16 적용 예

총 47h / 7 세션 플랜에서:

```
S46 (현재): 설계 3h — 설계 문서 풀 디테일
S47: 스파이크 6h — 플랜 풀 디테일 (코드 + 명령 + 기대 출력)
S48: 16a Vault 8h — 플랜 풀 디테일 (TDD 테스트 + 구현 코드)
S49: 16b 배포 10h — Outline (Task 목록 + 인터페이스 + DOD)
S50: 16c cluster 10h — Outline
S51: 16d UI + 16e JWKS 7h — Outline
S52: 통합 E2E 3h — Outline 체크박스
```

풀 디테일 영역 = 591줄 (S47 + S48)
Outline 영역 = 406줄 (S49~S52)
**총 997줄** (전체 풀 디테일 대비 약 1/3)

### 진입 시 확장 프로토콜

S48 종료 시점에 **반드시 다음을 수행**:
1. S47 스파이크 결과 (SP-017/018/019) 읽기
2. S49 Outline 의 Task 목록이 여전히 유효한지 검증
3. 변경 필요 시 S49 Outline → S49 풀 디테일로 확장 (추가 1~2h 투자)
4. 확장된 S49 플랜으로 다음 세션 착수

## 교훈

- **"쓸 수 있는 정보만 지금 쓴다"** — 플래닝에도 YAGNI 원칙 적용. 후속 세션 결과가 상세를 바꾼다면, 상세를 지금 쓰지 말고 때가 되면 쓴다.
- **Outline 의 최소 품질 기준**: Task 제목 / 파일 경로 / 공수 / 핵심 인터페이스 시그니처 / DOD 체크박스 — 이 5요소만 있으면 "대체 무엇을 만들지" 는 확정되고 "어떻게 만들지" 만 후속 세션으로 이연.
- **리스크 기준 분할**: 스파이크가 결과를 바꿀 가능성이 있는 영역일수록 outline 우선 (SP-018 결과에 따라 16b 설계가 바뀔 수 있음 → 16b는 outline). 독립된 영역(16a Vault 는 node:crypto 단독) 은 풀 디테일 가능.
- **self-review 체크리스트**: Outline 영역은 "type consistency" 와 "spec coverage" 만 검증, "placeholder scan" 은 의도된 계층이므로 통과 기준 완화.
