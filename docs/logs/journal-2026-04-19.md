**SP-014 JWKS 캐시** (소요 1.2h / 목표 3h)
- Phase 0 탐지: 현 시스템 HS256(대칭키), JWKS 엔드포인트 미존재
- 실험: Node v24 + jose@6.2.2 + ES256 키쌍 + mock JWKS endpoint (127.0.0.1:9001)
- 결과 핵심:
  - 캐시 없음: p95 1.340ms
  - cacheMaxAge=180s: **p95 0.189ms, hit rate 99.0%**
  - 키 회전 grace: JWKS가 both 키 서빙해야 성립 (단순 교체 시 ERR_JWKS_NO_MATCHING_KEY)
  - Cloudflare Tunnel RTT p95: 148.7ms (기준 100ms 48% 초과)
- **조건부 Go** — 실효 지연 1.62ms (hit 99%), Workers 캐시 도입 트리거 2건 정의

**SP-015 Session 인덱스** (소요 0.8h / 목표 2h)
- SQLite(better-sqlite3) + PG 16.13 이중 실험
- SQLite 10만 행 insert 536ms, 복합 인덱스 p95 0.053ms
- PG: EXPLAIN Bitmap Index Scan, p95 **48μs**
- **핵심 발견**: `CREATE INDEX ... WHERE expiresAt > NOW()` → ERROR NOW() STABLE → **cleanup job 대안** 채택
- Seq Scan p95 5105μs (인덱스 drop 시) — 106배 저하 → 인덱스 필수성 실증
- 1M extrapolation p95 ≈ 0.065ms (목표 2ms의 30배 여유)
- **Go** · Compound Knowledge 후보: "PG partial index + NOW() 비호환 → cleanup job 전환"

**SP-011 argon2id** (소요 0.6h / 목표 3h)
- `@node-rs/argon2` prebuilt binary 설치 3.3초
- WSL2 Ubuntu 24.04 + Node v24.14.1 확인 (spec 예상 Node 22 초과)
- bcrypt(12) hash/verify p95 ≈ 170ms
- argon2id default p95 19.8ms — **13배** 빠름 (spec 예상 5× 초과)
- 1000 사용자 점진 마이그레이션: 오류 0 / 100% 성공
- **Go** · ADR-022 신규 제안 · 사실관계 수정: 프로젝트 현행은 bcryptjs 아님 (bcrypt@6.0.0 N-API)

**SP-010 PM2 cluster** (소요 1.2h / 목표 4h)
- **치명적 운영 사고**: `pm2 delete all --namespace sp010` 이 namespace 필터를 무시하고 프로덕션 dashboard/cloudflared 삭제 → `pm2 resurrect`로 즉시 복구
- 복구 후 전략 전환: PM2 생략, Node `cluster` 모듈 직접 사용 (내부적으로 동일)
- autocannon 10s × 50 conn:
  - fork: RPS 54,692
  - cluster:4: RPS 76,489 = fork × **1.40** (+39.9%)
- SQLite WAL 4 worker_threads 10s: SQLITE_BUSY **0/1968 = 0.000%**
- advisory lock: PG 공식 보증으로 실증 축약
- **조건부 Go** · `/ypserver` 스킬에 `pm2 delete all` safeguard 보강 권장

**SP-012 isolated-vm v6** (소요 0.7h / 목표 4h)
- isolated-vm@6.1.2 Node v24.14.1 WSL2에서 **1.6초 prebuilt 설치**
- 기본 실행: `1 + 2*3 = 7` ✅
- Cold start 100 iter: p95 0.909ms (목표 50ms의 55배 여유)
- 메모리 격리 32MB: OOM 시 Isolate 자동 dispose, 호스트 안전
- 10s × 1,092 churns/s = 10,927 churns, delta +16MB (정상 부하 환산 0.09MB/10분)
- **Go** · ADR-009 재검토 트리거 1 "v6 Node 24 ABI break" **해소**

### [5] Step 6.4 — 라이브 디버깅 #2: PG TIMESTAMP(3) timezone-naive + JS Date 9시간 오프셋

`Retry-After: 32453s` (약 9시간) 발견. windowMs=60000인데 60s가 아닌 9시간이라니 명백한 시간대 버그.

**논의**: PG `TIMESTAMP(3)` (TIMESTAMPTZ 아님) + Prisma의 JS Date 변환 → JS Date가 UTC 값을 KST로 해석하여 9시간(UTC+9 KST) 오프셋 누적. windowStart - Date.now() 계산이 -9시간 → windowMs - elapsed = 60000 + 32400000 = 32460000ms.

**결론**: 클라이언트 elapsed 계산 폐기. PG가 직접 잔여 ms 계산하도록 변경: `EXTRACT(EPOCH FROM (window_start + INTERVAL - NOW())) * 1000`. NUMERIC → string 직렬화이므로 `parseFloat` 필수. 재배포 후 `Retry-After: 60` 정확.

**원칙**: "시간 계산은 가능한 한 DB 한 곳에 위임 — 클라이언트 시간대 변환이 끼어들면 디버깅 지옥"
