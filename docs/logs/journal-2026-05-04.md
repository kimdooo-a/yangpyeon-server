**논의**:

  - 자연 tick 경로 `runJob` (line 210-268): `dispatchCron` → `prisma.cronJob.update(lastRunAt, lastStatus)` → **`recordResult(jobId, success).catch(() => {})`** (line 229)

  - SUCCESS → `consecutiveFailures=0 + circuitState 전이 + circuitOpenedAt=null + lastSuccessAt=new Date()` 한 트랜잭션

- 결론: ADMIN runNow SUCCESS 후 cf 그대로/last_success_at NULL 유지 — 모니터링 대시보드가 false-negative 신호 표시.

### [8] WSL DB 호출 우회 패턴 — sudo hang vs PGPASSWORD direct

**논의**:
- 초기 검증 시도: `sudo -u postgres psql -d ypserver` → background 분기 + 0 byte 출력. 패턴: sudo 가 password prompt 로 hang.
- DB명도 틀림 — 실제 = `luckystyle4u` (이전 잘못 인지)
- 우회: `PGPASSWORD=<...> psql -h localhost -U postgres -d luckystyle4u` 직접 호출 → 정상 작동
- 테이블명도 정정: `source` → `content_sources` (Prisma 명명 규약), `tenant` → `tenants`, `audit_log` → `audit_logs`

**결론**: WSL DB 직접 호출은 PGPASSWORD 환경변수 + 명시적 -h/-U 패턴으로 통일. sudo 회피.
