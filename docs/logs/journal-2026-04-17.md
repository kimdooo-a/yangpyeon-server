## 토픽 13: curl E2E 자동화 — CSRF 구조 파악

`/api/v1/*`가 CSRF 제외(`proxy.ts` L79, "v1은 Bearer 기반")임을 발견 → 쿠키 세션만 있으면 curl로 CRUD 전부 자동화 가능.

흐름: `POST /api/v1/auth/login {email,password}` → accessToken → `POST /api/auth/login-v2 {accessToken}` → `Set-Cookie: dashboard_session=... Secure; HttpOnly; SameSite=lax` → 쿠키 동봉 후 임의 v1 API 호출.

세션 21 handover가 "dev 서버 curl 통합 테스트 위임"(CSRF 환경 의존)으로 기록한 것은 `/api/auth/login` 레거시 경로(env 비밀번호)만 고려한 판단. v1 경로로 우회하면 CSRF 선행 토큰 불필요.

## 토픽 14: S8a INSERT 1차 실패 — `updated_at` NOT NULL

payload `{values:{id,name,owner_id}}` (keep 기본값 시뮬레이션) → `QUERY_FAILED: null value in column "updated_at" ... violates not-null constraint` / HTTP 500.

`psql \d folders` 직접 조회:
```
created_at | not null | CURRENT_TIMESTAMP  ← OK
updated_at | not null | (DEFAULT 없음)      ← FAIL
```

**근본 원인**: Prisma `@default(now())`는 DB DEFAULT를 생성하지만 `@updatedAt`은 Prisma 클라이언트 전용이라 DB DEFAULT를 만들지 않음. Phase 14b raw SQL INSERT는 `@updatedAt` 메타데이터를 알지 못하므로 NULL 전달 → 제약 위반.

**연쇄 영향**: `RowFormModal`의 3상태 기본값이 "keep"이라 실사용자가 UI "행 추가"를 누르면 500이 뜸 — 세션 21 "E2E 통과" 기록이 UI 경로를 검증하지 못한 사실이 드러남. 세션 21 curl은 updated_at을 payload에 넣었던 것으로 판단(기록 누락).

**권장 수정(Phase 14c 1순위)**: `prisma/schema.prisma`의 모든 `@updatedAt` 필드에 `@default(now())` 병기 → 마이그레이션이 `ALTER TABLE ... SET DEFAULT CURRENT_TIMESTAMP` 생성. 영향 모델 최소 9개(User/Folder/File/SqlQuery/EdgeFunction/Webhook/CronJob/ApiKey/LogDrain).

상세: `docs/solutions/2026-04-17-phase-14b-updated-at-no-db-default.md`.
