### [1] 서버 종료 방법 안내

> **사용자**: "서버 종료 방법 실행은 이 컴퓨터 wsl에서 실행중"

PM2 상태 확인 후 두 프로세스(`cloudflared` id 1, `ypserver` id 2) 가 운영 중임을 식별. 종료 시나리오 4가지 제시:

- A. `pm2 stop ypserver` — 잠시 정지 후 재기동 (배포·디버깅용)
- B. `cloudflared → ypserver` 순서 정지 (외부 트래픽 차단 점검 모드)
- C. `pm2 delete + pm2 save + pm2 unstartup` (영구 제거)
- D. `wsl --shutdown` (WSL 자체 내림)

**원칙**: 외부 트래픽 입구인 cloudflared 먼저 끊고 앱(ypserver) 정지. 재기동은 역순. PM2 nvm PATH 함정으로 `bash -ilc` interactive login 셸 강제.

**결론**: 시나리오 자체는 안내만 — 실제 종료는 사용자 결정 대기.

### [8] 파일박스 1.4GB 전송 진단 — 4단 게이트 모두 막힘

> **사용자**: "파일 박스 확인좀해봐 데이터 들어오고있는지. 1.4GB 파일 다른 컴퓨터에서 전송시킨지 오래되었는데 아직도 전송중이래."
> **사용자**: "크롬에서 stylelucky4u.com 접속 후에 파일박스로 파일 전송시켰음."

먼저 시스템 차원 확인: Wi-Fi RX/TX 5초간 0 B/s, .partial/.crdownload 0개, 1.2~1.6GB 파일 0개. 서비스 식별 단계에서 사용자가 "stylelucky4u.com 접속 후 파일박스" 명시 — 우리 양평 서버 안의 `/api/v1/filebox/files` 라우트로 보낸 것.

**진단**:
1. 파일박스 저장소 `~/filebox/files/` — 2026-04-26 이후 신규 0개 (마지막 120KB)
2. 서버 로그 — filebox 라우트 호출 0건 (최근 11분)
3. PM2 ypserver — restart count 9, 마지막 restart 10:01:18 (Tenant context missing 에러 폭주 후로 추정)

**4단 게이트 분석**:
| 게이트 | 한계 | 1.4GB |
|--------|------|-------|
| ① Cloudflare Tunnel 무료/Pro | request body 100MB | 14× 초과 (가장 먼저 차단) |
| ② `MAX_FILE_SIZE` 코드 상수 | 50MB | 28× 초과 |
| ③ `DEFAULT_STORAGE_LIMIT` quota | 500MB (ADMIN 100GB) | 일반 유저면 2.8× 초과 |
| ④ `request.formData()` 메모리 | PM2 max_memory 512MB | 1.4GB 로딩 시 OOM 크래시 |

**브라우저 progress bar 가 무한 도는 이유**: Cloudflare 가 100MB 도달 시 stream RST → Chrome 자동 재시도 → 0%부터 다시 — 무한 루프.

**권고**:
1. 즉시: 송신 측 전송 취소 + 브라우저 탭 닫기 (어차피 안 들어옴)
2. 1.4GB 이번 전송: SendAnywhere PC 앱 (LAN 직접) / USB 직접 / SMB 공유 / scp over LAN
3. 파일박스 1.4GB 지원: R2 presigned URL + TUS chunked upload + `request.formData()` 제거 — 별도 ADR + spike 필요 (반나절~1일)

**터치하지 않음**: 파일박스 코드 변경 0줄 — 진단만.
