- **UI 범위 축소 논리**: 프롬프트는 인라인 셀 편집 + 폼 모달 + 행 삭제 세 가지 모두 명시. 그러나 DOD 10항목 중 인라인 편집을 명시한 항목 없음. 인라인은 타입별 에디터(체크박스/date/JSON textarea)·Enter 저장·Esc 취소·탭 네비게이션·낙관적 잠금까지 스코프 폭주 위험. 모달 하나를 신규/수정 겸용으로 완성도 있게 구현 → 인라인은 Phase 14c에서 낙관적 잠금과 묶음
- **감사 로그 경로 선택**: `audit-log.ts`(인메모리 500 링버퍼) vs `audit-log-db.ts`(Drizzle SQLite 영속). CRUD는 "누가 언제 어느 테이블 어떻게"라는 법적·운영적 추적이 필요한 이벤트이므로 PM2 재시작 시 소실 가능한 인메모리는 부적절. 이미 `/audit` 페이지와 `audit_logs` 스키마가 존재해 추가 인프라 필요 없음
- **runReadwrite fail-closed 논거**: runReadonly는 롤 부재 시 try/catch로 통과 허용(READ ONLY가 2차 방어). 쓰기에는 동등한 2차 방어 없음. 롤이 없는 환경에서 조용히 슈퍼유저(`postgres`) 권한으로 CRUD 실행 → users 등 민감 테이블 변경 위험. 배포 체크리스트(SQL 스크립트 수동 적용)와 런타임 동작을 강결합해 누락 사고 원천 차단
- **3상태 입력 근거**: 빈 입력=NULL 정책은 PG default 의미 상실 + UPDATE에서 "변경 의도 없는 컬럼"을 NULL로 덮어쓰는 사고 유발. `{action: "set"|"null"|"keep"}`로 세 의미를 명시적으로 분리. 서버는 `keep`인 컬럼을 payload에서 제외 → INSERT는 DB default 적용, UPDATE는 해당 컬럼 SET 절 생략
- **Vitest 미설치 대응**: ADR §5에는 Vitest 단위 테스트 언급했으나 `package.json`에 `vitest` 없음. 계획 실행 단계에서 Vitest 도입은 스코프 크립 — YAGNI. 대신 "TDD by curl" 접근(실패하는 curl 먼저 본 뒤 통과시키기)으로 재해석. ADR §5는 유지하되 계획에서 재조정 주석 명시
