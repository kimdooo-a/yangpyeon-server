## 1. Coolify 개요 및 포지셔닝

### 1.1 제품 정의

Coolify는 오픈소스 셀프호스팅 PaaS(Platform as a Service)로, Heroku, Vercel, Netlify의 기능을 자신의 서버에서 직접 실행할 수 있게 해주는 플랫폼이다. "Your Infrastructure, Your Data, Your Rules"를 슬로건으로 내세우며, 클라우드 벤더 종속 없이 프로덕션급 배포 환경을 구축할 수 있게 한다.

- **라이선스**: Apache 2.0 (Community Edition), 상용 라이선스 (Pro/Teams)
- **기술 스택**: Laravel(PHP) + Alpine.js + Livewire + Tailwind CSS
- **배포 방식**: Docker Compose 기반, 단일 서버에 설치
- **타깃 사용자**: 개인 개발자, 소규모 팀, 셀프호스팅 애호가
- **GitHub Stars**: 35,000+ (2026년 기준)

### 1.2 핵심 가치 제안

| 측면 | 기존 클라우드(Heroku/Vercel) | Coolify |
|------|------------------------------|---------|
| 비용 | 트래픽/리소스에 따라 급증 | 서버 비용 고정 |
| 데이터 주권 | 벤더 서버에 저장 | 자신의 서버 |
| 커스터마이징 | 제한적 | 완전 자유 |
| 오프라인 운영 | 불가 | 가능 |
| 학습 곡선 | 낮음 | 중간 |

## 2. Coolify 대시보드 구조

### 2.1 전체 레이아웃

Coolify의 UI는 전형적인 **좌측 사이드바 + 우측 콘텐츠** 구조를 따른다. 다크 테마가 기본이며, 깔끔한 단색 배경에 카드 기반 정보 표시를 채택하고 있다.

```
┌─────────────────────────────────────────────────────────┐
│  [ Coolify 로고 ]    헤더 (팀 선택, 알림, 프로필)         │
├─────────────┬───────────────────────────────────────────┤
│             │                                           │
│  사이드바    │           메인 콘텐츠 영역                  │
│  ─────────  │                                           │
│  대시보드   │  [프로젝트 카드] [프로젝트 카드] ...        │
│  프로젝트   │                                           │
│  서버       │  서버 상태 요약                             │
│  소스       │                                           │
│  ─────────  │  빠른 배포 버튼                             │
│  설정       │                                           │
│  팀 관리    │                                           │
│             │                                           │
└─────────────┴───────────────────────────────────────────┘
```

### 2.2 메인 대시보드 화면

로그인 직후 표시되는 메인 대시보드는 3개의 핵심 섹션으로 구성된다.

**섹션 1: 프로젝트 목록**
- 각 프로젝트는 카드 형태로 표시
- 카드에는 프로젝트명, 소속 리소스 수, 마지막 배포 시간 표시
- "새 프로젝트 추가" 버튼이 상단에 위치

**섹션 2: 서버 상태 요약**
- 연결된 서버 목록과 각 서버의 현재 상태 (온라인/오프라인)
- CPU·메모리·디스크 사용률 요약 수치 표시
- 서버 클릭 시 해당 서버의 상세 관리 화면으로 이동

**섹션 3: 빠른 액션**
- 새 애플리케이션 배포 버튼
- 데이터베이스 추가 버튼
- 서비스 추가 버튼 (294개 이상의 원클릭 서비스 템플릿 포함)

### 2.3 계층 구조 (핵심 개념)

Coolify는 다음과 같은 4계층 구조로 리소스를 조직한다. 이 계층 구조를 이해하는 것이 Coolify의 UX를 이해하는 핵심이다.

```
서버 (Server)
└── 프로젝트 (Project)
    └── 환경 (Environment)  ← Production / Staging / Dev
        └── 리소스 (Resource)
            ├── 애플리케이션 (App)  ← Git/Docker 기반
            ├── 데이터베이스 (DB)   ← PostgreSQL, MySQL, Redis 등
            └── 서비스 (Service)   ← WordPress, n8n, Plausible 등
```

**예시**: "내 블로그" 프로젝트
- 서버: my-vps (Ubuntu 22.04)
  - 프로젝트: personal-blog
    - 환경: production
      - 리소스: Ghost CMS (App), MySQL (DB), Redis (Service)
    - 환경: staging
      - 리소스: Ghost CMS (App, staging 버전)

## 3. Coolify 핵심 기능 심층 분석

### 3.1 서버 관리 화면

서버 상세 화면은 탭 기반 구조로 구성된다.

**탭 1: 개요 (Overview)**
- 서버 기본 정보: IP 주소, OS 버전, CPU 코어 수, 총 RAM
- 연결 상태: SSH 접속 가능 여부
- 실행 중인 리소스 수 (애플리케이션, 데이터베이스, 서비스)
- 리버스 프록시 상태 (Caddy 또는 Nginx)

**탭 2: 리소스 (Resources)**
- 해당 서버에서 실행 중인 모든 리소스 목록
- 각 리소스의 상태, 타입, 프로젝트 정보 표시
- 정렬 및 필터링 기능

**탭 3: 메트릭 (Metrics) — Sentinel 연동**
- CPU 사용률 시계열 그래프
- 메모리 사용률 시계열 그래프
- 디스크 I/O 그래프
- 네트워크 트래픽 (In/Out)
- 새로고침 주기 설정 가능

**탭 4: SSH 터미널**
- 브라우저 내 웹 터미널
- 직접 서버에 SSH 접속하는 것과 동일한 환경
- 권한 있는 사용자만 접근 가능

**탭 5: 보안**
- Firewall 설정
- SSL 인증서 관리
- SSH 키 관리

### 3.2 Sentinel — 경량 모니터링 에이전트

Coolify의 서버 모니터링은 **Sentinel**이라는 경량 에이전트 컨테이너가 담당한다.

```
[관리 서버 (Coolify)]
       ↑ HTTP push
[모니터링 대상 서버]
└── coolify-sentinel (Docker 컨테이너)
    ├── Linux API로 메트릭 수집 (CPU, RAM, Disk)
    ├── 컨테이너별 리소스 사용량 수집
    └── 주기적으로 Coolify 컨트롤 플레인에 데이터 전송
```

**Pull 방식이 아닌 Push 방식**을 채택한 것이 특징이다. 이는 방화벽 환경에서도 원활히 동작하고, Coolify 서버의 부하를 낮출 수 있는 설계다.

### 3.3 애플리케이션 배포 화면

**배포 소스 선택**
- Git 연동: GitHub, GitLab, Gitea, Bitbucket 지원
- Docker Image: Docker Hub, GHCR, 사설 레지스트리
- Docker Compose: 기존 compose 파일 직접 사용

**배포 설정 UI**
- 브랜치/태그 선택
- 빌드 팩 설정 (Nixpacks, Dockerfile, Docker Compose)
- 헬스체크 URL 및 주기 설정
- 자동 배포 트리거 설정 (Git push → 자동 빌드)

**배포 이력 화면**
- 각 배포의 타임스탬프, 커밋 해시, 상태 (성공/실패/진행 중)
- 빌드 로그 실시간 스트리밍
- 이전 버전으로 롤백 버튼

### 3.4 환경변수 관리 — 두 가지 뷰

Coolify의 환경변수 관리 UI는 사용자 숙련도에 따라 두 가지 뷰를 제공한다.

**일반 뷰 (Normal View)** — 기본값
```
┌─────────────────────────────────────────────────┐
│  환경변수 설정                                    │
│                                                  │
│  [ KEY              ] = [ VALUE            ]    │
│  □ 빌드 변수  □ 멀티라인  □ 리터럴  □ 시크릿      │
│  ─────────────────────────────────────────────  │
│  [ KEY              ] = [ VALUE            ]    │
│  □ 빌드 변수  □ 멀티라인  □ 리터럴  □ 시크릿      │
│                                                  │
│  + 변수 추가                          저장       │
└─────────────────────────────────────────────────┘
```
- 각 변수를 개별 카드로 표시
- 체크박스로 옵션 설정 (빌드 변수 여부, 멀티라인, 시크릿 마킹)
- 시크릿으로 마킹된 값은 저장 후 UI에서 숨김 처리

**개발자 뷰 (Developer View)** — 토글로 전환
```
DATABASE_URL=postgresql://user:pass@localhost:5432/db
REDIS_URL=redis://localhost:6379
NODE_ENV=production
SECRET_KEY=my-secret-key
```
- .env 파일 형식 그대로 편집
- 대량의 변수를 한번에 붙여넣기 가능
- 프로 개발자에게 적합한 빠른 편집 환경

**환경변수 유형**
- **빌드 변수**: Docker 이미지 빌드 중에만 사용
- **런타임 변수**: 컨테이너 실행 시 주입
- **프리뷰 변수**: PR 미리보기 배포에만 적용
- **매직 변수**: 자동 생성 (비밀번호, FQDN 등)

### 3.5 데이터베이스 관리 UI

**지원 데이터베이스**
- PostgreSQL, MySQL, MariaDB, MongoDB, Redis, KeyDB, Dragonfly, ClickHouse

**데이터베이스 상세 화면**
- 연결 정보 (호스트, 포트, 사용자명, 비밀번호) — 마스킹 후 표시
- 연결 문자열 복사 버튼
- 백업 설정 (S3 또는 로컬 스토리지)
- 백업 목록 및 복원 버튼
- 웹 기반 DB 클라이언트 (선택적 통합)

### 3.6 도메인 및 SSL 관리

- 커스텀 도메인 추가/삭제 UI
- Let's Encrypt 자동 SSL 발급 (Caddy 리버스 프록시 내장)
- Wildcard 도메인 지원
- 도메인별 리다이렉트 설정

### 3.7 알림 시스템

- 지원 채널: 이메일, Discord, Slack, Telegram, Microsoft Teams
- 알림 트리거: 배포 성공/실패, 서버 오프라인, 헬스체크 실패
- 알림 채널 테스트 버튼

### 3.8 팀 및 접근 제어

- 팀 생성 및 멤버 초대
- 역할 기반 권한 설정 (Owner, Member)
- API 토큰 관리 (Webhook 연동용)

## 4. Portainer 개요 및 포지셔닝

### 4.1 제품 정의

Portainer는 Docker, Kubernetes, Docker Swarm 환경을 위한 컨테이너 관리 대시보드다. "Making Docker and Kubernetes management easy"를 슬로건으로 내세우며, 복잡한 CLI 명령어 없이 웹 UI만으로 컨테이너 인프라를 관리할 수 있게 한다.

- **라이선스**: zlib (Community Edition), 상용 (Business Edition)
- **기술 스택**: Go (백엔드) + Angular (프론트엔드)
- **배포 방식**: Docker 컨테이너로 실행 (1분 설치)
- **타깃 사용자**: Docker/K8s 운영자, DevOps 팀, NAS 사용자
- **GitHub Stars**: 30,000+ (2026년 기준)
- **에디션**: Community Edition (무료), Business Edition (유료)

### 4.2 Coolify vs Portainer 포지셔닝 차이

| 관점 | Coolify | Portainer |
|------|---------|-----------|
| 주요 역할 | PaaS (배포 플랫폼) | 컨테이너 관리 도구 |
| 주요 관심사 | 코드 → 배포 자동화 | 실행 중인 컨테이너 가시성 |
| 추상화 수준 | 높음 (서비스/프로젝트 단위) | 낮음 (컨테이너/이미지 단위) |
| 대상 OS 의존성 | Docker 필요 | Docker/K8s/Swarm |
| 사용 시나리오 | 새 앱 배포·관리 | 기존 인프라 시각화 |

## 5. Portainer 대시보드 구조

### 5.1 전체 레이아웃

Portainer는 좌측 사이드바 네비게이션 구조를 채택하며, 환경(Environment) 단위로 컨텍스트가 전환되는 구조다.

```
┌─────────────────────────────────────────────────────────┐
│  [ Portainer 로고 ]    헤더 (환경 선택, 사용자, 알림)     │
├─────────────┬───────────────────────────────────────────┤
│             │                                           │
│  사이드바    │           메인 콘텐츠 영역                  │
│  ─────────  │                                           │
│  홈          │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  환경 선택   │  │스택  │ │컨테이너│ │이미지│ │볼륨  │   │
│  ─────────  │  │  3   │ │  12  │ │  8   │ │  5   │   │
│  대시보드   │  └──────┘ └──────┘ └──────┘ └──────┘   │
│  스택       │                                           │
│  컨테이너   │  환경 정보 영역                             │
│  이미지     │  (서버 OS, Docker 버전, CPU/RAM)           │
│  볼륨       │                                           │
│  네트워크   │  최근 이벤트 로그                           │
│  ─────────  │                                           │
│  설정       │                                           │
└─────────────┴───────────────────────────────────────────┘
```

### 5.2 홈 화면 — 환경 목록

Portainer 홈 화면은 관리 중인 모든 환경(엔드포인트)을 카드 형태로 나열한다.

각 환경 카드에 표시되는 정보:
- 환경 이름과 타입 아이콘 (Docker/K8s/Swarm)
- 연결 상태 (온라인/오프라인 뱃지)
- 실행 중인 컨테이너 수
- 스택 수
- CPU·메모리 사용률 미니 게이지
- "Go to environment" 링크 버튼

### 5.3 환경 대시보드 — 타일(Tile) 기반 구성

특정 환경 진입 시 표시되는 대시보드는 **요약 타일** 형태로 핵심 지표를 한눈에 보여준다.

**요약 타일 구성**

| 타일 | 표시 정보 |
|------|-----------|
| Stacks | 배포된 Compose 스택 개수 |
| Services | Docker Swarm 서비스 수 (Swarm 환경만) |
| Containers | 총 컨테이너 수 + 실행 중 / 중지 / 비정상 분류 |
| Images | 이미지 개수 + 총 디스크 사용량 |
| Volumes | 볼륨 개수 |
| Networks | 네트워크 개수 |
| GPU | GPU 개수 (감지된 경우만 표시) |

## 6. Portainer 핵심 기능 심층 분석

### 6.1 컨테이너 목록 화면

컨테이너 목록은 테이블 형식으로 구성된다.

**컬럼 구성**
- 체크박스 (일괄 선택용)
- 이름 (클릭 시 상세 화면)
- 상태 (running/stopped/paused/unhealthy — 색상 코딩)
- 이미지
- 생성 시간
- IP 주소
- 포트 매핑 (호스트포트:컨테이너포트)
- CPU 사용률 %
- 메모리 사용량

**일괄 액션 버튼**
- Start / Stop / Restart / Pause / Kill
- Remove (삭제 전 확인 다이얼로그)

### 6.2 컨테이너 상세 화면 — 탭 기반

특정 컨테이너 선택 시 탭 기반 상세 화면으로 이동한다.

**탭 1: 상세 정보 (Details)**
- 컨테이너 ID (전체/약식)
- 이미지 이름 및 태그
- 생성/시작 시간
- IP 주소 (네트워크별)
- 포트 매핑 테이블
- 마운트된 볼륨 목록
- 환경변수 (마스킹 가능)
- 레이블 목록

**탭 2: 로그 (Logs)**

로그 뷰어는 Portainer의 핵심 기능 중 하나로 다음 옵션을 제공한다.

```
┌─────────────────────────────────────────────────────────┐
│  [ 검색어 입력 ]  [ 날짜 선택 ]  줄 수: [1000 ▼]         │
│  □ 타임스탬프  □ 줄 번호  □ 자동 줄바꿈  [⟳ 자동 갱신]  │
│  ─────────────────────────────────────────────────────  │
│  2026-04-06 10:00:01  Server started on port 3000      │
│  2026-04-06 10:00:02  Connected to database            │
│  2026-04-06 10:00:05  Health check: OK                 │
│  ...                                                    │
│  ─────────────────────────────────────────────────────  │
│  [ 클립보드 복사 ]  [ 파일 다운로드 ]  [ 전체 화면 ]       │
└─────────────────────────────────────────────────────────┘
```

로그 뷰어 세부 기능:
- **검색 및 필터**: 키워드 입력 시 해당 줄만 표시
- **날짜 범위 선택기**: 특정 기간의 로그만 조회
- **줄 수 제한**: 기본 1000줄, 사용자 지정 가능
- **타임스탬프 토글**: 각 로그 항목 앞에 시간 표시/숨김
- **줄 번호 토글**: 라인 번호 표시
- **자동 줄바꿈**: 긴 로그 줄을 화면에 맞게 줄바꿈
- **자동 새로고침**: 수동/자동 모드 선택
- **클립보드 복사**: 현재 표시 로그 전체 복사
- **파일 다운로드**: 로그를 .txt 파일로 저장
- **전체 화면 모드**: 로그 뷰를 전체 화면으로 확대

**탭 3: 통계 (Stats)**

실시간 모니터링 화면으로 다음 데이터를 시각화한다.

| 지표 | 표시 방식 |
|------|-----------|
| CPU 사용률 | 실시간 퍼센트 + 시계열 차트 |
| 메모리 사용량 | 현재 사용량 / 최대값 + 시계열 차트 |
| 네트워크 I/O | RX(수신) / TX(송신) 누적 바이트 |
| 디스크 I/O | 읽기 / 쓰기 바이트 + 속도 |
| 실행 중인 프로세스 | 컨테이너 내부 프로세스 테이블 |

새로고침 빈도를 사용자가 조정 가능하여 실시간 모니터링의 세밀도를 조절할 수 있다.

**탭 4: 콘솔 (Console)**
- 브라우저 내 터미널 에뮬레이터
- 컨테이너 내부에 직접 접속 (docker exec와 동일)
- 쉘 선택: `/bin/bash`, `/bin/sh`, `/bin/ash` 등
- 명령어 실행 및 파일 탐색 가능

**탭 5: 검사 (Inspect)**
- `docker inspect` 출력을 JSON 형식으로 렌더링
- 전체 컨테이너 설정 정보 조회

### 6.3 스택 (Stack) 관리

Docker Compose 기반 멀티컨테이너 애플리케이션을 스택으로 관리한다.

**스택 생성 방법**
- Web 에디터: 브라우저 내 Monaco 에디터로 compose 파일 직접 작성
- Git 연동: GitHub/GitLab 저장소의 compose 파일 사용
- 파일 업로드: 로컬 compose 파일 업로드

**스택 상태 관리**
- 스택 전체 Start / Stop / Remove
- 스택 업데이트 (compose 파일 수정 후 재배포)
- 환경변수 편집 (.env 파일 형식)

### 6.4 이미지 관리

- 이미지 목록: 이름, 태그, 크기, 생성 시간 표시
- 이미지 Pull: 레지스트리에서 이미지 가져오기
- 이미지 Build: Dockerfile 기반 빌드
- 이미지 Push: 레지스트리에 업로드
- 미사용 이미지 일괄 삭제 (dangling images)

### 6.5 볼륨 및 네트워크 관리

**볼륨 관리**
- 볼륨 목록: 이름, 드라이버, 마운트 포인트, 생성 시간
- 볼륨 생성/삭제
- 미사용 볼륨 정리
- NFS/CIFS 볼륨 설정

**네트워크 관리**
- 네트워크 목록: 이름, 드라이버, 스코프, 연결된 컨테이너 수
- 네트워크 생성 (Bridge/Overlay/Macvlan)
- 컨테이너-네트워크 연결/해제

### 6.6 엔드포인트(환경) 관리

Portainer는 여러 서버를 단일 UI에서 관리할 수 있는 멀티 환경 기능을 제공한다.

**연결 방식**
- **Portainer Agent**: 대상 서버에 Agent 컨테이너 설치, 원격 관리
- **Docker Socket**: 로컬 Docker 소켓 직접 연결
- **TCP**: Docker 데몬의 TCP 포트 연결 (보안 주의)

**환경 그룹 및 태그**
- 환경들을 그룹으로 묶어 일괄 정책 적용
- 태그로 분류 (production, staging, edge 등)
- 권한 정책: 특정 팀이 특정 환경만 접근 가능

### 6.7 역할 기반 접근 제어 (RBAC) — Business Edition

| 역할 | 권한 |
|------|------|
| Administrator | 전체 권한 |
| Environment Administrator | 해당 환경 전체 관리 |
| Operator | 컨테이너 시작/중지, 로그 조회 |
| Helpdesk | 조회 전용 |
| Standard User | 제한된 리소스만 접근 |
| Read-only User | 읽기 전용 |

## 7. 양평 부엌 프로젝트와의 유사점

### 7.1 공통 운영 환경

**양평 부엌 서버 대시보드**와 Coolify·Portainer는 다음과 같은 공통 환경 제약을 공유한다.

| 요소 | 양평 부엌 | Coolify/Portainer |
|------|-----------|-------------------|
| 운영 규모 | 1인 운영 | 주 타깃: 개인/소규모 |
| 호스팅 | 셀프호스팅 (WSL2 Ubuntu) | 셀프호스팅 VPS/베어메탈 |
| 네트워크 | Cloudflare Tunnel | Caddy/Nginx 리버스 프록시 |
| 관리 방식 | PM2 기반 Node.js 프로세스 | Docker 컨테이너 기반 |
| 목적 | 서버 상태 가시성 | 서버 리소스 가시성 + 배포 |
| 접근성 | 웹 대시보드 | 웹 대시보드 |

### 7.2 이미 유사하게 구현된 기능

현재 양평 부엌 대시보드가 이미 Coolify/Portainer와 유사하게 제공하는 기능:

1. **서버 리소스 모니터링**: CPU, 메모리 실시간 표시 (Coolify Sentinel 방식과 유사)
2. **프로세스 관리**: PM2 프로세스 목록 및 상태 표시 (Portainer 컨테이너 목록과 유사)
3. **로그 뷰어**: 실시간 로그 스트리밍 페이지 (Portainer 로그 뷰어와 유사)
4. **네트워크 모니터링**: 포트 및 연결 상태 표시

### 7.3 아직 없는 기능 (갭 분석)

Coolify/Portainer가 제공하지만 양평 부엌이 아직 없는 기능:

1. **프로젝트/서비스 그룹화**: 리소스를 논리적 프로젝트로 묶는 개념 없음
2. **환경변수 관리 UI**: .env 파일 관리 인터페이스 없음
3. **배포 이력**: 배포 타임라인과 롤백 기능 없음
4. **알림 시스템**: 이상 감지 시 알림 발송 기능 없음
5. **멀티 환경**: staging/production 구분 관리 없음
6. **도메인/SSL 관리**: Cloudflare 설정 통합 UI 없음

## 8. 차용 가능한 핵심 UX 패턴

### 8.1 Coolify에서 차용할 패턴

**패턴 1: 프로젝트-환경-리소스 계층 구조**
- 현재 플랫 구조(단일 서버)에서 논리적 그룹화로 확장
- PM2 프로세스들을 "서비스" 단위로 그룹화
- 예: `luckystyle4u` 프로젝트 → `production` 환경 → `next-app`, `api-server` 서비스

```typescript
// 적용 예시: 서비스 그룹 개념
interface ServiceGroup {
  id: string;
  name: string;           // "luckystyle4u-frontend"
  type: 'pm2' | 'docker'; // PM2 프로세스 또는 도커
  environment: 'production' | 'staging';
  resources: PM2Process[];
}
```

**패턴 2: 두 가지 환경변수 편집 뷰 (Normal/Developer)**
- 일반 사용자를 위한 폼 기반 UI
- 개발자를 위한 .env 형식 텍스트 편집기
- 토글 버튼으로 즉시 전환
- 시크릿 값 마스킹 처리

**패턴 3: Push 기반 메트릭 수집 (Sentinel 방식)**
- 현재 대시보드가 서버에서 메트릭을 Pull하는 방식이라면
- Sentinel처럼 경량 에이전트가 주기적으로 Push하는 방식으로 전환 검토
- 네트워크 장애 시에도 과거 데이터 보존 가능

**패턴 4: 배포 이력 타임라인**
- 각 서비스의 배포 이력을 시계열로 시각화
- 커밋 해시, 시간, 상태를 타임라인 뷰로 표시
- 메트릭 그래프에 배포 시점 수직선(vertical marker) 오버레이

**패턴 5: 원클릭 서비스 템플릿**
- Coolify의 294개 서비스 템플릿처럼
- 자주 쓰는 서버 설정(PM2 프로세스 추가, Nginx 가상호스트 등)을 템플릿화
- 양평 부엌 환경에 맞는 프리셋 제공

**패턴 6: 알림 채널 통합**
- 서버 장애, 프로세스 크래시, 디스크 임계값 초과 시
- Discord/Telegram/이메일로 자동 알림
- 알림 규칙 UI (조건 + 채널 선택)

### 8.2 Portainer에서 차용할 패턴

**패턴 7: 요약 타일 (Summary Tiles) 대시보드**
- 홈 화면에 숫자 타일로 핵심 지표 한눈에 표시
- 현재 상태를 색상으로 즉시 파악 (초록/노랑/빨강)
- 타일 클릭 → 상세 목록으로 드릴다운

```
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ 프로세스  │ │ CPU 평균 │ │ 메모리   │ │ 디스크   │
│   12     │ │  23.5%  │ │  67.2%  │ │  45.1%  │
│ 실행 중  │ │ (정상)   │ │ (주의)  │ │ (정상)  │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
```

**패턴 8: 로그 뷰어 UX 개선**
- 검색/필터 + 타임스탬프 토글 + 자동 새로고침
- 로그 파일 다운로드 기능
- 전체 화면 모드
- 날짜 범위 선택기

현재 양평 부엌 로그 페이지가 기본 스트리밍만 제공한다면, 위 기능들로 강화 가능하다.

**패턴 9: 상세 화면 탭 구조**
- 프로세스/서비스 클릭 시 탭 기반 상세 화면
- 개요 / 로그 / 메트릭 / 환경변수 / 설정 탭
- Portainer의 Details/Logs/Stats/Console/Inspect 구조 참고

**패턴 10: 인라인 터미널 (Web Console)**
- 브라우저 내 터미널 에뮬레이터 통합
- PM2 관리 명령어를 GUI가 아닌 터미널로도 실행 가능
- 비상 시 직접 명령어 실행 가능한 탈출구 제공

**패턴 11: 멀티 환경 지원 구조**
- Portainer의 엔드포인트 개념처럼
- 향후 서버가 추가될 경우를 대비한 멀티서버 관리 구조
- 환경 카드 목록 → 환경 선택 → 해당 환경의 리소스 관리

**패턴 12: 볼륨/네트워크 시각화**
- 현재 없는 기능: 어떤 볼륨이 어디 마운트됐는지 시각화
- 네트워크 포트 사용 현황을 테이블로 표시
- 포트 충돌 탐지 및 경고

### 8.3 우선순위별 적용 로드맵

**즉시 적용 (Wave 1 — 현재 단계)**
- 요약 타일 대시보드 강화 (패턴 7)
- 로그 뷰어 기능 개선 (패턴 8)
- 프로세스 상세 탭 구조 (패턴 9)

**단기 적용 (Wave 2)**
- 환경변수 관리 UI (패턴 2)
- 알림 시스템 기초 (패턴 6)
- 배포 이력 타임라인 (패턴 4)

**중장기 적용 (Wave 3+)**
- 프로젝트-환경-리소스 계층 구조 (패턴 1)
- 서비스 템플릿 시스템 (패턴 5)
- 멀티 환경 지원 (패턴 11)
- 인라인 터미널 (패턴 10)

## 9. 종합 평가 및 우선순위

### 9.1 Coolify의 핵심 교훈

1. **계층 구조가 UX의 뼈대**: 서버 → 프로젝트 → 환경 → 리소스 계층은 복잡한 인프라를 이해 가능한 메탈 모델로 만들어준다.

2. **두 가지 유저를 동시에 수용**: Normal View / Developer View처럼 숙련도에 따른 UI 분기를 제공하면 초보자와 전문가 모두 만족시킬 수 있다.

3. **Push 기반 에이전트 아키텍처**: Sentinel의 Push 방식은 방화벽 뒤에 있는 서버도 모니터링 가능하게 하며, 확장성이 높다.

4. **즉시 가치 제공**: 배포 직후 접속 URL을 바로 보여주고, 알림으로 상태를 실시간 공유하는 UX가 사용자 신뢰를 높인다.

### 9.2 Portainer의 핵심 교훈

1. **타일 기반 홈 화면**: 숫자+색상으로 전체 상태를 1초 만에 파악 가능한 요약 화면이 운영자에게 매우 유용하다.

2. **로그 뷰어는 진지하게**: 단순 출력이 아니라 검색, 필터, 다운로드, 전체화면 등 완성도 높은 로그 뷰어가 실무 운영에서 핵심 도구가 된다.

3. **탭 기반 상세 화면**: 하나의 리소스(컨테이너/서비스)에 대한 모든 정보를 탭으로 깔끔하게 구조화하면 컨텍스트 전환 없이 필요한 정보에 빠르게 접근할 수 있다.

4. **비전문가도 사용 가능한 GUI**: `docker exec`, `docker logs`, `docker stats` 같은 CLI 명령어를 GUI로 제공하면 비전문가도 기본 운영 작업을 수행할 수 있다.

### 9.3 양평 부엌 대시보드에 대한 시사점

양평 부엌 서버 대시보드는 **이미 Portainer의 기본 수준을 구현**하고 있다 (CPU/메모리 모니터링, 프로세스 목록, 로그 뷰어). 다음 진화 방향은:

1. **Portainer 수준 완성**: 로그 뷰어 고도화, 프로세스 상세 탭 구조, 타일 대시보드
2. **Coolify 수준 확장**: 프로젝트 그룹화, 환경변수 관리, 배포 이력, 알림 시스템
3. **독자적 가치**: PM2 특화 기능, WSL2/Ubuntu 환경 최적화, Cloudflare Tunnel 통합

이 순서로 진화하면 Portainer의 "가시성"과 Coolify의 "배포 자동화"를 결합한 1인 서버 운영자를 위한 최적의 플랫폼으로 성장할 수 있다.
