# 공통 대시보드 UX 패턴 추출

> Wave 1 리서치 문서 05  
> 작성일: 2026-04-06  
> 대상 플랫폼: Supabase Studio, Firebase Console, Coolify, Portainer, Railway, Render

---

## 개요

관리형 플랫폼 대시보드 6종(Supabase Studio, Firebase Console, Coolify, Portainer, Railway, Render)을 분석하여 공통적으로 반복 등장하는 UX 패턴을 추출하고 분류한 문서입니다. 각 패턴은 이름, 설명, 사용 플랫폼 예시, 구현 복잡도(S/M/L), 추천도(⭐1–5)로 정리합니다.

---

## 1. 네비게이션 패턴

네비게이션은 대시보드의 뼈대입니다. 사용자가 현재 어디에 있고 어디로 갈 수 있는지를 명확히 전달하는 패턴 모음입니다.

---

### 1-1. 고정형 왼쪽 사이드바 (Fixed Left Sidebar)

**설명**  
화면 왼쪽에 항상 표시되는 수직 내비게이션 패널입니다. 아이콘 + 레이블 조합으로 섹션을 구분하며, 현재 활성 메뉴 항목은 배경색 또는 좌측 인디케이터(accent bar)로 강조됩니다. 상단에는 프로젝트/조직 셀렉터가 위치하고, 하단에는 설정 및 사용자 프로필 링크가 자리잡는 것이 표준입니다.

**사용 플랫폼**
- Supabase Studio: Table Editor / SQL Editor / Auth / Storage / Functions / Realtime / Reports를 좌측 아이콘 사이드바로 구성
- Firebase Console: Firestore / Auth / Storage / Hosting / Functions / Analytics를 카테고리 그룹으로 분류
- Coolify: Servers / Projects / Settings를 주요 섹션으로 배치
- Portainer: Home / Environments / Stacks / Containers 등을 계층적으로 나열
- Railway: 프로젝트 내 Services / Deployments / Metrics / Settings
- Render: Dashboard / Services / Databases / Environment Groups

**구현 복잡도**: S (간단)  
기본적인 `<nav>` + CSS flexbox/grid로 구현 가능. 반응형 처리와 collapse 기능 추가 시 M 수준.

**추천도**: ⭐⭐⭐⭐⭐  
모든 플랫폼이 채택. 관리 대시보드의 사실상 표준. 현재 프로젝트도 이미 채택 중.

**현재 프로젝트 적용 여부**: 적용됨 (`src/components/layout/sidebar.tsx`)

---

### 1-2. 접이식 사이드바 (Collapsible Sidebar)

**설명**  
사이드바를 접어서 아이콘만 보이는 "아이콘 모드"와 아이콘+레이블이 보이는 "확장 모드" 사이를 토글하는 패턴입니다. 넓은 화면에서 작업 공간을 최대화하거나, 좁은 화면에서 공간을 절약할 때 사용됩니다. 토글 버튼은 일반적으로 사이드바 상단 또는 중간 하단에 배치됩니다.

**사용 플랫폼**
- Supabase Studio: 사이드바 하단의 chevron 버튼으로 collapse/expand
- Portainer: 좌측 사이드바 토글 버튼 제공
- Coolify: 사이드바 상단 토글 지원

**구현 복잡도**: S  
CSS transition + React state로 간단하게 구현. 로컬스토리지에 상태 저장 시 약간 추가.

**추천도**: ⭐⭐⭐⭐  
사용성 개선 효과가 크고 구현이 간단. 현재 프로젝트에 미적용이므로 Quick Win 대상.

**현재 프로젝트 적용 여부**: 미적용

---

### 1-3. 프로젝트/환경 컨텍스트 스위처 (Context Switcher)

**설명**  
사이드바 최상단 또는 헤더에 위치하는 드롭다운/팝오버입니다. 현재 활성 프로젝트, 조직, 환경(dev/staging/prod)을 표시하고, 클릭하면 다른 컨텍스트로 전환할 수 있습니다. 대규모 플랫폼일수록 이 패턴의 중요도가 높아집니다.

**사용 플랫폼**
- Supabase Studio: 상단 프로젝트 선택 드롭다운 (개인 프로젝트 / 조직 프로젝트)
- Firebase Console: 상단 프로젝트 선택 팝오버 (Firebase 프로젝트 목록)
- Coolify: 서버 선택 드롭다운
- Railway: 프로젝트 캔버스 상단 환경 선택 (Production / Staging / PR Environments)

**구현 복잡도**: M  
단일 서버 환경에서는 불필요할 수 있으나, 멀티 환경 지원 시 필요. 드롭다운 + 전역 상태 관리 필요.

**추천도**: ⭐⭐⭐  
단일 서버 환경인 현재 프로젝트에서는 당장 필요 없음. 미래 확장 시 고려.

**현재 프로젝트 적용 여부**: 해당 없음 (단일 서버)

---

### 1-4. 브레드크럼 네비게이션 (Breadcrumb Navigation)

**설명**  
현재 페이지의 계층적 위치를 "/" 구분자로 나열하는 패턴입니다. "Dashboard / Processes / detail" 형태로 표시하며, 각 항목은 클릭 가능한 링크입니다. 깊은 계층 구조를 가진 페이지에서 사용자 방향 감각을 유지시켜 줍니다.

**사용 플랫폼**
- Supabase Studio: Storage 버킷 내 폴더 탐색 시 브레드크럼 표시
- Firebase Console: 데이터베이스 컬렉션/문서 계층 탐색 시 적극 활용
- Portainer: 컨테이너 > 로그 / 컨테이너 > 통계 등 서브 페이지에서 사용
- Railway: 프로젝트 > 서비스 > 배포 계층

**구현 복잡도**: S  
Next.js의 `usePathname`으로 동적 브레드크럼 생성 가능.

**추천도**: ⭐⭐⭐⭐  
특히 /processes/[id] 같은 상세 페이지가 생길 경우 필수적. 현재 프로젝트에 미적용.

**현재 프로젝트 적용 여부**: 미적용

---

### 1-5. 탭/세그먼트 컨트롤 (Tab / Segment Control)

**설명**  
단일 페이지 내에서 관련된 여러 뷰를 탭 또는 세그먼트 버튼으로 전환하는 패턴입니다. 페이지 전환 없이 콘텐츠 영역만 교체되므로 사용자 흐름을 유지합니다. 수평 탭(상단), 수직 탭(좌측), 세그먼트 버튼(pill 형태) 세 가지 변형이 있습니다.

**사용 플랫폼**
- Supabase Studio: SQL Editor 내 "Query / Results / Explain" 탭
- Supabase Studio: Auth 페이지 내 "Users / Policies / Configuration" 탭
- Firebase Console: 각 서비스 내 주요 섹션 탭
- Portainer: 컨테이너 상세 내 "Logs / Inspect / Stats / Exec / Attach" 탭
- Render: 서비스 상세 내 "Overview / Logs / Metrics / Events / Environment" 탭

**구현 복잡도**: S  
Tailwind CSS + React state로 간단하게 구현. URL 동기화 추가 시 M 수준.

**추천도**: ⭐⭐⭐⭐⭐  
프로세스 상세 모달, 로그 뷰어 등 현재 프로젝트에서도 즉시 활용 가능. 현재 일부 적용.

**현재 프로젝트 적용 여부**: 부분 적용 (processes 상세 모달)

---

### 1-6. 커맨드 팔레트 (Command Palette)

**설명**  
`Cmd+K` (Mac) 또는 `Ctrl+K` (Windows)로 열리는 전역 퀵 액션 검색창입니다. 페이지 이동, 액션 실행, 데이터 검색을 단일 인터페이스에서 처리합니다. fuzzy search를 지원하며, 최근 사용 항목, 즐겨찾기, 카테고리별 분류를 제공합니다. 숙련 사용자의 생산성을 크게 향상시킵니다.

**사용 플랫폼**
- Supabase Studio: Cmd+K로 전체 커맨드 팔레트 오픈. 페이지 이동 + SQL 쿼리 생성 + 설정 변경 등 통합
- Linear, Vercel 등 현대적 개발 도구의 표준 패턴으로 자리잡음

**구현 복잡도**: M  
`cmdk` 라이브러리(shadcn/ui의 Command 컴포넌트 기반)로 구현 가능. 정의된 액션 목록과 글로벌 키보드 이벤트 핸들러 필요.

**추천도**: ⭐⭐⭐⭐  
사용자 경험을 크게 향상. 구현 난이도 대비 효과가 뛰어남. 중기 목표로 권장.

**현재 프로젝트 적용 여부**: 미적용

## 2. 데이터 표시 패턴

데이터를 어떻게 시각화하고 탐색하게 할 것인지에 관한 패턴입니다.

---

### 2-1. 정렬/필터 가능한 데이터 테이블 (Sortable & Filterable Data Table)

**설명**  
컬럼 헤더 클릭으로 정렬, 헤더 아래 인라인 필터 입력, 행 선택 체크박스, 페이지네이션 또는 무한 스크롤을 갖춘 완전한 데이터 테이블입니다. 각 행에는 액션 버튼(편집/삭제/상세 보기)이 있습니다. 컬럼 너비 조절 및 숨기기 기능이 제공되기도 합니다.

**사용 플랫폼**
- Supabase Studio: Table Editor의 핵심 UI. 스프레드시트 느낌의 셀 편집 포함
- Firebase Console: Firestore 문서 목록 테이블
- Portainer: 컨테이너/이미지/볼륨 목록 테이블
- Railway: 배포 이력 테이블
- Render: 서비스 목록

**구현 복잡도**: M  
TanStack Table(React Table v8)을 사용하면 정렬/필터/페이지네이션을 빠르게 구현 가능. 인라인 편집까지 포함하면 L.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /processes 페이지에서 PM2 프로세스 목록에 이미 테이블 형태 사용. 정렬/필터 기능 강화 권장.

**현재 프로젝트 적용 여부**: 부분 적용 (정렬/필터 미구현)

---

### 2-2. 메트릭 카드 그리드 (Metric Card Grid)

**설명**  
핵심 지표를 카드 형태로 배치하는 패턴입니다. 각 카드에는 지표명, 현재 값, 변화량(전일 대비 %), 미니 차트(spark line)가 포함됩니다. 그리드 레이아웃(보통 3–4열)으로 배치되어 한눈에 시스템 상태를 파악하게 합니다.

**사용 플랫폼**
- Supabase Studio: Reports 페이지의 API 요청 수, DB 연결 수, 캐시 히트율 카드
- Firebase Console: Analytics 개요 페이지
- Railway: 프로젝트 개요의 메모리/CPU/네트워크 사용량 카드
- Render: 서비스 메트릭 카드

**구현 복잡도**: S  
현재 프로젝트에서 이미 구현됨. 미니 차트 추가 시 S~M 수준.

**추천도**: ⭐⭐⭐⭐⭐  
현재 / (홈) 페이지에서 이미 CPU/메모리/디스크 카드 + 미니 차트 구현. 핵심 패턴 적용 완료.

**현재 프로젝트 적용 여부**: 적용됨 (`src/components/dashboard/stat-card.tsx`, `mini-chart.tsx`)

---

### 2-3. 트리/계층 뷰 (Tree / Hierarchical View)

**설명**  
부모-자식 관계가 있는 데이터를 들여쓰기와 expand/collapse 화살표로 표시하는 패턴입니다. 파일 시스템, DB 스키마, 컨테이너 계층 등 계층 구조가 명확한 데이터에 적합합니다.

**사용 플랫폼**
- Supabase Studio: Schema Visualizer에서 테이블 관계 트리 표시
- Firebase Console: Firestore 컬렉션 > 문서 > 서브컬렉션 트리
- Portainer: 컨테이너 스택 내 서비스 계층 표시

**구현 복잡도**: M  
재귀 컴포넌트 또는 라이브러리(react-arborist 등) 사용 필요.

**추천도**: ⭐⭐⭐  
현재 프로젝트에서는 당장 필요성이 낮음. PM2 앱 그룹화 시 유용할 수 있음.

**현재 프로젝트 적용 여부**: 미적용

---

### 2-4. 토폴로지/캔버스 뷰 (Topology / Canvas View)

**설명**  
서비스 간 연결 관계를 시각적 그래프/캔버스로 표현하는 패턴입니다. 노드(서비스/컨테이너)와 엣지(연결/의존성)로 구성됩니다. 드래그로 레이아웃을 조정할 수 있고, 노드 클릭 시 상세 패널이 열립니다.

**사용 플랫폼**
- Railway: 프로젝트 캔버스가 이 패턴의 대표 사례. 서비스/DB 노드를 캔버스에 배치하고 연결선으로 관계 표시
- Docker Compose 시각화 도구들도 유사한 패턴 사용

**구현 복잡도**: L  
ReactFlow 또는 D3.js 필요. 구현 난이도 최상.

**추천도**: ⭐⭐  
현재 단일 서버 환경에서는 과도한 복잡성. 멀티 서버 확장 후 고려.

**현재 프로젝트 적용 여부**: 미적용

---

### 2-5. JSON/YAML 트리 뷰어 (JSON / YAML Tree Viewer)

**설명**  
JSON 또는 YAML 데이터를 접이식 트리 구조로 시각화하는 패턴입니다. 키-값 쌍, 배열, 중첩 객체를 색상과 들여쓰기로 구분합니다. 검색 하이라이팅, 경로 복사, 원시 텍스트/트리 뷰 전환 기능을 포함합니다.

**사용 플랫폼**
- Firebase Console: Firestore 문서 뷰어 (JSON 트리 형태로 필드 표시)
- Supabase Studio: SQL 쿼리 결과의 JSON 셀 확장 뷰
- Portainer: 컨테이너 Inspect 결과 JSON 뷰어

**구현 복잡도**: M  
`react-json-view` 또는 직접 구현. 읽기 전용이면 S, 편집 가능하면 M~L.

**추천도**: ⭐⭐⭐  
PM2 프로세스 상세 정보 표시 시 유용. 현재 모달에서 단순 텍스트로 표시 중이므로 개선 가능.

**현재 프로젝트 적용 여부**: 미적용

---

### 2-6. 상태 배지/레이블 (Status Badge / Label)

**설명**  
엔티티의 현재 상태를 색상이 있는 작은 배지로 표시하는 패턴입니다. Online/Offline, Running/Stopped/Error, Active/Inactive 등의 상태를 초록/빨강/노랑/회색 등으로 직관적으로 전달합니다. 배지에 작은 원형 인디케이터(dot)를 추가하면 실시간 업데이트 느낌을 줍니다.

**사용 플랫폼**
- 모든 6개 플랫폼이 공통으로 사용하는 핵심 패턴
- Supabase Studio: DB 연결 상태, Edge Function 배포 상태
- Portainer: 컨테이너 running/stopped/exited 상태
- Railway: 배포 상태 (Deploying / Success / Failed)
- Render: 서비스 상태 (Live / Building / Failed / Suspended)

**구현 복잡도**: S  
색상 매핑 객체 + Tailwind CSS 클래스 조합으로 구현.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /processes 페이지에서 이미 사용 중. 더 많은 상태 표현에 확장 권장.

**현재 프로젝트 적용 여부**: 적용됨

---

## 3. 데이터 조작 패턴

사용자가 데이터를 생성, 수정, 삭제하는 방식에 관한 패턴입니다.

---

### 3-1. 모달 폼 (Modal Form)

**설명**  
현재 페이지 위에 오버레이로 뜨는 다이얼로그 형태의 폼입니다. 엔티티 생성/수정 시 사용되며, 배경 클릭이나 Escape 키로 닫을 수 있습니다. 제출 후에는 모달이 닫히고 목록이 새로고침됩니다. 중간 크기(400–600px 폭)의 데이터 입력에 적합합니다.

**사용 플랫폼**
- Supabase Studio: Auth 사용자 생성, Storage 버킷 생성
- Firebase Console: Firestore 컬렉션/문서 추가
- Coolify: 서비스 생성, 환경변수 추가
- Portainer: 네트워크 생성, 볼륨 생성

**구현 복잡도**: S~M  
shadcn/ui의 Dialog 컴포넌트 또는 Headless UI Modal 사용.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /processes 페이지의 상세 모달이 이 패턴. 환경변수 편집, 설정 변경 등에 확장 가능.

**현재 프로젝트 적용 여부**: 부분 적용 (조회 모달만 있음, 편집 모달 없음)

---

### 3-2. 슬라이드오버 패널 (Slide-over Panel)

**설명**  
화면 오른쪽에서 슬라이드인하는 넓은 사이드 패널입니다. 모달보다 더 많은 정보를 표시할 수 있으며, 배경 페이지를 부분적으로 볼 수 있어 컨텍스트를 유지합니다. 상세 정보 조회 + 편집을 동시에 처리할 때 적합합니다.

**사용 플랫폼**
- Supabase Studio: Table Editor에서 행 클릭 시 우측 슬라이드오버로 상세 편집
- Railway: 서비스 클릭 시 우측 패널로 상세 설정 표시
- Render: 환경변수 편집 시 우측 슬라이드 패널

**구현 복잡도**: S~M  
CSS transform + transition으로 구현. 배경 오버레이 dim 처리 포함.

**추천도**: ⭐⭐⭐⭐  
현재 모달보다 더 많은 정보를 표시해야 하는 경우(PM2 프로세스 상세, 로그 필터 설정 등)에 유용.

**현재 프로젝트 적용 여부**: 미적용

---

### 3-3. 인라인 편집 (Inline Edit)

**설명**  
테이블이나 카드에서 셀/필드를 직접 클릭하면 편집 모드로 전환되는 패턴입니다. Enter 또는 blur 이벤트에서 자동 저장됩니다. 스프레드시트와 유사한 경험을 제공하여 대량의 단순 데이터 편집 시 효율적입니다.

**사용 플랫폼**
- Supabase Studio: Table Editor에서 셀 클릭으로 직접 편집 (대표 사례)
- Firebase Console: Firestore 문서 필드 인라인 편집

**구현 복잡도**: M  
클릭 상태 관리, 포커스 처리, 자동 저장 로직, 낙관적 업데이트 처리 필요.

**추천도**: ⭐⭐⭐  
현재 프로젝트에서는 환경변수 편집 등에 활용 가능하나 즉각적인 필요성은 낮음.

**현재 프로젝트 적용 여부**: 미적용

---

### 3-4. 벌크 액션 (Bulk Action)

**설명**  
테이블에서 여러 행을 체크박스로 선택한 후 한 번에 동일한 액션(삭제/중지/재시작 등)을 적용하는 패턴입니다. 선택 시 하단 또는 상단에 "N개 선택됨" + 액션 버튼 바가 나타납니다.

**사용 플랫폼**
- Supabase Studio: Auth 사용자 다중 삭제
- Firebase Console: Firestore 다중 문서 삭제
- Portainer: 여러 컨테이너 동시 시작/중지/삭제

**구현 복잡도**: M  
선택 상태 관리 + 확인 다이얼로그 + 배치 API 호출 처리 필요.

**추천도**: ⭐⭐⭐⭐  
/processes 페이지에서 PM2 프로세스 다중 재시작/중지 시 유용.

**현재 프로젝트 적용 여부**: 미적용

---

### 3-5. 위험 액션 확인 다이얼로그 (Destructive Action Confirmation Dialog)

**설명**  
삭제, 중지, 롤백 등 되돌리기 어려운 작업 전에 "정말 삭제하시겠습니까?" 확인 다이얼로그를 표시하는 패턴입니다. 고위험 액션에는 엔티티 이름을 직접 입력하게 하는 이중 확인 방식도 사용됩니다 (예: "프로세스 이름을 입력하여 확인하세요").

**사용 플랫폼**
- 모든 6개 플랫폼에서 공통으로 사용
- Supabase Studio: 프로젝트 삭제 시 프로젝트 이름 입력 확인
- Railway: 서비스 삭제 시 서비스 이름 입력 확인
- Portainer: 컨테이너 삭제 시 확인 다이얼로그

**구현 복잡도**: S  
Alert Dialog 컴포넌트(shadcn/ui의 AlertDialog)로 간단하게 구현.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /processes에서 재시작/중지/삭제 버튼에 이미 일부 구현. 삭제 시 이름 입력 확인은 미구현.

**현재 프로젝트 적용 여부**: 부분 적용

---

## 4. 모니터링 패턴

시스템 상태를 실시간으로 파악하고 문제를 감지하는 패턴입니다.

---

### 4-1. 실시간 메트릭 차트 (Real-time Metric Chart)

**설명**  
시간축(X축)과 메트릭 값(Y축)으로 구성된 라인 차트 또는 영역 차트입니다. 일정 간격(1초~60초)으로 새 데이터 포인트가 추가되며 오래된 데이터는 왼쪽으로 밀려납니다. CPU, 메모리, 네트워크 I/O 등 연속적인 수치 데이터 표시에 적합합니다.

**사용 플랫폼**
- Supabase Studio: Reports 페이지의 API 요청 수 시계열 차트
- Railway: 서비스 메트릭 (CPU/Memory/Network) 실시간 라인 차트
- Render: 서비스 메트릭 차트
- Portainer: 컨테이너 Stats 차트 (CPU/Memory 실시간)

**구현 복잡도**: M  
Recharts 또는 Chart.js + SSE/WebSocket or polling으로 구현.

**추천도**: ⭐⭐⭐⭐⭐  
현재 / 페이지에서 미니 차트 구현. 전체 차트 페이지 확장 권장. 핵심 기능.

**현재 프로젝트 적용 여부**: 부분 적용 (mini chart만 있음, 전체 차트 페이지 없음)

---

### 4-2. 시스템 헬스 인디케이터 (System Health Indicator)

**설명**  
시스템 전체 또는 주요 구성 요소의 건강 상태를 단일 색상 신호(초록/노랑/빨강)로 표시하는 패턴입니다. 사이드바 상단, 헤더, 또는 대시보드 카드에 위치합니다. 여러 지표를 종합하여 전반적인 상태를 계산합니다.

**사용 플랫폼**
- Supabase Studio: 상태 표시등으로 DB 연결 상태 표시
- Railway: 배포 상태 전체 요약
- Render: 서비스별 헬스 체크 상태
- Coolify: 서버 전체 헬스 상태 (Running/Warning/Error)

**구현 복잡도**: S~M  
임계값 기반 상태 계산 로직 + 색상 배지 조합.

**추천도**: ⭐⭐⭐⭐  
현재 PM2 프로세스 상태를 종합한 헬스 인디케이터가 없음. 사이드바에 추가하면 효과적.

**현재 프로젝트 적용 여부**: 미적용

---

### 4-3. 토스트 알림 (Toast Notification)

**설명**  
액션 성공/실패 결과를 화면 우상단 또는 우하단에 일시적으로 표시하는 비모달 알림입니다. 2–5초 후 자동 사라지며, 사용자가 수동으로 닫을 수도 있습니다. 성공(초록), 오류(빨강), 경고(노랑), 정보(파랑) 4가지 변형이 있습니다.

**사용 플랫폼**
- 모든 6개 플랫폼에서 공통으로 사용. 사실상 필수 패턴.
- Supabase Studio: 쿼리 실행 결과, 설정 저장 완료 등
- Coolify: 배포 시작/완료/실패 알림
- Portainer: 컨테이너 시작/중지/삭제 결과

**구현 복잡도**: S  
react-hot-toast 또는 shadcn/ui의 Toast 컴포넌트로 즉시 구현 가능.

**추천도**: ⭐⭐⭐⭐⭐  
현재 프로세스 재시작/중지 후 피드백이 부족. 즉시 적용 권장.

**현재 프로젝트 적용 여부**: 미적용 (확인 필요)

---

### 4-4. 진행 표시기 (Progress Indicator)

**설명**  
배포, 빌드, 마이그레이션 등 시간이 걸리는 작업의 진행 상황을 시각적으로 표시하는 패턴입니다. 단계별 스텝 표시기(1단계 → 2단계 → 완료), 진행률 바(0%→100%), 스피너 등의 형태가 있습니다.

**사용 플랫폼**
- Railway: 배포 단계별 진행 표시 (Build → Deploy → Health Check)
- Coolify: 서비스 배포 진행 상황 스텝 표시
- Render: 빌드 및 배포 진행률 표시
- Firebase Console: 파일 업로드 진행률 바

**구현 복잡도**: M  
비동기 작업 상태 polling + 단계별 표시 컴포넌트 필요.

**추천도**: ⭐⭐⭐  
현재 PM2 프로세스 재시작 등 즉각적인 작업에는 불필요하나, 나중에 배포 기능 추가 시 필요.

**현재 프로젝트 적용 여부**: 미적용

---

### 4-5. 활동 피드 (Activity Feed)

**설명**  
시스템에서 발생한 이벤트를 타임라인 형식으로 나열하는 패턴입니다. 각 이벤트에는 타임스탬프, 이벤트 유형 아이콘, 설명, 관련 사용자가 표시됩니다. 최근 이벤트가 상단에 위치하며 무한 스크롤로 과거 이벤트를 조회합니다.

**사용 플랫폼**
- Supabase Studio: Auth 사용자 로그인 이벤트 피드
- Railway: 배포 이력 타임라인
- Render: 서비스 이벤트 로그
- Coolify: 배포/설정 변경 활동 피드

**구현 복잡도**: M  
이벤트 저장소 + 타임라인 UI 컴포넌트.

**추천도**: ⭐⭐⭐  
현재 /logs 페이지가 유사한 역할을 하나, 구조화된 활동 피드와는 다름.

**현재 프로젝트 적용 여부**: 간접 적용 (/logs 페이지)

---

## 5. 검색/필터 패턴

원하는 데이터를 빠르게 찾고 범위를 좁히는 패턴입니다.

---

### 5-1. 인라인 테이블 필터 (Inline Table Filter)

**설명**  
테이블 상단 또는 각 컬럼 헤더 아래에 위치한 필터 입력 필드입니다. 타이핑과 동시에 즉각적인 결과 필터링이 이루어집니다. 텍스트 검색 외에 상태 선택 드롭다운, 날짜 범위 선택기를 함께 제공합니다.

**사용 플랫폼**
- Supabase Studio: Table Editor의 컬럼별 필터 빌더
- Portainer: 컨테이너 목록 상단의 이름/상태 필터
- Railway: 배포 이력 필터 (성공/실패/진행중)

**구현 복잡도**: S~M  
controlled input + 클라이언트 사이드 필터링은 S. 서버 사이드 필터링은 M.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /processes 페이지에서 PM2 프로세스 이름 검색 필터 미구현. Quick Win 대상.

**현재 프로젝트 적용 여부**: 미적용

---

### 5-2. 로그 검색 및 필터 (Log Search & Filter)

**설명**  
로그 뷰어에서 키워드 검색, 로그 레벨 필터(INFO/WARN/ERROR), 시간 범위 선택, 프로세스 선택을 제공하는 패턴입니다. 검색 결과 하이라이팅, 컨텍스트 라인 표시(앞뒤 N줄), 정규식 지원이 포함될 수 있습니다.

**사용 플랫폼**
- Supabase Studio: Log Explorer. 프로젝트별 로그, 쿼리 필터, 시간 범위 선택
- Railway: 실시간 로그 스트리밍 + 키워드 검색
- Render: 로그 스트리밍 + 검색
- Portainer: 컨테이너 로그 뷰어 + 검색 + 타임스탬프 토글

**구현 복잡도**: M  
현재 /logs 페이지에 이미 일부 구현. 정규식 지원, 시간 범위 필터 추가 시 M.

**추천도**: ⭐⭐⭐⭐⭐  
현재 /logs 페이지에 검색/필터 기능 있음. 로그 레벨별 색상 구분, 시간 범위 선택 강화 권장.

**현재 프로젝트 적용 여부**: 부분 적용 (/logs 페이지에 기본 검색)

---

### 5-3. 날짜/시간 범위 선택기 (Date-Time Range Picker)

**설명**  
모니터링 차트나 로그 뷰어에서 조회 기간을 선택하는 패턴입니다. 프리셋(최근 1시간, 24시간, 7일 등) + 커스텀 날짜 범위 입력 조합으로 제공됩니다. 선택한 범위가 URL 쿼리 파라미터에 반영되어 공유 가능합니다.

**사용 플랫폼**
- Supabase Studio: Log Explorer의 시간 범위 선택
- Railway: 메트릭 차트 기간 선택
- Render: 메트릭 기간 선택 (1h / 24h / 7d / 30d)

**구현 복잡도**: M  
react-day-picker 또는 shadcn/ui의 DatePicker 컴포넌트 활용. URL 동기화 추가 시 M.

**추천도**: ⭐⭐⭐⭐  
현재 /logs 페이지에 시간 범위 선택 없음. 로그가 많아질수록 필수적.

**현재 프로젝트 적용 여부**: 미적용

---

### 5-4. 글로벌 검색 (Global Search)

**설명**  
헤더 중앙 또는 커맨드 팔레트를 통해 전체 데이터를 검색하는 패턴입니다. 프로세스, 로그, 설정 항목 등 모든 범주를 한 번에 검색하고 결과를 카테고리별로 그룹화하여 표시합니다.

**사용 플랫폼**
- Supabase Studio: Cmd+K 커맨드 팔레트에서 글로벌 검색
- Firebase Console: 상단 검색 바로 서비스/문서 검색

**구현 복잡도**: M~L  
검색 인덱스 구축, 결과 랭킹, 카테고리 분류 로직 필요.

**추천도**: ⭐⭐⭐  
현재 규모에서는 커맨드 팔레트가 이 역할을 겸할 수 있음.

**현재 프로젝트 적용 여부**: 미적용

## 6. 설정 패턴

시스템 설정을 관리하는 패턴입니다.

---

### 6-1. 환경변수 편집기 (Environment Variable Editor)

**설명**  
KEY=VALUE 형태의 환경변수를 관리하는 전용 UI입니다. 추가/수정/삭제 행 단위로 이루어지며, 값은 기본적으로 마스킹(**)됩니다. 눈 아이콘으로 값 표시/숨기기 토글, 복사 버튼, 일괄 편집(raw text paste) 기능이 포함됩니다.

**사용 플랫폼**
- Railway: 서비스별 환경변수 편집기. Key/Value 테이블 + 추가 버튼
- Render: 환경변수 그룹 관리. 서비스 간 공유 환경변수 지원
- Coolify: 서비스 환경변수 편집. Docker 환경변수와 동기화
- Supabase Studio: Edge Function 환경변수 관리

**구현 복잡도**: M  
행 CRUD + 마스킹 처리 + 저장 로직 필요.

**추천도**: ⭐⭐⭐⭐  
현재 .env 파일을 직접 편집해야 함. 웹 UI에서 환경변수 편집 기능은 중기 목표로 권장.

**현재 프로젝트 적용 여부**: 미적용

---

### 6-2. 토글/스위치 설정 (Toggle / Switch Setting)

**설명**  
활성화/비활성화가 명확한 기능 설정에 사용하는 토글 스위치 패턴입니다. 변경 즉시 저장(auto-save) 또는 저장 버튼 클릭 후 적용 방식 두 가지가 있습니다. 토글 옆에는 기능 이름과 간단한 설명이 함께 표시됩니다.

**사용 플랫폼**
- Supabase Studio: Auth 소셜 로그인 공급자 활성화 토글
- Firebase Console: 기능 활성화 토글
- Coolify: 서비스 자동 배포, HTTPS 강제 리다이렉션 토글

**구현 복잡도**: S  
shadcn/ui의 Switch 컴포넌트 + API 호출.

**추천도**: ⭐⭐⭐⭐  
향후 설정 페이지에서 알림 설정, 모니터링 옵션 등에 활용 가능.

**현재 프로젝트 적용 여부**: 미적용

---

### 6-3. 설정 섹션 페이지 (Settings Section Page)

**설명**  
좌측에 설정 카테고리 메뉴, 우측에 해당 설정 내용이 표시되는 2단 레이아웃의 설정 페이지입니다. 각 카테고리는 General / Security / Notifications / Advanced 등으로 분류됩니다. 위험 작업(계정 삭제, 초기화 등)은 페이지 하단의 "Danger Zone" 섹션에 별도 배치됩니다.

**사용 플랫폼**
- Supabase Studio: 프로젝트 Settings 페이지 (General / Auth / Storage / API / Functions / Database)
- Coolify: 서버 설정, 팀 설정
- Railway: 프로젝트 설정, 팀 설정

**구현 복잡도**: M  
2단 레이아웃 + 다수의 설정 폼 컴포넌트.

**추천도**: ⭐⭐⭐  
현재는 설정 항목이 적으나 향후 기능 확장 시 필요.

**현재 프로젝트 적용 여부**: 미적용 (설정 페이지 자체가 없음)

## 7. 에디터 패턴

코드, SQL, JSON 등 구조화된 텍스트를 입력하고 편집하는 패턴입니다.

---

### 7-1. SQL 에디터 (SQL Editor)

**설명**  
구문 하이라이팅, 자동 완성, 실행 버튼, 결과 테이블 표시를 갖춘 SQL 쿼리 편집기입니다. 쿼리 이력, 즐겨찾기 저장, 공유 URL, 설명(Explain) 실행 기능이 포함됩니다. Monaco Editor(VS Code 엔진) 기반이 가장 일반적입니다.

**사용 플랫폼**
- Supabase Studio: SQL Editor가 핵심 기능 중 하나. Monaco Editor 기반
- Firebase Console: 유사한 쿼리 빌더 제공

**구현 복잡도**: L  
Monaco Editor 통합 + SQL 파싱 + DB 연결 필요.

**추천도**: ⭐⭐  
현재 프로젝트(서버 모니터링)에서는 불필요. DB 관리 기능 추가 시 고려.

**현재 프로젝트 적용 여부**: 해당 없음

---

### 7-2. 코드/설정 에디터 (Code / Config Editor)

**설명**  
환경변수, Nginx 설정, PM2 설정 파일 등을 Monaco Editor 또는 CodeMirror 기반으로 편집하는 패턴입니다. 구문 하이라이팅, 자동 들여쓰기, 오류 표시를 제공합니다. 저장 버튼 또는 Ctrl+S로 저장합니다.

**사용 플랫폼**
- Coolify: Nginx 설정, Docker Compose 파일 편집
- Railway: 서비스 Dockerfile, 빌드 설정 편집
- Supabase Studio: Edge Function 코드 편집

**구현 복잡도**: M~L  
Monaco Editor 통합. 단순 텍스트 에디터라면 CodeMirror 6으로 경량 구현 가능.

**추천도**: ⭐⭐⭐  
PM2 ecosystem.config.js 웹 편집 기능으로 활용 가능. 중기 목표.

**현재 프로젝트 적용 여부**: 미적용

---

### 7-3. 폼 기반 설정 빌더 (Form-based Config Builder)

**설명**  
원시 설정 파일 편집 대신, UI 폼(레이블 + 입력 필드)으로 설정을 구성하는 패턴입니다. 폼 제출 시 내부적으로 설정 파일을 생성/업데이트합니다. 코드 에디터보다 진입 장벽이 낮지만, 복잡한 설정 표현에는 한계가 있습니다.

**사용 플랫폼**
- Coolify: 서비스 배포 설정 폼 (Git URL, 브랜치, 빌드 명령 등)
- Railway: 서비스 생성 마법사

**구현 복잡도**: M  
React Hook Form + Zod 스키마 검증.

**추천도**: ⭐⭐⭐⭐  
환경변수 편집, PM2 프로세스 생성 폼에 적합. 에디터보다 UX 친화적.

**현재 프로젝트 적용 여부**: 미적용

## 8. 응답 패턴

로딩, 오류, 빈 상태 등 다양한 UI 상태를 처리하는 패턴입니다.

---

### 8-1. 스켈레톤 로딩 (Skeleton Loading)

**설명**  
데이터 로딩 중에 실제 컨텐츠와 동일한 형태의 회색 플레이스홀더를 표시하는 패턴입니다. 스피너보다 레이아웃 이동(Layout Shift)이 적고, 사용자가 기다리는 동안 무엇을 기다리는지 명확하게 전달합니다.

**사용 플랫폼**
- Supabase Studio: 테이블 데이터 로딩 시 행 스켈레톤
- Railway: 서비스 목록 로딩 시 카드 스켈레톤
- Firebase Console: 데이터 로딩 시 스켈레톤

**구현 복잡도**: S  
Tailwind CSS의 `animate-pulse` 클래스로 간단하게 구현.

**추천도**: ⭐⭐⭐⭐⭐  
현재 대시보드에서 데이터 로딩 시 스켈레톤 미사용. Quick Win 대상.

**현재 프로젝트 적용 여부**: 미적용

---

### 8-2. 빈 상태 (Empty State)

**설명**  
데이터가 없을 때 단순히 빈 화면 대신, 일러스트/아이콘 + 설명 문구 + CTA 버튼을 표시하는 패턴입니다. "아직 서비스가 없습니다. 첫 서비스를 추가해보세요!" 같은 문구로 다음 행동을 안내합니다.

**사용 플랫폼**
- 모든 6개 플랫폼에서 공통으로 사용
- Supabase Studio: 테이블이 없을 때 "Create your first table" 빈 상태
- Railway: 프로젝트가 없을 때 "New Project" 빈 상태

**구현 복잡도**: S  
단순 JSX 컴포넌트.

**추천도**: ⭐⭐⭐⭐  
현재 로그나 프로세스가 없을 때의 빈 상태 처리 개선 권장.

**현재 프로젝트 적용 여부**: 부분 적용

---

### 8-3. 오류 경계 (Error Boundary)

**설명**  
API 오류, 네트워크 장애 등 예기치 않은 오류가 발생했을 때 전체 페이지가 깨지는 대신 오류 메시지 + 재시도 버튼을 표시하는 패턴입니다. React의 ErrorBoundary 컴포넌트 또는 try-catch + 오류 UI로 구현합니다.

**사용 플랫폼**
- 모든 플랫폼의 기본 안전 패턴
- Supabase Studio: API 요청 실패 시 오류 카드 표시 + 재시도 버튼
- Render: 메트릭 데이터 로드 실패 시 "Failed to load data" + "Try again"

**구현 복잡도**: S~M  
React ErrorBoundary + 컴포넌트별 error state 처리.

**추천도**: ⭐⭐⭐⭐⭐  
PM2 API 호출 실패 시 적절한 오류 표시가 없을 수 있음. 기본 안전 패턴으로 적용 권장.

**현재 프로젝트 적용 여부**: 부분 적용 (확인 필요)

---

### 8-4. 낙관적 업데이트 (Optimistic Update)

**설명**  
API 응답을 기다리지 않고 즉시 UI에 변경사항을 반영하는 패턴입니다. 서버에서 오류가 반환되면 변경사항을 되돌리고 토스트로 오류를 알립니다. 응답성이 높아 사용자 경험이 크게 향상됩니다.

**사용 플랫폼**
- Supabase Studio: Table Editor의 셀 편집 즉시 반영
- Railway: 환경변수 변경 즉시 반영

**구현 복잡도**: M  
이전 상태 저장 + 롤백 로직 + TanStack Query의 낙관적 업데이트 기능 활용.

**추천도**: ⭐⭐⭐  
현재 PM2 재시작/중지 액션에서는 실제 서버 응답이 필요하므로 적용 범위가 제한적.

**현재 프로젝트 적용 여부**: 미적용

---

### 8-5. 폴링 기반 자동 새로고침 (Auto-refresh via Polling)

**설명**  
일정 간격(5초, 30초, 1분 등)으로 API를 자동 호출하여 데이터를 업데이트하는 패턴입니다. SSE나 WebSocket보다 구현이 간단하며, 서버 부하도 낮습니다. 사용자가 직접 새로고침 주기를 선택하는 컨트롤을 제공하기도 합니다.

**사용 플랫폼**
- Portainer: 컨테이너 목록 자동 새로고침 (주기 선택 가능)
- Railway: 배포 상태 자동 업데이트
- Render: 서비스 상태 자동 폴링

**구현 복잡도**: S  
`useEffect` + `setInterval` 또는 TanStack Query의 `refetchInterval` 옵션.

**추천도**: ⭐⭐⭐⭐⭐  
현재 대시보드에서 CPU/메모리 데이터 자동 새로고침 필요. Quick Win 대상.

**현재 프로젝트 적용 여부**: 미적용 (확인 필요)

## Quick Win 패턴 (즉시 적용 권장)

구현 복잡도가 낮고(S 수준) 추천도가 높은 패턴 목록입니다.

1. **토스트 알림** — react-hot-toast 1줄 설치로 즉시 구현. 모든 액션에 피드백 제공.
2. **스켈레톤 로딩** — `animate-pulse` Tailwind 클래스만으로 구현. UX 품질 즉시 개선.
3. **접이식 사이드바** — CSS transition + localStorage. 작업 공간 확장.
4. **인라인 테이블 필터** — controlled input으로 PM2 프로세스 검색 구현.
5. **폴링 기반 자동 새로고침** — `refetchInterval` 옵션 1줄 추가.
6. **브레드크럼** — usePathname으로 동적 생성. 상세 페이지 탐색성 향상.
