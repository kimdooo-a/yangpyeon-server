# Supabase 대시보드 UX 해부

> Wave 1 리서치 문서 | 작성일: 2026-04-06  
> 목적: 대시보드 플랫폼 설계자 관점에서 Supabase Studio의 UX 구조를 해부하고, 양평 부엌 대시보드에 차용 가능한 패턴을 추출한다.

---

## 1. 개요

Supabase Studio는 Next.js 기반의 오픈소스 웹 애플리케이션으로, Postgres 데이터베이스를 중심으로 인증, 스토리지, Edge Functions, 실시간 기능을 하나의 통합 인터페이스에서 관리할 수 있는 개발자 플랫폼이다. 단순한 관리 도구를 넘어, 복잡한 인프라를 직관적으로 시각화하고 조작할 수 있는 "개발자용 대시보드 플랫폼"의 대표 사례로 평가받는다.

### 1.1 설계 철학

Supabase의 대시보드 설계는 다음 세 가지 철학을 기반으로 한다.

**단일 진실 소스(Single Source of Truth)**  
모든 서비스(DB, Auth, Storage, Functions)를 하나의 대시보드에서 관리한다. Firebase가 기능별로 여러 콘솔로 분산된 것과 대비된다. 개발자는 컨텍스트를 잃지 않고 하나의 탭 안에서 데이터를 확인하고 쿼리를 실행하며 배포까지 진행할 수 있다.

**투명성과 제어권**  
자동 생성 API, SQL 에디터, 스키마 시각화 등을 통해 내부 동작을 감추지 않고 노출한다. 개발자가 블랙박스 뒤에 무엇이 있는지 직접 볼 수 있도록 설계되어 있다.

**구성 가능성(Composability)**  
각 기능 모듈은 독립적으로 작동하면서도 서로를 10배 강화한다. Table Editor에서 데이터를 보다가 SQL Editor로 전환하거나, Auth 설정에서 RLS 정책으로 이동하는 흐름이 자연스럽다.

## 2. 네비게이션 구조

### 2.1 전체 레이아웃 골격

```
┌─────────────────────────────────────────────────────────┐
│  [상단 헤더]  Organization Picker  User Account  알림   │
├──────────┬──────────────────────────────────────────────┤
│          │                                              │
│  사이드바  │           메인 콘텐츠 영역                   │
│  (좌측)  │                                              │
│          │  ┌─────────────────────────────────────────┐ │
│  - 섹션1 │  │  페이지 헤더 (타이틀 + 브레드크럼)        │ │
│  - 섹션2 │  ├─────────────────────────────────────────┤ │
│  - 섹션3 │  │  탭 네비게이션 (해당 페이지)              │ │
│          │  ├─────────────────────────────────────────┤ │
│  [하단]  │  │  콘텐츠 본문                              │ │
│  Docs    │  └─────────────────────────────────────────┘ │
│  Settings│                                              │
└──────────┴──────────────────────────────────────────────┘
```

Supabase Studio의 레이아웃은 **3계층 구조**로 구성된다. 상단 헤더(조직/프로젝트 선택 + 계정), 좌측 사이드바(기능 모듈 네비게이션), 우측 메인 콘텐츠 영역이다.

### 2.2 사이드바 구조

Supabase 사이드바는 기능을 **논리적 그룹**으로 묶어 명확한 계층 구조를 형성한다. 주요 구성은 다음과 같다.

**Table Editor 그룹**
- Table Editor (스프레드시트형 데이터 CRUD)
- Schema Visualizer (ERD 다이어그램)

**SQL Editor 그룹**
- SQL Editor (쿼리 작성 및 실행)
- Saved Queries (저장된 쿼리 목록)

**Database 그룹**
- Tables (테이블 목록)
- Views, Functions, Triggers, Indexes
- Migrations, Roles, Extensions, Wrappers

**Auth 그룹**
- Users (사용자 목록)
- Policies (RLS 정책)
- Providers (OAuth 설정)
- Hooks, Email Templates

**Storage 그룹**
- Buckets (파일 버킷 관리)
- Policies (스토리지 접근 정책)

**Edge Functions 그룹**
- Functions (배포된 함수 목록)
- Logs

**Realtime 그룹**
- Inspector (실시간 이벤트 모니터)
- Channels

**Monitoring / Reports 그룹**
- Log Explorer
- Database → Disk Usage, Query Performance
- Reports (API, Auth, Storage 사용량)

**하단 고정 섹션**
- Docs (문서 링크)
- Settings (프로젝트 설정)

### 2.3 사이드바 설계 원칙

**그룹 분리 시각화**: 각 그룹 사이에 얇은 구분선(divider)과 레이블을 사용하여 카테고리를 명확하게 구분한다. 시각적 무게감을 최소화하면서도 논리적 경계를 전달한다.

**아이콘 + 텍스트 조합**: 각 메뉴 항목은 아이콘(16~18px Lucide Icons)과 텍스트를 함께 표시한다. 아이콘만 표시하는 축소 모드(collapsed sidebar)도 지원하여 좁은 화면에서의 공간 효율을 높인다.

**선택 상태 강조**: 현재 활성화된 메뉴 항목은 배경색 변경(밝은 초록/에메랄드 계열)과 텍스트 색상 강조로 구분한다. 서브메뉴가 있는 경우 드롭다운으로 확장된다.

**컨텍스트 사이드바**: 특정 기능 영역(Table Editor, Auth 등) 진입 시 해당 섹션 전용 사이드바가 메인 사이드바와 별도로 나타나는 구조를 도입했다. 이를 통해 중첩된 콘텐츠를 탐색하는 인지 부하를 줄인다.

### 2.4 상단 헤더 구조

```
[Supabase 로고] [Organization Picker ▼] / [Project Picker ▼]     [알림] [계정 ▼]
```

- **Organization Picker**: 조직 단위로 프로젝트를 그룹화하여 전환한다. 클릭 시 드롭다운으로 조직 목록 표시.
- **Project Picker**: 현재 조직 내 프로젝트 목록. 최근 방문 순서로 정렬.
- **계정 드롭다운**: 우상단 고정. 프로필, 설정, 로그아웃 포함.

### 2.5 브레드크럼과 페이지 헤더

각 기능 페이지는 상단에 명확한 페이지 헤더를 가진다.

```
[Database] / [Tables] / [users]

users                                                    [+ New Row] [Edit Table]
─────────────────────────────────────────────────────────────────────────────────
[Data] [Indexes] [Triggers] [Foreign Keys] [Policies]
```

브레드크럼은 현재 위치를 명확히 전달하며, 각 단계를 클릭하여 상위 레벨로 이동할 수 있다. 페이지 헤더 오른쪽에는 해당 컨텍스트에서 사용 가능한 주요 액션 버튼(CTA)이 배치된다.

### 2.6 탭 네비게이션

Table Editor, SQL Editor, Auth 등 주요 기능은 **탭 네비게이션**을 적극적으로 사용한다.

**Table Editor 탭**: 여러 테이블을 동시에 열어두고 탭으로 전환. 브라우저의 탭 개념을 차용하여 친숙함을 제공한다.

**SQL Editor 탭**: 여러 SQL 스니펫을 동시에 편집. 저장된 쿼리와 임시 쿼리를 구분하여 표시.

**기능 내부 탭**: 테이블 상세 페이지 내 [Data] / [Indexes] / [Triggers] / [Foreign Keys] / [Policies] 탭처럼, 하나의 리소스에 대한 다양한 관점을 탭으로 구분한다.

## 3. 핵심 기능별 UI 패턴

### 3.1 Table Editor

Table Editor는 Supabase 대시보드의 가장 핵심적인 기능이다. Postgres 테이블을 스프레드시트처럼 시각화하고 직접 편집할 수 있다.

**그리드 레이아웃**

```
┌─ 테이블 목록 ──┐┌─────────────────── 데이터 그리드 ──────────────────────┐
│ users          ││ id │ email          │ name    │ created_at │ ...       │
│ posts          ││────┼────────────────┼─────────┼────────────┼──────────│
│ comments       ││  1 │ admin@test.com │ 관리자   │ 2026-01-01 │          │
│ + New Table    ││  2 │ user@test.com  │ 사용자   │ 2026-01-15 │          │
└────────────────┘│  * │ (새 행 입력)    │         │            │          │
                  └────────────────────────────────────────────────────────┘
```

- **인라인 편집**: 셀을 클릭하면 즉시 편집 가능한 상태로 전환된다. 엔터로 저장, Escape로 취소.
- **행 추가**: 그리드 하단의 `+` 버튼 또는 빈 행 클릭으로 새 행 추가.
- **열 관리**: 열 헤더 우클릭으로 정렬/숨기기/삭제/타입 변경 컨텍스트 메뉴 표시.
- **행 선택**: 좌측 체크박스로 다중 선택 후 일괄 삭제/수정 가능.

**React Data Grid 기반**: Supabase는 고인터랙션 그리드를 위해 React Data Grid를 채택했다. 이는 가상화(virtualization)를 통해 수만 건의 데이터를 성능 저하 없이 렌더링한다.

**필터와 정렬 UI**

```
[필터 추가 ▼]  [정렬 ▼]  [열 숨기기 ▼]              검색: [          ]
─────────────────────────────────────────────────────────────────────
필터: email contains "admin"  ×    created_at > 2026-01-01  ×
```

- 칩(chip) 형태로 활성 필터를 표시하여 현재 적용된 필터를 한눈에 파악.
- 각 칩의 `×` 버튼으로 개별 필터 제거 가능.
- 필터 조합은 AND 조건으로 연결되며, 복잡한 쿼리는 SQL Editor로 유도.

**Left Panel (테이블 목록)**

좌측 패널은 현재 스키마의 테이블 목록을 보여주며, 스키마 전환 셀렉터가 상단에 위치한다. `public`, `auth`, `storage` 등 내장 스키마와 사용자 정의 스키마를 구분한다.

### 3.2 SQL Editor

SQL Editor는 개발자가 직접 SQL을 작성하고 실행할 수 있는 강력한 도구다.

**레이아웃 구조**

```
┌─ 저장된 쿼리 ──┐┌──────────── SQL 편집 영역 ─────────────────┐
│ My Queries     ││ SELECT *                                   │
│  └ user_stats  ││   FROM users                               │
│  └ daily_count ││  WHERE created_at > NOW() - INTERVAL '7d' │
│ Templates      ││                                            │
│  └ Check Size  │├────────────────────────────────────────────│
│  └ Slow Queries││ [▶ 실행]  [저장]  [AI 어시스턴트]          │
└────────────────┘├────────────────────────────────────────────│
                  │ 결과 테이블                                  │
                  │ id │ email          │ created_at           │
                  │  1 │ test@test.com  │ 2026-03-15           │
                  └────────────────────────────────────────────┘
```

**핵심 UI 요소**

- **Monaco Editor 기반 코드 에디터**: VSCode와 동일한 편집 경험. 신택스 하이라이팅, 자동완성(테이블명/컬럼명 포함), 오류 감지.
- **AI SQL 어시스턴트**: 자연어로 쿼리 의도를 설명하면 SQL을 자동 생성. "최근 7일간 신규 가입 사용자 수를 일별로 집계해줘"와 같은 요청을 SQL로 변환.
- **실행 단축키**: `Ctrl+Enter`로 쿼리 실행. 마우스 없이 작업 완결.
- **결과 테이블**: 쿼리 결과를 그리드 형태로 표시. CSV 다운로드, 행 수 표시.
- **실행 시간 표시**: 쿼리 완료 후 실행 시간(ms)을 표시하여 성능 최적화 인사이트 제공.
- **저장/공유**: 쿼리를 팀과 공유하거나 개인 라이브러리에 저장. 폴더 구조로 정리 가능.

### 3.3 Auth 관리

Auth 섹션은 사용자 인증 전반을 관리하는 컨트롤 패널이다.

**사용자 목록 UI**

```
Users                                          [+ Invite User] [+ Create User]
──────────────────────────────────────────────────────────────────────────────
[검색: 이메일, 전화번호...            ]  [필터 ▼]

이메일                    │ 최근 로그인       │ 생성일       │ Provider   │ 상태
admin@test.com           │ 2시간 전         │ 2026-01-01  │ email      │ ✓
user@test.com            │ 3일 전           │ 2026-02-15  │ google     │ ✓
banned@test.com          │ -               │ 2026-03-01  │ email      │ ✗ (차단)
```

- 사용자 클릭 시 슬라이드오버 패널(우측에서 슬라이드인)으로 상세 정보 표시.
- 상세 패널: UUID, 메타데이터, 마지막 로그인, 세션 목록, 비밀번호 초기화, 차단/삭제 액션.
- Provider 배지로 로그인 방식 시각화(email, google, github 등).

**Policies 편집기**

RLS(Row Level Security) 정책을 GUI와 SQL로 모두 편집 가능한 하이브리드 편집기를 제공한다.

```
┌─ 테이블 선택 ──────────────────────────────────────────────────────────┐
│ users 테이블의 RLS 정책                              [+ New Policy]     │
├───────────────────────────────────────────────────────────────────────│
│ ✓ RLS Enabled                                                         │
│                                                                       │
│ 정책명                    │ Command │ Check                           │
│ Users can view own data  │ SELECT  │ auth.uid() = id               │
│ Users can update own     │ UPDATE  │ auth.uid() = id               │
└───────────────────────────────────────────────────────────────────────┘
```

**Providers 설정**

각 OAuth Provider(Google, GitHub, Apple 등)를 카드 형태로 나열하고, 토글로 활성화/비활성화. 활성화 시 Client ID/Secret 입력 필드가 인라인으로 확장된다.

### 3.4 Storage 관리

Storage는 파일 관리자와 유사한 UI를 제공한다.

**버킷 브라우저**

```
Buckets                                                    [+ New Bucket]
──────────────────────────────────────────────────────────────────────────
avatars (public)         │ 1,234 objects │ 45.2 MB  │ [Browse] [Settings]
documents (private)      │   892 objects │ 128.5 MB │ [Browse] [Settings]
```

**파일 브라우저 (버킷 내부)**

```
avatars /                                        [Upload Files] [New Folder]
──────────────────────────────────────────────────────────────────────────
[파일 탐색기 뷰 / 그리드 뷰 전환]   정렬: [이름 ▼]   검색: [         ]

□ 이름               │ 크기    │ 타입      │ 최종 수정
□ 📁 2026/           │ -       │ folder    │ 2026-03-01
□ 🖼 avatar_001.jpg  │ 45.2 KB │ image/jpg │ 2026-02-15
□ 🖼 avatar_002.png  │ 89.1 KB │ image/png │ 2026-02-20
```

- 파일 클릭 시 우측 패널에서 미리보기 + 메타데이터 표시.
- 이미지의 경우 썸네일 미리보기 즉시 로드.
- 공개 URL 복사, 다운로드, 삭제 액션을 컨텍스트 메뉴로 제공.
- 드래그 앤 드롭으로 파일 업로드.

### 3.5 Edge Functions 관리

배포된 Edge Functions 목록과 상태를 관리하는 섹션이다.

```
Edge Functions                                          [+ Deploy New Function]
──────────────────────────────────────────────────────────────────────────────
이름                │ 상태     │ 호출 횟수 │ 마지막 배포     │ 리전
send-welcome-email  │ 🟢 Active │ 1,234    │ 2시간 전       │ Global
process-payment     │ 🟢 Active │   892    │ 1일 전         │ Global
legacy-webhook      │ 🔴 Error  │     5    │ 7일 전         │ Global
```

- 함수명 클릭 시 상세 페이지로 이동. 코드 미리보기, 로그 탭, 테스트 도구, 환경변수 설정 포함.
- 대시보드에서 직접 함수 코드를 작성하고 배포 가능(CLI 없이도 동작).

### 3.6 Log Explorer

Log Explorer는 Supabase 스택 전체의 로그를 SQL로 쿼리하는 독특한 접근법을 취한다.

```
Log Explorer                                                      [Templates ▼]
──────────────────────────────────────────────────────────────────────────────
로그 소스: [postgres_logs ▼]       시간 범위: [최근 1시간 ▼]

SELECT timestamp, event_message
FROM postgres_logs
WHERE event_message ILIKE '%error%'
ORDER BY timestamp DESC
LIMIT 100
                                                                [▶ 실행]
──────────────────────────────────────────────────────────────────────────────
timestamp              │ event_message
2026-04-06 10:23:45   │ ERROR: duplicate key value violates unique constraint
2026-04-06 10:22:11   │ ERROR: connection timeout
```

- 로그 소스: postgres_logs, auth_logs, edge_logs, function_logs, storage_logs, realtime_logs
- SQL로 로그를 필터링하고 집계할 수 있어 고급 분석이 가능.
- 미리 만들어진 템플릿으로 일반적인 조사 패턴을 빠르게 시작.

### 3.7 Schema Visualizer (ERD)

데이터베이스 스키마를 시각적 ERD(Entity-Relationship Diagram)로 표현한다.

- 테이블을 노드로, 외래키를 엣지(화살표)로 표현.
- 노드를 드래그로 재배치하여 원하는 레이아웃으로 정리.
- 테이블 노드 클릭 시 컬럼 목록과 제약 조건 표시.
- 복잡한 데이터 모델을 한눈에 파악하는 데 효과적.

## 4. 데이터 CRUD 패턴

### 4.1 인라인 편집 패턴

Supabase는 스프레드시트 인터페이스에서 **인라인 편집**을 핵심 상호작용으로 사용한다.

**동작 방식**
1. 사용자가 셀을 단일 클릭 → 셀이 편집 가능한 상태로 전환(테두리 강조).
2. 값 수정 후 엔터 → 즉시 DB에 반영(낙관적 업데이트 + 에러 시 롤백).
3. Escape → 변경 취소.
4. Tab → 다음 셀로 이동.

**적용 범위**: Table Editor의 모든 셀, Auth 사용자의 일부 속성 인라인 편집.

**장점**: 컨텍스트 전환 없이 빠른 데이터 수정. 스프레드시트 사용자에게 친숙.

**단점**: 복잡한 관계형 데이터나 JSON 필드의 경우 인라인 편집이 제한적. 이 경우 모달 폼으로 전환.

### 4.2 모달 폼 패턴

새 레코드 생성, 복잡한 데이터 편집에는 **모달 폼**을 사용한다.

```
┌─────────────────── New Row ───────────────────────────────┐
│                                                           │
│  id           [자동 생성          ]  (bigint, generated)  │
│  email        [                  ]  *필수                 │
│  name         [                  ]                        │
│  role         [user           ▼  ]  (enum: user, admin)  │
│  metadata     {                  }  (jsonb)               │
│               }                  }                        │
│  created_at   [2026-04-06        ]  (timestamptz)        │
│                                                           │
│                                      [취소] [저장]        │
└───────────────────────────────────────────────────────────┘
```

- 각 필드는 컬럼 타입에 맞는 입력 컴포넌트를 자동 선택(텍스트, 숫자, 날짜피커, 셀렉트, 토글, JSON 에디터).
- 필수 필드는 `*` 표시.
- 유효성 검사 오류는 필드 하단에 인라인으로 표시.
- JSON 필드는 Monaco Editor 기반의 JSON 편집기로 표시.

### 4.3 슬라이드오버(Side Panel) 패턴

Auth 사용자 상세, 파일 메타데이터, 설정 등 **상세 정보 표시**에는 우측에서 슬라이드인하는 패널을 사용한다.

```
메인 콘텐츠 영역 (좌측)          │   슬라이드오버 (우측, 폭 400px)
──────────────────────────────────│──────────────────────────────────
사용자 목록                        │  ← [닫기 ×]
  admin@test.com  [선택됨]  →    │
  user@test.com               │  admin@test.com
                               │  ──────────────
                               │  UUID: abc-123...
                               │  생성일: 2026-01-01
                               │  마지막 로그인: 2시간 전
                               │
                               │  [비밀번호 초기화] [차단] [삭제]
```

- 목록 컨텍스트를 유지하면서 상세 정보를 볼 수 있어 편리.
- 여러 항목을 순서대로 검토할 때 효율적.

### 4.4 필터/정렬 패턴

**칩 기반 필터 UI**: 활성 필터를 작은 칩(pill badge)으로 표시한다.

```
필터: [email contains "admin" ×]  [created_at > 2026-01-01 ×]  [+ 필터 추가]
```

**정렬 우선순위 표시**: 다중 정렬 시 각 컬럼에 번호(1, 2, 3...)를 표시하여 우선순위를 명확히 전달.

**실시간 적용**: 필터/정렬 변경 시 즉시 데이터가 업데이트됨. 별도의 "적용" 버튼 없음.

## 5. 상태 표시 패턴

### 5.1 실시간 지표

Supabase Reports와 대시보드 홈에서 사용하는 실시간 지표 표시 패턴이다.

**미니 차트(Spark Line)**: 콤팩트한 공간에 추세를 보여주는 단순한 선 그래프. 수치 레이블과 함께 표시.

```
┌──────────────────────┐  ┌──────────────────────┐
│  Database Size        │  │  API Requests         │
│  128 MB               │  │  1,234 / min          │
│  ╱╲   ╱╲___╱         │  │  ________╱╲___        │
│  지난 24시간           │  │  지난 24시간            │
└──────────────────────┘  └──────────────────────┘
```

**게이지 차트**: 플랜 한도 대비 사용량을 반원 게이지로 표시. 80% 이상에서 주황, 100%에서 빨간색 경고.

**Realtime Inspector**: 실시간 이벤트를 스트리밍 로그 형태로 표시. 새 이벤트는 상단에 추가(역순 스크롤).

### 5.2 상태 배지 패턴

```
[🟢 Active]   [🟡 Pending]   [🔴 Error]   [⚫ Inactive]
```

- 색상 코딩 + 텍스트 조합으로 상태 전달. 색맹 사용자를 위해 색상만으로 상태를 전달하지 않는다.
- 소형 도트(8px) + 레이블 형식을 가장 많이 사용.
- 테이블 행의 상태 컬럼에는 배지(badge) 컴포넌트 사용.

### 5.3 알림 및 토스트

**토스트 알림**: 우하단에 나타나는 임시 알림. 성공(초록), 오류(빨강), 경고(주황), 정보(파랑).

```
✅  Table "users" updated successfully.           [닫기]
```

**인라인 오류 메시지**: 폼 필드 바로 아래에 오류 메시지를 표시. 전역 알림보다 맥락에 맞는 오류 전달.

**배너 경고**: 페이지 상단 전체 폭 배너로 중요한 시스템 메시지 표시. 이메일 미확인, 플랜 초과, 서비스 장애 등.

```
⚠️  You've reached 80% of your database storage limit. Upgrade your plan to continue.   [Upgrade]
```

### 5.4 스켈레톤 로딩 패턴

데이터 로딩 중에는 실제 콘텐츠와 동일한 형태의 회색 스켈레톤을 표시한다. 레이아웃 시프트를 방지하고 로딩 속도가 빠른 것처럼 느껴지게 한다.

```
[████████████████  ]  [████████  ]  [████  ]
[████████████████  ]  [████████  ]  [████  ]
[████████████████  ]  [████████  ]  [████  ]
```

## 6. 정보 밀도와 레이아웃 전략

### 6.1 다크 테마 + 에메랄드 액센트

Supabase는 다크 테마를 기본으로 하며, 에메랄드(#3ECF8E) 색상을 주요 액센트로 사용한다. 이 컬러 선택은 브랜드 정체성이자 기술적 전문성을 시각적으로 전달하는 수단이다.

**배경 계층**:
- 최심부 배경: `#0F0F0F` (거의 검정)
- 카드/패널 배경: `#1C1C1C`
- 입력 필드/테이블 행 배경: `#262626`
- 호버 상태: `#303030`

**텍스트 계층**:
- 주요 텍스트: `#EDEDED` (거의 흰색)
- 보조 텍스트: `#A0A0A0` (회색)
- 비활성 텍스트: `#606060`

### 6.2 정보 밀도 조절

Supabase는 **높은 정보 밀도**와 **가독성 유지** 사이의 균형을 다음 전략으로 달성한다.

**일관된 여백 시스템**: 4px 단위의 스페이싱 시스템 사용. 컴팩트하지만 숨막히지 않는 밀도.

**타이포그래피 계층**: 섹션 헤더(16px, 중간 굵기) → 레이블(13px, 일반) → 값(13px, 일반) → 도움말 텍스트(12px, 회색)의 명확한 계층.

**점진적 공개(Progressive Disclosure)**: 기본적으로는 핵심 정보만 표시하고, 더 자세한 내용은 확장/클릭으로 공개. 테이블 행에서 "더 보기" 클릭 시 전체 JSON 펼치기가 대표 예.

**빈 상태(Empty State) 디자인**: 데이터가 없을 때 단순한 공백 대신 안내 메시지와 CTA를 제공. "Edge Function이 없습니다. 첫 번째 함수를 배포해보세요" + [Deploy Function] 버튼.

### 6.3 반응형 전략

Supabase Studio는 주로 데스크톱 환경을 타겟으로 설계되었다. 모바일 대응보다는 넓은 화면에서의 효율성을 우선시한다.

- 사이드바: 기본 펼침(240px), 축소 모드(아이콘만, 60px), 모바일에서 오버레이.
- 그리드: 컬럼 수를 고정하지 않고 수평 스크롤을 허용.
- 모달/슬라이드오버: 항상 충분한 너비 확보(최소 400px).

### 6.4 Command Menu (Cmd+K)

Supabase의 Command Menu는 키보드 중심 사용자를 위한 빠른 네비게이션 도구다.

```
┌─────────────────────────────────────────────────────────────┐
│  🔍  검색 또는 명령어 입력...                                  │
│─────────────────────────────────────────────────────────────│
│  최근                                                        │
│  > Table: users                                             │
│  > SQL: daily_count                                         │
│  > Auth: Policies                                           │
│─────────────────────────────────────────────────────────────│
│  작업                                                        │
│  > New Table                                               │
│  > New SQL Query                                           │
│  > Invite Team Member                                      │
└─────────────────────────────────────────────────────────────┘
```

- `Cmd+K` (Mac) / `Ctrl+K` (Windows)로 어디서든 즉시 열기.
- 페이지 네비게이션, 최근 방문 리소스, 빠른 액션을 하나의 검색 인터페이스로 통합.
- 퍼지 검색(fuzzy search)으로 정확한 이름 몰라도 검색 가능.

## 7. Radix UI + Tailwind CSS 디자인 시스템

Supabase Studio의 UI 구현 기반을 이해하면 동일한 기술 스택(Next.js + Tailwind)으로 유사한 경험을 재현하는 데 도움이 된다.

### 7.1 디자인 시스템 구성

**기반 라이브러리**:
- Radix UI: 접근성이 검증된 헤드리스(headless) 컴포넌트 (다이얼로그, 드롭다운, 토스트, 탭 등)
- Tailwind CSS: 유틸리티 클래스 기반 스타일링
- Radix UI Colors: 일관된 색상 팔레트 (다크/라이트 모드 자동 전환)
- Lucide Icons: 일관된 아이콘 세트

**공개된 UI 컴포넌트 라이브러리**: Supabase는 2025년에 자체 UI 라이브러리(supabase/ui)를 공개했다. shadcn/ui 방식(복사-붙여넣기 컴포넌트)으로 배포되어 프로젝트에 직접 통합 가능.

### 7.2 핵심 컴포넌트 패턴

**Data Table 패턴**:
- TanStack Table (구 react-table) 기반
- 가상화, 정렬, 필터링, 페이지네이션 내장
- 컬럼 정의를 선언적으로 작성

**Data Grid 패턴** (Table Editor용):
- React Data Grid 기반
- 인라인 편집, 셀 선택, 키보드 네비게이션 지원
- 고성능 가상 스크롤

## 8. 양평 부엌 대시보드에 차용 가능한 핵심 UX 요소

### 8.1 즉시 적용 가능한 패턴 (높은 우선순위)

**1. 탭 기반 멀티뷰**
- 현재: 단일 페이지에 모든 정보를 나열
- 개선: 탭으로 기능을 분리 (예: 프로세스 페이지에서 [실행 중] [전체] [오류] 탭)
- 구현 난이도: 낮음

**2. 상태 배지(Status Badge)**
- 현재: 텍스트로만 상태 표시
- 개선: 색상 도트 + 텍스트 배지 (🟢 Online / 🔴 Stopped / 🟡 Restarting)
- 구현 난이도: 낮음

**3. 칩 기반 필터 UI**
- 현재: 단순 검색 또는 필터 없음
- 개선: 로그 페이지에서 레벨별 필터 칩 ([ERROR ×] [WARN ×] [INFO])
- 구현 난이도: 낮음

**4. 슬라이드오버 상세 패널**
- 현재: 별도 페이지 이동 또는 모달
- 개선: PM2 프로세스 클릭 시 우측 슬라이드오버로 상세 정보 표시
- 구현 난이도: 중간

**5. 스켈레톤 로딩**
- 현재: 로딩 스피너 또는 빈 화면
- 개선: 콘텐츠 형태를 반영한 스켈레톤 로딩
- 구현 난이도: 낮음

### 8.2 중기 적용 패턴 (중간 우선순위)

**6. 미니 차트(Spark Line) 카드**
- CPU/메모리 사용량을 숫자 + 추세 선 그래프로 표현
- 24시간 추세를 한눈에 파악

**7. Command Menu (Cmd+K)**
- 페이지 간 빠른 이동
- PM2 프로세스 시작/중지 명령 직접 실행
- 구현 난이도: 중간 (cmdk 라이브러리 활용)

**8. 점진적 공개(Progressive Disclosure)**
- 로그 항목의 기본 표시는 1줄, 클릭 시 전체 JSON 메타데이터 펼치기

**9. 빈 상태(Empty State) 디자인**
- 데이터 없을 때 안내 메시지 + CTA 버튼
- 예: "오류 로그가 없습니다. 서버가 정상 동작 중입니다." (아이콘 + 메시지)

### 8.3 장기 적용 패턴 (낮은 우선순위)

**10. SQL/쿼리 기반 로그 탐색기**
- 현재의 로그 뷰어를 SQL 쿼리 가능한 탐색기로 발전
- 로그 파일을 파싱하여 SQLite/메모리 DB에 로드 후 쿼리

**11. 인라인 편집 데이터 그리드**
- PM2 환경변수, nginx 설정 등을 테이블 형식으로 편집
- React Data Grid 기반

**12. 시각적 ERD 스타일 서비스 맵**
- 실행 중인 PM2 프로세스와 포트 연결을 시각적 그래프로 표현
- 서비스 간 의존성 파악

| 맥락 유지 | 슬라이드오버, 탭으로 컨텍스트 보존 | 프로세스 목록 보면서 상세 정보 확인 |
| 키보드 효율 | Cmd+K, 단축키, 인라인 편집 | 자주 쓰는 명령에 단축키 제공 |
| 점진적 공개 | 기본 뷰는 간결, 클릭으로 상세 표시 | 로그/메트릭의 기본 뷰 단순화 |
| 상태의 시각화 | 배지, 색상, 차트로 상태 즉시 파악 | PM2 프로세스 상태, 리소스 사용량 배지화 |
| 다크 테마 일관성 | 에메랄드 액센트 + 다크 배경 | 이미 다크 테마 적용됨, 액센트 컬러 정비 |
