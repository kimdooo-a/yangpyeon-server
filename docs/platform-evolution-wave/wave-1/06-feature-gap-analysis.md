# 현재 대시보드 갭 분석

> Wave 1 리서치 문서 06  
> 작성일: 2026-04-06  
> 비교 대상: Supabase Studio, Firebase Console, Coolify, Portainer, Railway, Render

---

## 개요

현재 양평 부엌 서버 대시보드의 기능 수준을 롤모델 플랫폼 6종과 비교하여 갭을 정량화하고, 우선순위를 결정하는 분석 문서입니다. 1인 개발, WSL2+PM2 환경, 셀프호스팅이라는 제약 조건을 반드시 고려합니다.

---

## 현재 대시보드 기능 현황

```
/ (홈 대시보드)
├─ CPU 사용률 카드 + 미니 차트
├─ 메모리 사용률 카드 + 미니 차트
├─ 디스크 사용률 카드 + 미니 차트
└─ 시스템 정보 (OS, 업타임, 커널 등)

/processes
├─ PM2 프로세스 목록 테이블
├─ 재시작 / 중지 / 삭제 버튼
└─ 상세 모달 (PM2 상태 정보)

/logs
├─ PM2 로그 뷰어
├─ 검색 필터
└─ 로그 레벨 필터 (일부)

/network
└─ (미완성 — 네트워크 정보 표시 예정)

인증
└─ 비밀번호 기반 쿠키 인증
```

## 갭 분석 매트릭스

| 기능 영역 | 현재 상태 | Supabase Studio | Coolify / Portainer | 갭 크기 | 적용 난이도 |
|-----------|----------|----------------|---------------------|---------|-----------|
| **시스템 모니터링** | CPU/메모리/디스크 카드+미니차트 | Reports: API 요청, DB 연결, 캐시 히트율 시계열 | Portainer: CPU/Memory 실시간 차트, 컨테이너별 Stats | 중간 — 전체 차트 페이지 없음 | 낮음 |
| **프로세스 관리** | PM2 목록+기본 제어(재시작/중지/삭제) | Edge Functions 관리 (배포/중지/로그) | Portainer: 컨테이너 전체 생명주기 (create/run/stop/remove/inspect/exec) | 중간 — exec/inspect 없음 | 중간 |
| **로그 관리** | PM2 로그 뷰어+기본 검색 | Log Explorer: 구조화된 로그, 쿼리 빌더, 시간 범위, SQL 쿼리 | Portainer: 실시간 스트리밍, 타임스탬프, 다운로드 | 중간 — 실시간 스트리밍/시간 범위 없음 | 중간 |
| **사용자/인증 관리** | 단일 비밀번호 쿠키 인증 | Auth 관리자: 사용자 CRUD, 소셜 로그인, MFA, 세션 관리 | Coolify: 팀 멤버, 역할(Admin/Member), 초대 | 큼 — 다중 사용자, 역할 기반 접근 없음 | 높음 |
| **파일 관리** | 없음 | Storage: 버킷 관리, 파일 업로드/다운로드, 접근 정책 | Portainer: 볼륨 관리, 바인드 마운트 브라우저 | 큼 — 파일 시스템 탐색 없음 | 높음 |
| **환경변수/설정** | .env 파일 직접 편집 | Edge Function 환경변수 UI | Railway: 서비스별 환경변수 편집기 / Coolify: 환경변수 그룹 | 큼 — 웹 UI 환경변수 관리 없음 | 중간 |
| **네트워크 모니터링** | 미완성 페이지 | 해당 없음 | Portainer: 네트워크 목록/생성/삭제, 컨테이너 연결 관리 | 중간 — 기본 정보도 미완성 | 낮음 |
| **실시간 기능** | 없음 (폴링 여부 불명확) | Realtime Inspector: 채널 구독 현황, 메시지 모니터링 | Portainer: 컨테이너 Stats 실시간 스트리밍 | 큼 — 실시간 업데이트 없음 | 중간 |
| **검색/탐색** | 로그 검색만 있음 | Cmd+K 커맨드 팔레트, 글로벌 검색 | 각 페이지별 인라인 필터 | 중간 — 글로벌 검색/커맨드 팔레트 없음 | 중간 |
| **보안/감사** | 인증 로그 없음 | Auth: 로그인 이력, 보안 이벤트 로그 | Portainer: 감사 로그 (Enterprise 버전) | 큼 — 감사 로그 없음 | 중간 |
| **알림/노티피케이션** | 없음 | 없음 (외부 연동으로 해결) | Coolify: Slack/Email/Webhook 알림 | 큼 — 이상 감지 알림 없음 | 높음 |
| **API 관리** | 없음 | API 문서, API 키 관리, Rate Limit 설정 | 해당 없음 | 큼 — 하지만 현재 필요성 낮음 | 높음 |
| **데이터베이스 관리** | 없음 | Table Editor, SQL Editor, 스키마 시각화 | Portainer: DB 컨테이너 관리 | 매우 큼 — 하지만 PM2 환경에서 우선순위 낮음 | 매우 높음 |
| **배포 관리** | 없음 | Edge Function 배포 | Railway/Coolify/Render의 핵심 기능 | 큼 — 자동 배포 파이프라인 없음 | 높음 |
| **UI/UX 품질** | 기본 다크 테마, 사이드바, 카드 | Supabase 수준의 완성도 높은 UI | Portainer: 기능적이지만 디자인 기본적 | 중간 — 스켈레톤/토스트/빈상태 등 미세 UX | 낮음 |

## 기능 영역별 상세 분석

### 1. 시스템 모니터링

**현재 상태**
- CPU, 메모리, 디스크 사용률을 카드 + 미니 spark line으로 표시
- `/` 홈 페이지에서 시스템 정보(OS, 업타임, 커널 버전) 표시
- 히스토리 데이터 없음 (현재 순간 값만 표시)
- 네트워크 I/O, 디스크 I/O 모니터링 없음

**롤모델 플랫폼 수준**
- Railway: 서비스별 CPU/Memory/Network 시계열 차트, 1시간~30일 범위 선택
- Portainer: 컨테이너별 실시간 Stats (CPU%, 메모리 사용/한계, 네트워크 I/O, 디스크 I/O)
- Render: 서비스 메트릭 (CPU, Memory, Requests, Response Time) 히스토리 차트

**갭 설명**
1. 시계열 히스토리 차트 없음 — 지난 1시간/24시간 트렌드 파악 불가
2. 네트워크 I/O, 디스크 I/O 지표 없음
3. 임계값 알림 없음 — CPU 90% 이상 시 알림 없음
4. 프로세스별 메트릭 없음 — PM2 앱별 CPU/메모리 사용량 추적 불가

**해소 방안**
- 단기: `/api/metrics` 에서 히스토리 데이터를 메모리에 저장하고 Recharts 라인 차트로 시각화 (별도 `/metrics` 페이지)
- 중기: 메트릭 데이터를 SQLite 또는 파일로 영속화하여 장기 히스토리 지원
- 장기: Prometheus + Grafana 연동 (성숙한 솔루션, 복잡도 높음)

---

### 2. 프로세스 관리

**현재 상태**
- PM2 프로세스 목록 테이블 표시 (이름, 상태, PID, CPU%, 메모리, 업타임, 재시작 횟수)
- 재시작 / 중지 / 삭제 버튼 제공
- 상세 모달 (PM2 상태 정보 전체 표시)
- 프로세스 생성 기능 없음
- 필터/정렬 없음
- 벌크 액션(다중 선택) 없음

**롤모델 플랫폼 수준**
- Portainer: 컨테이너 전체 생명주기 관리
  - 생성 (이미지 선택, 포트 매핑, 환경변수, 볼륨 등 폼 기반 설정)
  - 시작/중지/재시작/일시정지/제거
  - 로그 뷰어 (실시간 스트리밍)
  - Inspect (전체 JSON 정보)
  - Stats (실시간 CPU/메모리 차트)
  - Exec (컨테이너 내 명령 실행)
  - 다중 선택 벌크 액션
  - 이름/상태/이미지 기준 필터

**갭 설명**
1. 프로세스 생성 UI 없음 — ecosystem.config.js를 직접 편집해야 함
2. 프로세스별 메트릭 차트 없음 — 현재 스냅샷만 표시
3. 다중 선택 벌크 재시작/중지 없음
4. 인라인 필터 없음 — 프로세스 수가 많아질 경우 탐색 불편
5. 프로세스 상세에서 실시간 CPU/메모리 미니 차트 없음

**해소 방안**
- 단기: 인라인 필터(이름 검색) + 벌크 선택 + 토스트 알림 추가
- 중기: 프로세스별 메트릭 히스토리 차트, 슬라이드오버 상세 패널
- 장기: 폼 기반 PM2 프로세스 생성 UI

### 3. 로그 관리

**현재 상태**
- PM2 로그 파일을 읽어 뷰어에 표시
- 기본 키워드 검색 필터
- 일부 로그 레벨 필터

**롤모델 플랫폼 수준**
- Supabase Log Explorer:
  - 프로젝트 구성 요소별 로그 선택 (API/DB/Functions/Auth/Storage)
  - SQL 쿼리로 로그 필터링
  - 시간 범위 선택 (최근 1시간~7일)
  - 로그 레벨 필터 (INFO/WARN/ERROR)
  - 실시간 로그 스트리밍
  - 로그 다운로드

- Railway 로그:
  - 서비스별 로그 선택
  - 실시간 스트리밍 (WebSocket)
  - 타임스탬프 필터
  - 키워드 검색 + 하이라이팅

- Portainer 컨테이너 로그:
  - 실시간 스트리밍
  - 타임스탬프 표시 토글
  - 로그 행 수 제한 설정
  - 다운로드 버튼

**갭 설명**
1. 실시간 로그 스트리밍 없음 — 새 로그를 보려면 페이지 새로고침 필요
2. 시간 범위 선택 없음 — 특정 시간대 로그 조회 불가
3. PM2 앱별 로그 분리 선택 없음 — 전체 로그만 표시
4. 로그 다운로드 없음
5. 로그 레벨별 색상 구분 없음 (WARN=노랑, ERROR=빨강)

**해소 방안**
- 단기: PM2 앱 선택 드롭다운 + 로그 레벨 색상 구분 + 다운로드 버튼
- 중기: SSE(Server-Sent Events)로 실시간 로그 스트리밍 구현
- 장기: 시간 범위 필터 + 로그 영속화

### 4. 사용자/인증 관리

**현재 상태**
- 단일 비밀번호 쿠키 인증 (하드코딩된 단일 비밀번호)
- 로그인/로그아웃 페이지
- 세션 관리 없음 (다중 세션, 세션 만료 등)
- 감사 로그 없음

**롤모델 플랫폼 수준**
- Supabase Auth 관리자:
  - 사용자 목록, 생성/삭제, 이메일 인증 상태
  - 소셜 로그인 공급자 설정 (Google, GitHub 등)
  - MFA 강제 설정
  - 세션 토큰 만료 시간 설정
  - 로그인 이력 조회
  
- Coolify 인증:
  - 팀 멤버 초대 (이메일)
  - 역할 관리 (Admin / Member)
  - OAuth 연동 (GitHub, GitLab)
  - 2FA 지원

**갭 설명**
1. 다중 사용자 지원 없음 — 비밀번호를 아는 모든 사람이 접근 가능
2. 역할 기반 접근 제어(RBAC) 없음 — 읽기 전용/관리자 권한 구분 불가
3. 인증 감사 로그 없음 — 누가 언제 로그인했는지 추적 불가
4. 세션 관리 없음 — 로그인 세션 목록, 강제 로그아웃 불가

**해소 방안**
- 단기: 인증 감사 로그 파일에 로그인 이벤트 기록 (`src/lib/audit-log.ts` 이미 있음)
- 중기: 다중 사용자 + 역할(Admin/Viewer) 구분 — 로컬 사용자 DB(SQLite)
- 장기: 외부 OAuth 연동 (GitHub OAuth App)

> 참고: `src/lib/audit-log.ts`가 이미 존재. 인증 감사 로그 기반이 마련되어 있음.

### 5. 파일 관리

**현재 상태**
- 없음. 파일 시스템 탐색 기능 전혀 없음.

**롤모델 플랫폼 수준**
- Supabase Storage:
  - 버킷 생성/삭제, 공개/비공개 설정
  - 파일 업로드/다운로드/이동/삭제
  - 이미지 변환 API
  - 접근 정책(RLS) 설정
  
- Portainer 볼륨:
  - Docker 볼륨 목록/생성/삭제
  - 볼륨 내 파일 브라우저 (Enterprise 버전)
  - 바인드 마운트 경로 정보

**갭 설명**
1. 서버 파일 시스템 탐색 기능 없음
2. 로그 파일 직접 다운로드 불가 (로그 뷰어로만 접근)
3. 설정 파일 (PM2 ecosystem, Nginx 등) 웹 편집 불가

**해소 방안**
- 단기: 로그 파일 다운로드 버튼 (/logs 페이지에 추가)
- 중기: 제한된 파일 브라우저 (지정된 디렉토리만 탐색 가능, 보안 고려)
- 장기: PM2 ecosystem.config.js 웹 편집기 (Monaco Editor 기반)

> 보안 고려: 웹 기반 파일 브라우저는 path traversal 공격 등 보안 위험이 높으므로 구현 시 엄격한 경로 화이트리스트 필요.

### 6. 환경변수/설정 관리

**현재 상태**
- .env 파일을 SSH 접속 후 직접 편집
- 웹 UI에서 설정 변경 기능 없음
- 설정 페이지 자체가 없음

**롤모델 플랫폼 수준**
- Railway 환경변수 편집기:
  - KEY/VALUE 테이블 편집
  - 값 마스킹/표시 토글
  - 복사 버튼
  - Raw 텍스트 붙여넣기 (bulk import)
  - 환경별 분리 (Production/Staging)
  
- Coolify 환경변수:
  - 서비스별 환경변수 그룹
  - Docker secrets 연동
  - 일반/민감값 구분

- Render 환경변수:
  - 환경변수 그룹 (여러 서비스 간 공유)
  - 민감 값 마스킹
  - 변경 이력 표시

**갭 설명**
1. 환경변수 웹 편집 UI 완전 없음 — SSH 직접 접속 필요
2. 설정 변경 이력 없음
3. 민감값/일반값 구분 없음

**해소 방안**
- 단기: `/settings` 페이지 생성 + 조회 가능한 환경변수 목록 표시 (값은 마스킹)
- 중기: 환경변수 편집 UI (추가/수정/삭제) + .env 파일 자동 업데이트 + PM2 reload
- 장기: 변경 이력 + 환경별 분리

> 보안 고려: 환경변수 편집은 인증된 사용자만 접근 가능해야 하며, `NEXT_PUBLIC_` 변수와 서버 변수를 구분하여 클라이언트 노출 방지 필요.

## 영향도 × 난이도 2×2 우선순위 매트릭스

### 분류 기준

| | **낮은 난이도** | **높은 난이도** |
|---|---|---|
| **높은 영향도** | 🟢 **Quick Win** — 즉시 적용 | 🟡 **Strategic** — 계획적 구현 |
| **낮은 영향도** | 🔵 **Easy Add** — 여유 시 적용 | ⚪ **Defer** — 후순위 |

---

### 🟢 Quick Win (높은 영향 × 낮은 난이도)

이 항목들은 구현 시간이 짧고(1–3일 이내) 사용자 경험 개선 효과가 큰 갭입니다. **즉시 적용을 권장합니다.**

| 번호 | 기능 | 갭 설명 | 예상 구현 시간 | 근거 |
|------|------|---------|--------------|------|
| QW-1 | **토스트 알림** | PM2 재시작/중지/삭제 후 성공/실패 피드백 없음 | 1–2시간 | react-hot-toast 1줄 설치로 해결 가능 |
| QW-2 | **스켈레톤 로딩** | 데이터 로딩 시 빈 화면. UX 품질 저하 | 2–4시간 | Tailwind `animate-pulse` 클래스만으로 구현 |
| QW-3 | **폴링 자동 새로고침** | CPU/메모리/프로세스 상태가 정적. 실시간 모니터링 불가 | 1–3시간 | TanStack Query `refetchInterval` 옵션 추가 |
| QW-4 | **PM2 프로세스 인라인 검색 필터** | 프로세스 수 증가 시 탐색 불편 | 2–4시간 | controlled input + 클라이언트 사이드 필터 |
| QW-5 | **로그 앱 선택 드롭다운** | 전체 로그만 표시. 특정 앱 로그 분리 불가 | 3–6시간 | PM2 앱 목록 API + 드롭다운 필터 |
| QW-6 | **로그 레벨 색상 구분** | 에러/경고가 일반 로그와 동일 색상 | 1–2시간 | 정규식으로 레벨 감지 + 색상 클래스 적용 |
| QW-7 | **접이식 사이드바** | 화면 공간 절약 불가. 작은 화면에서 불편 | 2–4시간 | CSS transition + localStorage 상태 저장 |
| QW-8 | **네트워크 페이지 완성** | 미완성 페이지가 사이드바에 노출 | 4–8시간 | `/proc/net/dev` 파싱으로 기본 정보 표시 |
| QW-9 | **감사 로그 연결** | audit-log.ts 파일 있으나 미연결 | 4–8시간 | 기존 파일 활용, PM2 액션에 훅 추가 |
| QW-10 | **브레드크럼** | 상세 페이지에서 현재 위치 파악 불가 | 1–2시간 | usePathname으로 동적 브레드크럼 생성 |

---

### 🟡 Strategic (높은 영향 × 높은 난이도)

이 항목들은 전략적 가치가 높지만 구현에 상당한 시간과 설계가 필요한 갭입니다. **계획을 수립하고 단계적으로 구현합니다.**

| 번호 | 기능 | 갭 설명 | 예상 구현 시간 | 전략적 가치 |
|------|------|---------|--------------|-----------|
| ST-1 | **메트릭 히스토리 차트 페이지** | 과거 CPU/메모리 트렌드 파악 불가. 장애 원인 분석 불가 | 1–2주 | 서버 안정성 관리의 핵심 |
| ST-2 | **실시간 로그 스트리밍(SSE)** | 로그를 보려면 새로고침 필요. 실시간 문제 추적 불가 | 3–5일 | 운영 대응 속도 향상 |
| ST-3 | **환경변수 웹 편집 UI** | SSH 접속 없이 .env 수정 불가 | 3–5일 | 운영 편의성 대폭 향상 |
| ST-4 | **커맨드 팔레트 (Cmd+K)** | 키보드 기반 탐색 불가. 숙련자 생산성 저하 | 2–3일 | 플랫폼 사용성 크게 향상 |
| ST-5 | **PM2 프로세스 메트릭 차트** | 프로세스별 CPU/메모리 추적 불가 | 1–2주 | 앱 성능 모니터링 핵심 |
| ST-6 | **다중 사용자 + 역할 기반 접근** | 단일 비밀번호로 모든 권한. 보안 취약 | 1–2주 | 보안 수준 향상 |

---

### 🔵 Easy Add (낮은 영향 × 낮은 난이도)

이 항목들은 구현이 쉽지만 즉각적인 운영 가치는 낮습니다. 여유가 있을 때 추가합니다.

| 번호 | 기능 | 갭 설명 | 예상 구현 시간 |
|------|------|---------|--------------|
| EA-1 | **로그 다운로드 버튼** | 로그 파일 다운로드 불가 | 1–2시간 |
| EA-2 | **빈 상태 개선** | 프로세스/로그 없을 때 빈 화면만 표시 | 2–4시간 |
| EA-3 | **날짜/시간 범위 선택기** | 로그 뷰어에서 시간 범위 선택 불가 | 4–8시간 |
| EA-4 | **벌크 액션 (다중 선택)** | PM2 프로세스 다중 재시작/중지 불가 | 4–8시간 |
| EA-5 | **설정 페이지 생성** | 대시보드 설정(새로고침 주기 등) 변경 UI 없음 | 4–8시간 |
| EA-6 | **JSON 트리 뷰어** | 프로세스 상세 정보가 원시 텍스트로 표시 | 2–4시간 |

---

### ⚪ Defer (낮은 영향 × 높은 난이도)

이 항목들은 현재 단일 서버 + 1인 운영 환경에서 투자 대비 가치가 낮습니다. 프로젝트 규모가 커지면 재평가합니다.

| 번호 | 기능 | 이유 |
|------|------|------|
| DF-1 | 토폴로지/캔버스 뷰 | 단일 서버에서 시각화 필요성 낮음. ReactFlow 복잡도 높음 |
| DF-2 | SQL 에디터 | PM2 서버 환경에서 DB 편집 UI 불필요 |
| DF-3 | 외부 OAuth 연동 (GitHub) | 1인 개발 환경에서 과도한 구현 |
| DF-4 | 알림 시스템 (Slack/Email) | 단독 운영자 환경에서 우선순위 낮음. 추후 고려 |
| DF-5 | 배포 자동화 파이프라인 | CI/CD 별도 도구 활용이 더 효율적 |
| DF-6 | 데이터베이스 관리 UI | Supabase/DB 서비스는 별도 관리 도구 사용 |

---

## 핵심 결론: 상위 5개 갭과 권장 적용 순서

현재 프로젝트의 주요 목적(개인 서버 운영 효율화)과 1인 개발 제약을 고려하여 가장 가치 있는 상위 5개 갭을 선정합니다.

---

### 1위: 토스트 알림 + 스켈레톤 로딩 + 폴링 새로고침 (QW-1, QW-2, QW-3)

**왜 최우선인가?**  
현재 대시보드의 모든 액션(PM2 재시작, 중지, 삭제)에 피드백이 없습니다. 사용자가 버튼을 눌렀을 때 성공했는지 실패했는지 알 수 없습니다. 또한 데이터가 항상 최신 상태가 아닙니다. 이 세 가지는 모두 구현 시간이 수 시간 이내이며, 완성된 대시보드와 미완성된 대시보드를 구분하는 핵심 UX 요소입니다.

**구현 우선순위**: 1번째 스프린트에서 처리

**구체적 작업**:
- `react-hot-toast` 또는 `sonner` 설치
- PM2 API 호출 성공/실패 시 토스트 호출
- 데이터 로딩 중 `animate-pulse` 스켈레톤 표시
- TanStack Query `refetchInterval: 5000` 설정

---

### 2위: 메트릭 히스토리 차트 페이지 (ST-1)

**왜 2위인가?**  
현재 대시보드는 "지금 이 순간"만 보여줍니다. 서버 관리에서 가장 중요한 것은 트렌드입니다. "CPU가 지난 24시간 동안 높았는가?", "특정 시간에 메모리가 급등했는가?"를 파악하려면 히스토리 차트가 필수입니다. 롤모델 플랫폼(Railway, Render, Portainer)이 모두 제공하는 핵심 기능입니다.

**구현 우선순위**: 2번째 스프린트에서 처리

**구체적 작업**:
- `/api/metrics/history` 엔드포인트 — 메모리 또는 파일 기반 시계열 데이터 저장
- `/metrics` 페이지 — Recharts를 사용한 CPU/메모리/디스크 라인 차트
- 시간 범위 프리셋 버튼 (최근 1시간, 6시간, 24시간)

---

### 3위: 실시간 로그 스트리밍 (ST-2)

**왜 3위인가?**  
현재 /logs 페이지는 정적 스냅샷입니다. 서버에서 문제가 발생했을 때 로그를 실시간으로 추적하는 것은 운영의 기본입니다. SSE는 WebSocket보다 구현이 간단하고 Next.js Route Handler에서 바로 지원됩니다. 이 기능이 구현되면 SSH 접속 없이 웹 브라우저만으로 서버 문제를 진단할 수 있습니다.

**구현 우선순위**: 2번째 스프린트에서 처리 (메트릭 차트와 병행)

**구체적 작업**:
- `/api/logs/stream` Route Handler — `pm2 logs --raw` 출력을 SSE로 전달
- 클라이언트에서 `EventSource` 구독
- 자동 스크롤(하단 고정) + 일시정지 버튼

---

### 4위: 환경변수 웹 편집 UI (ST-3)

**왜 4위인가?**  
현재 서버 설정 변경은 반드시 SSH 접속이 필요합니다. 웹 대시보드의 핵심 가치는 "어디서나 브라우저만으로 서버를 관리"하는 것인데, 환경변수 변경을 위해 SSH를 열어야 한다면 이 가치가 절반으로 줄어듭니다. Railway, Render, Coolify가 모두 제공하는 기본 기능입니다.

**구현 우선순위**: 3번째 스프린트에서 처리

**구체적 작업**:
- `/settings/env` 페이지 — 환경변수 목록 표시 (값 마스킹)
- KEY/VALUE 편집 폼 + 저장 시 .env 파일 업데이트
- 저장 후 PM2 reload 자동 실행
- 보안: Admin 역할만 접근 가능

---

### 5위: 커맨드 팔레트 Cmd+K (ST-4)

**왜 5위인가?**  
커맨드 팔레트는 대시보드를 일반 도구에서 전문가 도구로 격상시키는 기능입니다. `cmdk` 라이브러리를 사용하면 3–4일 안에 구현 가능하며, 이후 새 기능(페이지 이동, PM2 액션, 설정 변경 등)을 팔레트에 추가하는 것이 매우 쉬워집니다. 숙련 사용자 생산성을 크게 향상시킵니다.

**구현 우선순위**: 3번째 스프린트에서 처리

**구체적 작업**:
- `cmdk` 라이브러리 설치
- 글로벌 `Cmd+K` 키보드 단축키 바인딩
- 초기 액션 목록: 페이지 이동(6개) + PM2 전체 재시작/중지 + 다크/라이트 테마 토글

## 권장 구현 로드맵

```
[스프린트 1 — 즉시] (~1주)
├─ QW-1: 토스트 알림
├─ QW-2: 스켈레톤 로딩
├─ QW-3: 폴링 자동 새로고침 (5초 주기)
├─ QW-4: PM2 프로세스 인라인 검색 필터
├─ QW-6: 로그 레벨 색상 구분
└─ QW-8: 네트워크 페이지 완성

[스프린트 2 — 단기] (~2–3주)
├─ ST-1: 메트릭 히스토리 차트 페이지 (/metrics)
├─ ST-2: 실시간 로그 스트리밍 (SSE)
├─ QW-5: 로그 앱 선택 드롭다운
├─ QW-9: 감사 로그 연결
├─ EA-1: 로그 다운로드 버튼
└─ EA-3: 날짜/시간 범위 선택기

[스프린트 3 — 중기] (~1개월)
├─ ST-3: 환경변수 웹 편집 UI (/settings/env)
├─ ST-4: 커맨드 팔레트 (Cmd+K)
├─ ST-5: PM2 프로세스별 메트릭 차트
├─ EA-4: 벌크 액션 (다중 선택)
└─ EA-5: 설정 페이지 생성

[스프린트 4 — 장기 / 필요 시]
├─ ST-6: 다중 사용자 + RBAC
├─ 파일 브라우저 (보안 고려)
└─ 알림 시스템 (Slack/Email)
```

## 요약

현재 대시보드는 기본 모니터링 + PM2 관리 기능을 갖추고 있으나, 운영 대시보드로서 완성도를 높이기 위해 아래 세 가지 축에서 개선이 필요합니다.

1. **UX 완성도** (즉시 가능): 토스트 알림, 스켈레톤, 자동 새로고침 — 현재 대시보드가 "완성된 제품"처럼 느껴지게 하는 필수 요소
2. **모니터링 깊이** (단기): 히스토리 차트, 실시간 로그 스트리밍 — 실제 운영에서 문제를 진단하고 추적하는 핵심 기능
3. **운영 편의성** (중기): 환경변수 편집, 커맨드 팔레트 — SSH 없이 브라우저만으로 서버를 완전히 관리하는 목표 달성
