## Part 1: 비전

### 1.1 프로젝트 비전 선언문

> **"집에서 운영하는 서버도 프로덕션처럼 — 양평 부엌의 모든 데이터와 프로세스를 하나의 대시보드에서 안전하게 관리한다."**

이 한 문장이 모든 기능 결정의 기준이 된다. 기능을 추가하거나 제거할 때 "이것이 이 선언문을 달성하는 데 기여하는가?"를 묻는다.

### 1.3 핵심 가치 3가지

**가치 1: 운영 단순성 (Operational Simplicity)**
> "도구가 복잡하면 운영자가 된다, 단순하면 창작자가 된다."

- 1인 운영자가 SSH 없이 모든 관리 작업을 브라우저에서 수행 가능해야 한다.
- 새 기능 추가 시 항상 "이것이 운영 복잡도를 높이는가?"를 묻는다.
- 설정 파일, 환경변수, 마이그레이션 — 모두 UI에서 가능해야 한다.

**가치 2: 데이터 신뢰성 (Data Reliability)**
> "재시작 후에도 기록은 살아있어야 한다."

- 감사 로그, 메트릭 히스토리, 설정값은 서버 재시작 후에도 유지된다.
- 백업은 자동화하고, 복구는 단순해야 한다 (파일 하나 복사).
- 모든 데이터 변경은 추적 가능하다.

**가치 3: 보안 우선 (Security First)**
> "홈 서버라도 프로덕션 수준의 보안을 갖춘다."

- 모든 관리 액션은 감사 로그에 기록된다.
- 인증 없이 접근 가능한 API 엔드포인트는 없다.
- 사용자 입력은 서버 사이드에서 검증된다 (Zod).

| 속성 | 설명 |
|------|------|
| 역할 | 프로젝트 협업자, 제한적 접근 권한 |

### 1.5 안티 스코프 — 이것은 하지 않는다

아래 항목은 의도적으로 제외된다. 이를 요청받더라도 "프로젝트 범위 외"로 거절한다.

| 안티 스코프 항목 | 제외 이유 |
|----------------|---------|
| **다중 서버 관리** | 1인 단일 서버 운영 도구. 복수 서버 관리는 Portainer/Coolify 영역 |
| **클라우드 DB 연동 (Supabase, PlanetScale 등)** | 셀프호스팅 원칙. 외부 서비스 의존 최소화 |
| **Docker/컨테이너 관리** | PM2 기반 운영. 컨테이너화는 별개 프로젝트 |
| **CI/CD 파이프라인** | 배포 자동화는 현재 수동 스크립트로 충분 |
| **팀 협업 기능 (다중 사용자 역할)** | 단일 관리자. 역할 기반 접근제어는 P2 이하 |
| **공개 API 제공** | 내부 관리 도구. 외부 API 게이트웨이 역할 아님 |
| **상용 클라우드 배포 (Vercel, Fly.io 등)** | WSL2 + PM2 + Cloudflare Tunnel이 목표 환경 |
| **AI/ML 기능** | 범위 외. 운영 도구에 AI 기능 추가는 복잡도만 증가 |
| **소셜 로그인 (OAuth)** | 단일 관리자에게 소셜 로그인은 불필요한 복잡도 |
| **알림 SMS/전화** | 이메일/Slack 알림까지만 (P2). SMS/전화는 범위 외 |

## Part 2: 사용자 스토리 (핵심)

각 사용자 스토리는 다음 형식으로 작성된다:
> **"[역할]로서, [목표]를 원한다. [이유]."**
> 수용 기준: [완료 조건 목록]

---

### US-01: 메트릭 히스토리 차트

> **"서버 운영자로서, 지난 1시간 CPU·메모리 사용률을 시계열 차트로 보고 싶다. 현재는 순간값만 보여서 트렌드를 파악할 수 없기 때문이다."**

**수용 기준**
- [ ] 메트릭 히스토리가 SQLite에 저장된다 (최소 24시간 보존)
- [ ] 홈 대시보드에 CPU, 메모리 시계열 라인 차트가 표시된다
- [ ] 시간 범위 선택: 최근 1시간 / 6시간 / 24시간
- [ ] 차트는 자동으로 최신 데이터로 갱신된다 (SSE 또는 폴링)
- [ ] 서버 재시작 후에도 이전 데이터가 유지된다

**우선순위**: P0  
**연관 기술**: SQLite, SSE, Recharts 또는 Chart.js

---

### US-02: 감사 로그 영속화

> **"서버 운영자로서, 누가 언제 어떤 관리 액션을 했는지 기록이 서버 재시작 후에도 남아 있길 원한다. 현재는 인메모리라 재시작하면 모든 기록이 사라지기 때문이다."**

**수용 기준**
- [ ] 감사 로그가 SQLite `audit_logs` 테이블에 기록된다
- [ ] 기록 항목: 액션 유형, 대상, IP 주소, 타임스탬프, 결과(성공/실패)
- [ ] 서버 재시작 후에도 로그가 보존된다
- [ ] /audit 페이지에서 날짜/액션 유형 필터로 검색 가능
- [ ] 로그 내보내기 (CSV 다운로드) 가능

**우선순위**: P0  
**연관 기술**: SQLite, Drizzle, TanStack Table

---

### US-03: 웹 기반 데이터베이스 편집

> **"서버 운영자로서, 로컬 SQLite DB 테이블을 SSH 없이 브라우저에서 직접 조회하고 편집하고 싶다. 지금은 DB를 확인하려면 SSH로 접속해서 CLI를 써야 하기 때문이다."**

**수용 기준**
- [ ] /database 페이지에서 SQLite DB의 테이블 목록이 표시된다
- [ ] 테이블 클릭 시 TanStack Table로 데이터 조회 가능
- [ ] 셀 인라인 편집 가능 (더블클릭 → 수정 → 저장)
- [ ] 행 삭제 시 확인 모달 표시
- [ ] 편집 액션은 감사 로그에 기록됨
- [ ] 조회 전용 모드와 편집 모드 구분 가능

**우선순위**: P1  
**연관 기술**: SQLite, Drizzle, TanStack Table, shadcn/ui

---

### US-04: SQL 쿼리 에디터

> **"서버 운영자로서, 브라우저에서 직접 SQL 쿼리를 작성하고 실행하여 결과를 테이블로 확인하고 싶다. 복잡한 데이터 조회 시 CLI 없이 처리하기 위해서다."**

**수용 기준**
- [ ] /database/sql 페이지에 Monaco Editor 기반 SQL 에디터가 있다
- [ ] SQL 구문 강조 및 기본 자동완성 (테이블명, 컬럼명)
- [ ] 쿼리 실행 버튼 (Ctrl+Enter 단축키)
- [ ] 결과가 TanStack Table로 표시된다
- [ ] 최근 쿼리 히스토리 저장 (로컬스토리지 또는 DB)
- [ ] 에러 메시지 명확히 표시
- [ ] SELECT만 허용 (INSERT/UPDATE/DELETE는 별도 확인 모달 필요)

**우선순위**: P1  
**연관 기술**: Monaco Editor, SQLite, Drizzle, TanStack Table

---

### US-05: PM2 로그 실시간 스트리밍

> **"서버 운영자로서, /logs 페이지에서 PM2 로그가 실시간으로 스트리밍되어 새로고침 없이 확인하고 싶다. 현재는 로그를 보려면 페이지를 수동으로 새로고침해야 하기 때문이다."**

**수용 기준**
- [ ] /logs 페이지에서 SSE 연결 후 로그가 실시간으로 append 됨
- [ ] 연결 상태 표시기 (연결됨 / 끊김 / 재연결 중)
- [ ] 로그 일시정지/재개 버튼 (스크롤 중 멈추기)
- [ ] 새 로그 도착 시 자동 스크롤 (하단에 있을 때만)
- [ ] 최대 1000줄 유지 후 오래된 로그 제거 (메모리 관리)
- [ ] 연결 끊김 시 3초 후 자동 재연결

**우선순위**: P0  
**연관 기술**: SSE, Next.js Route Handler, EventSource API

---

### US-06: 환경변수 웹 편집기

> **"서버 운영자로서, PM2 프로세스의 환경변수를 브라우저에서 확인하고 수정하고 싶다. 현재는 .env 파일을 직접 수정하고 PM2를 재시작해야 하기 때문이다."**

**수용 기준**
- [ ] /processes/[id]/env 에서 해당 프로세스 환경변수 목록 표시
- [ ] KEY=VALUE 형식 편집기 (react-simple-code-editor 기반)
- [ ] 저장 시 .env 파일 업데이트 + PM2 reload 자동 실행
- [ ] 민감한 값(PASSWORD, SECRET, KEY 포함 키)은 마스킹 표시
- [ ] 변경 전 확인 모달 표시
- [ ] 변경 내역 감사 로그 기록 (값은 마스킹)

**우선순위**: P1  
**연관 기술**: PM2 API, react-simple-code-editor, 감사 로그

---

### US-07: 시스템 메트릭 자동 갱신

> **"서버 운영자로서, 홈 대시보드의 CPU·메모리·디스크 수치가 자동으로 갱신되길 원한다. 현재는 수동 새로고침 없이는 최신 값을 볼 수 없기 때문이다."**

**수용 기준**
- [ ] 메트릭 카드가 5초마다 자동 갱신된다 (SSE 또는 폴링)
- [ ] 갱신 중 로딩 표시 없음 (이전 값 유지 상태에서 조용히 업데이트)
- [ ] 갱신 실패 시 카드에 오류 상태 표시 (재시도 버튼)
- [ ] 페이지 비활성화(탭 숨김) 시 갱신 중지 (배터리/서버 부하 절약)

**우선순위**: P0  
**연관 기술**: SSE, Page Visibility API

---

### US-08: 네트워크 정보 완성

> **"서버 운영자로서, 서버의 네트워크 인터페이스, 외부 IP, Cloudflare Tunnel 연결 상태를 한 페이지에서 확인하고 싶다. /network 페이지가 현재 미완성이기 때문이다."**

**수용 기준**
- [ ] /network 페이지에서 네트워크 인터페이스 목록 표시 (이름, IP, MAC)
- [ ] 외부 IP 주소 표시 (IP 조회 API 활용)
- [ ] Cloudflare Tunnel 상태 표시 (연결됨 / 끊김)
- [ ] 네트워크 트래픽 현황 표시 (수신/송신 바이트)
- [ ] 활성 TCP 연결 수 표시

**우선순위**: P0  
**연관 기술**: systeminformation, Cloudflare API

---

### US-09: 모바일에서 긴급 재시작

> **"서버 운영자로서, 외출 중 모바일에서 PM2 프로세스를 긴급 재시작할 수 있어야 한다. 이슈 발생 시 노트북 없이도 신속하게 대응하기 위해서다."**

**수용 기준**
- [ ] /processes 페이지가 모바일(375px)에서 정상 동작
- [ ] 재시작 버튼이 모바일에서 쉽게 탭 가능 (최소 44px 터치 영역)
