> 목적: Wave 1에서 발견된 모든 기능 후보를 체계적으로 평가하여 1인 개발자 관점의 구현 우선순위를 결정한다.

## 1. 개요

Wave 1 갭 분석에서 15개 영역, 약 40여 개의 기능 후보가 식별되었다. 이 매트릭스는 그 중 대표적인 30개 기능을 5가지 기준으로 점수화하여 구현 순서를 결정하는 로드맵 문서이다.

### 1.1 평가 원칙

이 프로젝트의 제약 조건을 반드시 반영한다.

- **1인 개발**: 구현 난이도와 유지보수 비용이 가장 중요한 변수다.
- **WSL2 + PM2 + Cloudflare Tunnel**: Docker/Kubernetes 없는 단순한 Node.js 프로세스 관리 환경이다.
- **셀프호스팅**: 외부 SaaS 의존성을 최소화한다. 운영 부담이 낮은 기술 스택을 선호한다.
- **현재 목적**: 서버 운영자(본인 1명)를 위한 운영 도구이다. 멀티테넌시, 상업적 기능은 후순위다.
- **인메모리 한계**: 감사 로그 휘발, 설정 재시작 시 초기화가 현재 가장 큰 운영 고통이다.

### 1.2 평가 기준 정의

| 기준 | 코드 | 가중치 | 설명 |
|------|------|--------|------|
| 사용자 가치 (User Value) | V | 30% | 운영자에게 얼마나 즉각적인 유용함을 제공하는가 (5=매우 높음, 1=낮음) |
| 구현 난이도 (Effort) | E | 25% | 1인 개발자가 구현하는 데 드는 시간/복잡도 (5=쉬움, 1=매우 어려움) |
| 기술 위험도 (Risk) | R | 15% | 미지의 기술, 복잡한 통합, 보안 이슈 (5=위험 없음, 1=매우 위험) |
| 기반 효과 (Foundation) | F | 20% | 다른 기능 구현의 전제가 되는가, 기술 부채 해소 (5=높음, 1=낮음) |
| 현재 갭 크기 (Gap) | G | 10% | 현재 대비 롤모델과의 차이가 크고 운영 고통이 큰가 (5=큼, 1=없음) |

**종합 점수 산출식**:  
`Score = V×0.30 + E×0.25 + R×0.15 + F×0.20 + G×0.10`

### 4.2 상위 10개 기능 선정 이유

**G-02 Zod 입력 검증** (점수: 4.55)
- 이미 구현 진행 중이며(세션 4 잔여 작업), 보안 기반이다.
- 모든 API에 적용하면 코드 일관성이 높아지고 예상치 못한 입력으로 인한 서버 오류를 방지한다.
- 구현 난이도가 낮고(E=4) 위험도가 없어(R=5) 즉시 착수하기 최적이다.

**F-02 토스트 알림** (점수: 4.55)
- PM2 재시작, 프로세스 중지 등의 액션 후 피드백이 현재 전혀 없다.
- `sonner` 또는 `react-hot-toast` 라이브러리 설치 후 1~2시간 작업으로 전체 UX가 크게 개선된다.
- 다른 기능에 의존하지 않으며, 완성된 기능처럼 느껴지게 하는 "폴리시(polish)" 효과가 크다.

**A-02 감사 로그 영속화** (점수: 4.35)
- 현재 가장 큰 운영 고통: PM2 재시작 시 모든 감사 로그가 초기화된다.
- `src/lib/audit-log.ts`가 이미 존재하므로, SQLite 파일로 저장 대상만 변경하면 된다.
- A-01(SQLite 도입)과 병행 구현 가능하다.

**D-02 고급 필터** (점수: 4.25)
- 로그 페이지 기능 개선으로, 현재 기본 키워드 검색만 있는 상태에서 큰 도약이다.
- PM2 앱 선택 드롭다운 + 로그 레벨 필터 개선은 프론트엔드만 수정하면 된다.
- 백엔드 변경 없이 단계적으로 구현 가능하다.

**B-01 시계열 히스토리 차트** (점수: 4.15)
- 운영자가 "CPU가 어제 오전 3시에 왜 높았지?"를 알 수 없는 현재 상태를 해결한다.
- Recharts는 이미 설치되어 있어 추가 의존성이 없다.
- 히스토리는 인메모리 순환 버퍼(최근 1시간, 60개 데이터 포인트)로 시작하면 DB 없이도 구현 가능하다.

**E-01 감사 로그 UI** (점수: 4.10)
- A-02 완성 후 자연스럽게 이어지는 기능이다.
- 로그인 이벤트, API 호출 이벤트를 테이블로 조회할 수 있는 페이지 추가.
- 보안 감사 목적으로 "누가 언제 무엇을 했는지" 추적이 가능해진다.

**D-03 로그 파일 다운로드** (점수: 4.05)
- 구현이 가장 단순한 기능 중 하나다(E=5).
- `/api/logs/download` 엔드포인트 + 버튼 하나로 완성된다.
- 장애 분석 시 로그를 로컬에 저장해 편집기로 열 수 있어 운영 편의성이 크다.

**D-01 실시간 로그 스트리밍** (점수: 4.00)
- 현재 로그를 보려면 매번 페이지를 새로고침해야 한다.
- Server-Sent Events(SSE)는 WebSocket보다 단순하고 Next.js Route Handler에서 기본 지원된다.
- `tail -f` 방식으로 구현하면 실시간 스트리밍이 가능하다.

**A-01 SQLite DB 도입** (점수: 3.90)
- 단독으로는 사용자에게 직접 가치를 제공하지 않지만, A-02/A-03/A-04/E-01/G-03의 전제 기반이다.
- `better-sqlite3` 설치 + 스키마 초기화 스크립트 작성으로 1~2시간이면 완료된다.
- Foundation 점수가 5점인 이유: 이 기능 없이는 영속화 관련 기능을 전혀 구현할 수 없다.

**G-01 IP 화이트리스트** (점수: 3.85)
- 현재 미들웨어(`src/middleware.ts`)에 IP 필터링 로직을 추가하면 된다.
- 셀프호스팅 환경에서 Cloudflare Tunnel 뒤에 있더라도 내부 네트워크에서의 접근 제어가 필요하다.
- 구현이 단순하고(E=4) 보안 기반이 강화된다.

## 5. 구현 순서 제안 (Phase 1~3)

### 5.1 Phase 1: Quick Win + 기반 구축 (예상: 1~2주)

**목표**: 즉각적인 UX 개선 + 데이터 영속화 기반 마련

| 순서 | ID | 기능명 | 예상 소요 | 완료 기준 |
|------|----|--------|---------|---------|
| 1 | G-02 | Zod 입력 검증 전체 API 적용 | 2~3시간 | 모든 API route에 Zod 스키마 적용 완료 |
| 2 | F-02 | 토스트 알림 | 1~2시간 | PM2 액션 후 성공/실패 토스트 표시 |
| 3 | D-03 | 로그 파일 다운로드 | 1시간 | /logs 페이지에 다운로드 버튼 동작 |
| 4 | D-02 | 고급 필터 개선 | 2~3시간 | 앱별 선택 드롭다운 + 레벨 색상 구분 |
| 5 | A-01 | SQLite DB 도입 | 1~2시간 | DB 파일 초기화 + 연결 유틸리티 완성 |
| 6 | A-02 | 감사 로그 영속화 | 2~3시간 | PM2 재시작 후에도 로그 유지 확인 |
| 7 | G-01 | IP 화이트리스트 | 1~2시간 | middleware에서 허용 IP 외 차단 동작 |

- 운영자가 서버 재시작 없이 감사 로그를 영구 보존할 수 있다.
- 모든 API 입력이 검증되어 예상치 못한 서버 오류가 줄어든다.
- 로그 다운로드와 필터 개선으로 장애 분석 효율이 높아진다.

## 7. 장기 후보 기능 (Phase 4+)

점수가 낮거나 복잡도가 매우 높아 단기 계획에서 제외한 기능들.

### 7.1 A-06 SQL Editor

**이유**: 웹에서 임의의 SQL을 실행할 수 있는 인터페이스는 매우 높은 보안 위험을 내포한다. SQL 인젝션 방어뿐만 아니라 DROP TABLE, UPDATE 권한 제어가 복잡하다. Monaco Editor + 샌드박스 쿼리 실행 레이어가 필요하다.

**구현 조건**: A-01 완성 + 읽기 전용 권한 분리 + 허용 쿼리 화이트리스트 + 쿼리 결과 페이지네이션이 선행되어야 한다.

### 7.2 A-07 스키마 시각화 (ERD)

**이유**: `react-flow` 기반의 ERD 렌더링은 구현 자체보다 스키마 파싱과 관계 추출 로직이 복잡하다. 1인 개발 기준 20~30시간의 작업량이 예상된다.

**구현 조건**: A-01 + A-05 완성 후 DB 스키마가 충분히 성숙했을 때 구현한다.

### 7.3 C-03 PM2 프로세스 생성 폼

**이유**: `ecosystem.config.js` 형식을 폼으로 추상화하는 것은 PM2 API 이해, 폼 유효성 검사, 재시작 로직이 얽혀 복잡하다. 현재는 1~3개 프로세스 운영이므로 필요성이 낮다.

### 7.4 E-04 2FA (TOTP)

**이유**: `speakeasy` 또는 `otplib` 라이브러리로 구현은 가능하지만, QR 코드 등록 플로우와 백업 코드 관리를 포함하면 E-02(다중 사용자)보다 후순위다. 현재 1인 운영 환경에서 우선순위가 낮다.

### 7.5 파일 관리 / 파일 브라우저

**이유**: 웹 기반 파일 브라우저는 path traversal 공격 등 심각한 보안 위험을 가진다. 엄격한 경로 화이트리스트, 심볼릭 링크 방어, 파일 크기 제한 등 보안 레이어가 필수다. 충분한 보안 설계 없이 구현하면 서버 전체 노출 위험이 있다.

**구현 조건**: 철저한 보안 감사 후 특정 디렉토리(예: `~/dashboard/logs/`)만 읽기 전용으로 노출하는 방식으로 한정한다.

## 8. 의사결정 기록

### 8.1 SQLite를 PostgreSQL보다 우선한 이유

- WSL2 단일 서버 환경에서 PostgreSQL 데몬을 추가 운영하는 것은 불필요한 복잡도다.
- SQLite는 파일 하나로 데이터베이스가 완성되며, PM2 재시작 후에도 데이터가 유지된다.
- `better-sqlite3`는 동기 API를 제공하여 Next.js Route Handler에서 사용하기 간단하다.
- 추후 규모가 커지면 `libsql`(Turso) 또는 PostgreSQL로 마이그레이션 가능하도록 추상화 레이어를 설계한다.

### 8.2 Drizzle을 Prisma보다 선호한 이유

- Prisma는 별도의 `prisma generate` 빌드 스텝이 필요하고, 번들 크기가 크다.
- Drizzle은 TypeScript 타입 추론이 뛰어나고, `better-sqlite3`와의 통합이 더 자연스럽다.
- SQL-like API를 제공하여 SQL을 이해하는 개발자에게 학습 비용이 낮다.
- 상세 내용은 `02-db-management-ui.md` 참조.

### 8.3 Phase 1에 B-01(히스토리 차트)이 아닌 A-01(DB 도입)을 넣은 이유

- B-01은 인메모리 순환 버퍼로도 구현 가능하지만, 재시작 시 모든 히스토리가 사라진다.
- 운영자 입장에서 "재시작 후 빈 차트"는 오히려 혼란을 줄 수 있다.
- A-01 → A-03 → B-01 순서로 구현해야 "재시작해도 유지되는 히스토리 차트"를 만들 수 있다.

### 8.4 감사 로그 영속화를 최우선 과제로 선정한 이유

- 현재 `src/lib/audit-log.ts`는 인메모리 배열에 로그를 저장한다.
- PM2 재시작(배포 포함) 시 모든 로그가 초기화된다.
- 보안 목적(누가 언제 접근했는지)과 운영 목적(어떤 액션이 있었는지) 모두에서 영속화가 필수다.
- 코드베이스에 이미 기반이 마련되어 있어(A-01 도입 후) 변경 범위가 최소화된다.

## 10. 위험 관리 계획

### 10.1 식별된 기술 위험

| 위험 항목 | 가능성 | 영향 | 대응 방안 |
|---------|------|------|---------|
| SQLite WAL 모드에서 파일 락 충돌 | 낮음 | 중간 | PM2 앱 1개 환경이므로 단일 쓰기 프로세스 보장됨 |
| SSE 연결이 많아지면 메모리 누수 | 낮음 | 낮음 | 연결 수 제한 + 유휴 타임아웃 구현 |
| 메트릭 히스토리 DB 크기 증가 | 중간 | 낮음 | 30일 이상 자동 삭제 + DB 크기 모니터링 |
| PM2 환경변수 편집 시 민감 정보 노출 | 중간 | 높음 | Admin 역할 + HTTPS 강제 + 값 마스킹 |
| 파일 브라우저 path traversal 공격 | 높음 | 매우 높음 | 구현 전 철저한 보안 설계 필수. Phase 4+로 연기 |
| Monaco Editor 번들 크기 (약 1MB+) | 확실 | 낮음 | 동적 import + 코드 스플리팅으로 초기 로딩 시간 최소화 |
