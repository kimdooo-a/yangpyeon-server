# 기술 스택 비교 — 플랫폼 진화를 위한 기술 선택

> Wave 2+3 리서치 문서 06  
> 작성일: 2026-04-06  
> 작성 맥락: 양평 부엌 서버 대시보드 → Supabase-like 플랫폼 진화  
> 기준: 1인 개발 · WSL2 Ubuntu · PM2 · Cloudflare Tunnel · Next.js 16

---

## 개요

현재 대시보드는 인메모리 데이터만 사용하며 재시작 시 모든 상태가 사라진다. 플랫폼으로 진화하기 위해서는 영속성 있는 데이터 계층, 풍부한 UI 컴포넌트, 실시간 통신 등 여러 기술 결정을 내려야 한다.

이 문서는 각 기술 영역별로 후보군을 비교하고, **1인 개발자가 WSL2 셀프호스팅 환경에서 장기 운영**할 수 있는 최적 스택을 도출한다.

### 평가 기준 정의

| 기준 | 설명 | 가중치 |
|------|------|--------|
| **성능** | 쿼리 속도, 처리량, 메모리 풋프린트 | 15% |
| **운영 편의** | 설치·설정·백업·모니터링의 간단함 | 30% |
| **기능** | 필요한 기능의 지원 여부 | 20% |
| **생태계** | 문서, 커뮤니티, 패키지 성숙도 | 15% |
| **1인 적합도** | 러닝커브, 유지보수 비용, 인지 부하 | 20% |

점수는 1(최하) ~ 5(최고). 가중 합계로 최종 점수 산출.

## 카테고리 1: 데이터베이스

### 배경

현재 감사 로그, 메트릭 히스토리, 세션 데이터 등 영속화해야 할 데이터가 증가하고 있다. 인메모리 저장은 PM2 재시작 시 전부 소실되므로 실용성 없다. 어떤 데이터베이스를 선택하느냐는 이후 ORM, 백업, 마이그레이션 전략 전체에 영향을 미친다.

### 후보 1: SQLite (better-sqlite3)

**개요**
- 파일 기반 관계형 DB. 서버 프로세스 불필요, 라이브러리로 직접 임베드.
- better-sqlite3는 Node.js용 동기 바인딩으로 성능이 매우 우수.

**장점**
- 제로 설정: `npm install better-sqlite3` 후 파일 경로만 지정하면 바로 사용.
- 파일 하나(`dashboard.db`)가 전체 DB — 백업이 `cp` 명령 하나.
- WSL2 환경에서 별도 서비스 데몬 없이 동작.
- 쿼리 속도: WAL 모드에서 읽기 100k qps 이상 (단일 서버 모니터링 용도에 충분).
- TypeScript와 완벽 통합. better-sqlite3는 동기 API라 async/await 없이 단순.
- Prisma, Drizzle 모두 SQLite 백엔드 지원.

**단점**
- 동시 쓰기: WAL 모드에서도 단일 라이터. 대규모 동시 쓰기 불가.
- HTTP 원격 접근 없음 (단일 프로세스에서만 접근 가능).
- 컬럼 타입 변경 등 일부 DDL 제한.

**적합도 평가**
- 1인 운영자 단일 Next.js 프로세스: 동시 쓰기 제한 문제없음.
- 모니터링 데이터(메트릭 히스토리, 감사 로그)는 순차 삽입 위주.

### 후보 2: PostgreSQL

**개요**
- 엔터프라이즈급 관계형 DB. Supabase의 원본 데이터베이스.
- WSL2에 설치 가능하지만 별도 서비스 데몬 필요.

**장점**
- 기능 완전성: JSON 컬럼, 전문 검색, 윈도우 함수, CTE, 파티셔닝.
- Supabase 마이그레이션 옵션 보존: 언젠가 Supabase로 마이그레이션 시 호환성.
- 동시 쓰기 지원, MVCC.
- Prisma 공식 지원 최우선.

**단점**
- 운영 부담: 별도 프로세스 관리, pg_hba.conf, 사용자 권한, WAL 설정.
- WSL2 재시작 시 PostgreSQL 서비스도 재시작 필요.
- 메모리: 최소 50-100MB 상시 점유.
- 백업: pg_dump 필요 (SQLite cp와 비교해 복잡).
- 1인 개발자에게 운영 오버헤드가 상당함.

### 후보 3: Turso (libSQL)

**개요**
- SQLite 호환 프로토콜 + HTTP/WebSocket 접근 지원.
- 로컬 파일 모드와 원격 클라우드 모드 모두 지원.

**장점**
- SQLite 호환: 기존 SQLite 코드 거의 그대로 사용 가능.
- HTTP API: 동일 프로세스 외부에서도 접근 가능.
- 클라우드 Turso 서비스를 쓰면 복제·백업 자동화.
- Drizzle ORM과 네이티브 통합.

**단점**
- 로컬 모드: SQLite 대비 이점이 거의 없음 (오히려 의존성 추가).
- 클라우드 모드: 외부 서비스 의존성 (셀프호스팅 원칙에 반함).
- 생태계: SQLite, PostgreSQL 대비 작음. 문서도 아직 발전 중.
- @libsql/client 패키지 추가 설치 필요.

### 후보 4: LowDB

**개요**
- JSON 파일 기반 데이터 저장소. TypeScript 네이티브.
- 실제 DB가 아니라 "JSON을 파일에 저장하는 편의 라이브러리".

**장점**
- 초경량: 의존성 거의 없음.
- JSON 파일 직접 읽기 가능 → 디버깅 초간단.
- TypeScript 제네릭으로 타입 안전 데이터 접근.
- 설정 제로.

**단점**
- 쿼리 성능: 전체 JSON 파일 메모리 로드 후 JS로 필터링 → 데이터 수천 건 넘으면 느려짐.
- 인덱스 없음 → 감사 로그 수만 건에서 검색 불가.
- 관계 데이터 표현 어려움 (JOIN 없음).
- 트랜잭션 없음.
- 진지한 데이터 관리 도구로는 부적합.

### 데이터베이스 점수 매트릭스

| 기준 (가중치) | SQLite | PostgreSQL | Turso | LowDB |
|--------------|--------|-----------|-------|-------|
| 성능 (15%) | 5 | 5 | 4 | 2 |
| 운영 편의 (30%) | 5 | 2 | 3 | 5 |
| 기능 (20%) | 4 | 5 | 4 | 2 |
| 생태계 (15%) | 5 | 5 | 3 | 3 |
| 1인 적합도 (20%) | 5 | 2 | 3 | 3 |

**가중 점수 계산**

| 후보 | 계산식 | 최종 점수 |
|------|--------|---------|
| SQLite | 5×0.15 + 5×0.30 + 4×0.20 + 5×0.15 + 5×0.20 | **4.80** |
| PostgreSQL | 5×0.15 + 2×0.30 + 5×0.20 + 5×0.15 + 2×0.20 | **3.50** |
| Turso | 4×0.15 + 3×0.30 + 4×0.20 + 3×0.15 + 3×0.20 | **3.30** |
| LowDB | 2×0.15 + 5×0.30 + 2×0.20 + 3×0.15 + 3×0.20 | **3.25** |

### 카테고리 1 최종 추천

> **SQLite (better-sqlite3)** — 4.80점

**근거**
1. WSL2 단일 프로세스 환경에서 "제로 운영 부담"은 핵심 가치다.
2. 감사 로그, 메트릭 히스토리, 세션 데이터는 수십만 건 수준 — SQLite WAL 모드로 충분.
3. 백업이 파일 복사 한 줄. PM2 크론잡으로 자동화 가능.
4. Drizzle + SQLite 조합은 타입 안전 + 경량 + 마이그레이션을 모두 만족.
5. 나중에 PostgreSQL 마이그레이션이 필요하다면 Drizzle 스키마 재사용 가능.

## 카테고리 2: ORM / 쿼리빌더

### 배경

SQLite를 선택한 후 어떤 방식으로 쿼리를 작성하고 스키마를 관리할지 결정해야 한다. 타입 안전성, 마이그레이션 관리, 러닝커브가 핵심 평가 항목이다.

### 후보 1: Prisma

**개요**
- 스키마 퍼스트 ORM. `schema.prisma` 파일에 모델 정의 → 자동 타입 생성.
- 가장 널리 사용되는 Node.js ORM.

**장점**
- 타입 안전: PrismaClient 자동 생성으로 완전한 TypeScript 타입 지원.
- Prisma Studio: 웹 UI로 데이터 조회/수정 기능 내장.
- 마이그레이션: `prisma migrate dev`로 스키마 변경 이력 관리.
- 문서 방대, 예제 풍부, 커뮤니티 크다.
- SQLite, PostgreSQL 모두 지원 → 나중에 DB 마이그레이션 시 코드 재사용.

**단점**
- 번들 크기: Prisma Client 엔진이 무거움 (Edge 환경 이슈 — App Router Edge Runtime 비호환).
- 쿼리 추상화: 복잡한 JOIN, 윈도우 함수 등에서 Raw SQL 필요.
- 세대 전환 비용: schema 변경 후 `generate` 실행 필수 (개발 흐름 끊김).
- SQLite 지원이 PostgreSQL 대비 일부 제한 (예: 열 삭제 제한).

### 후보 2: Drizzle ORM

**개요**
- "SQL을 알면 Drizzle을 안다"는 철학. TypeScript 코드로 스키마 정의.
- 최근 빠른 성장세, Next.js 생태계에서 인기 상승 중.

**장점**
- 경량: 런타임 의존성 최소. 번들 크기 작음.
- SQL-like API: `db.select().from(users).where(eq(users.id, 1))` — SQL을 알면 직관적.
- Edge Runtime 호환: Next.js App Router Edge에서도 동작.
- 마이그레이션: `drizzle-kit generate` → SQL 마이그레이션 파일 생성.
- Turso, SQLite, PostgreSQL, MySQL 지원.
- 타입 추론이 강력 — 스키마에서 타입 자동 추출.

**단점**
- Prisma 대비 문서/예제 수가 적음 (빠르게 따라잡는 중).
- ORM 패턴보다 쿼리빌더에 가까워 ORM 철학 기대 시 이질감.
- Studio (데이터 UI 도구) 없음 (하지만 이 프로젝트가 직접 만들 것이므로 오히려 무관).

### 후보 3: Kysely

**개요**
- 타입 안전 SQL 쿼리빌더. ORM이 아니라 "타입 안전 SQL 작성 도우미".
- Kanel-Kysely로 기존 DB에서 타입 자동 생성 가능.

**장점**
- SQL에 가장 가까운 API — SQL 전문가에게 자연스러움.
- 타입 안전 보장 수준이 매우 높음.
- 런타임 오버헤드 거의 없음.
- PostgreSQL 지원 가장 성숙.

**단점**
- 마이그레이션 도구 없음 — 별도 관리 필요.
- SQLite 지원은 있지만 커뮤니티 예제 드문 편.
- 스키마 퍼스트가 아니라 코드 작성량이 많음.
- 러닝커브: SQL을 잘 모르면 오히려 어려움.

### 후보 4: better-sqlite3 직접 사용

**개요**
- ORM/쿼리빌더 없이 Raw SQL을 better-sqlite3 API로 직접 실행.

**장점**
- 의존성 최소화.
- 완전한 SQL 제어 — 최적화 여지 최대.
- 동기 API라 코드가 단순 (async/await 없음).

**단점**
- 타입 안전 없음 — 쿼리 결과가 `any` 또는 수동 타입 캐스팅.
- 마이그레이션 수동 관리.
- 유지보수: 테이블명/컬럼명 변경 시 모든 쿼리 직접 찾아서 수정.
- SQL 문자열 오타가 런타임에서만 발견됨.

|--------------|---------|--------|--------|---------|
| 성능 (15%) | 5 | 4 | 5 | 5 |
| 운영 편의 (30%) | 5 | 4 | 3 | 2 |
| 기능 (20%) | 4 | 5 | 4 | 5 |
| 생태계 (15%) | 4 | 5 | 3 | 5 |
| 1인 적합도 (20%) | 5 | 4 | 3 | 2 |

**가중 점수 계산**

| 후보 | 최종 점수 |
|------|---------|
| Drizzle | 5×0.15 + 5×0.30 + 4×0.20 + 4×0.15 + 5×0.20 = **4.65** |
| Prisma | 4×0.15 + 4×0.30 + 5×0.20 + 5×0.15 + 4×0.20 = **4.35** |
| Kysely | 5×0.15 + 3×0.30 + 4×0.20 + 3×0.15 + 3×0.20 = **3.50** |
| Raw SQL | 5×0.15 + 2×0.30 + 5×0.20 + 5×0.15 + 2×0.20 = **3.50** |

### 카테고리 2 최종 추천

> **Drizzle ORM** — 4.65점

**근거**
1. SQLite 추천과 자연스럽게 연결 — Drizzle은 SQLite + better-sqlite3 조합에 최적화.
2. Edge Runtime 호환성 — Next.js App Router에서 완벽 동작.
3. 마이그레이션 파일 자동 생성으로 스키마 이력 관리 가능.
4. 번들 경량 — WSL2 서버에서 메모리 부담 최소화.
5. SQL-like API는 대시보드 데이터 쿼리 작성 시 이해하기 쉬움.

## 카테고리 3: UI 컴포넌트 라이브러리

### 배경

현재 UI는 Tailwind CSS 4를 이용해 직접 구현한 컴포넌트를 사용한다. 플랫폼 진화 시 테이블, 모달, 토스트, 탭, 드롭다운 등 다양한 컴포넌트가 필요해진다. 직접 구현을 계속할 것인가, 외부 라이브러리를 채택할 것인가.

### 후보 1: shadcn/ui

**개요**
- Radix UI 프리미티브 위에 Tailwind CSS로 스타일링한 컴포넌트 컬렉션.
- "설치"가 아니라 "복사해서 소유" 방식 — 컴포넌트 소스가 내 프로젝트에 직접 추가됨.

**장점**
- Tailwind CSS와 완벽 통합 — 현재 스택과 충돌 없음.
- 커스터마이징 자유도 최대 — 소스 직접 수정 가능.
- Next.js App Router 공식 지원. Server Component, Client Component 모두 명확히 표시.
- 다크 테마 기본 지원 — CSS 변수 기반 테마 시스템.
- Supabase Studio 자체도 Radix 기반 컴포넌트 사용.
- 의존성이 내 프로젝트 코드 — 버전 충돌, 브레이킹 체인지 없음.
- Button, Input, Table, Dialog, Select, Toast 등 대시보드에 필요한 컴포넌트 전부 있음.

**단점**
- 컴포넌트별 개별 추가 (CLI로 `npx shadcn add button`).
- 처음 세팅 시 설정 파일 작성 필요 (components.json).
- 직접 구현과 달리 Radix 패턴 학습 필요.

### 후보 2: Radix Themes

**개요**
- Radix UI의 완성형 디자인 시스템. 프리미티브보다 한 단계 위의 완성된 컴포넌트.

**장점**
- 설치 후 바로 사용 가능한 완성형.
- 접근성(a11y) 최고 수준.
- 일관된 디자인 언어 자동 적용.

**단점**
- Tailwind CSS와 통합이 자연스럽지 않음 — Tailwind 유틸리티 클래스 우선순위 충돌.
- 현재 다크 테마/다크 배경 스타일과 맞지 않을 가능성.
- 커스터마이징 제한 — Radix Themes 디자인 언어에 종속.
- Tailwind CSS 4와 충돌 가능성 높음.

### 후보 3: Headless UI (Tailwind Labs 공식)

**개요**
- Tailwind Labs에서 만든 접근성 우선 헤드리스 컴포넌트.
- 스타일은 완전히 직접 작성, 동작(접근성, 키보드 네비게이션)만 제공.

**장점**

## 카테고리 5: 코드 에디터

### 배경

Supabase Studio의 SQL Editor는 이 프로젝트가 지향하는 핵심 기능 중 하나다. 또한 환경변수 편집기, JSON 설정 편집기 등에도 코드 에디터가 필요하다. 웹 기반 코드 에디터를 어떻게 구현할 것인가.

**개요**

- @monaco-editor/react 패키지로 React에 통합.

**장점**
- SQL 자동완성 및 구문 강조 최고 수준.

- 멀티커서, 단어 래핑, 미니맵, 폴딩 등 VS Code 수준 기능.

**단점**
- 번들 크기 매우 큼: gzip 후 ~2MB. lazy import 필수.
- 메모리 사용량 많음.

- VS Code 경험자에게 UX 차이 느껴짐.

## 카테고리 6: 실시간 통신

### 배경

현재 대시보드는 실시간 업데이트가 없다 (새로고침 필요). 플랫폼 진화 시 다음 기능에 실시간 통신이 필요하다.
- CPU/메모리 메트릭 자동 갱신
- PM2 프로세스 상태 변경 알림
- 감사 로그 스트리밍
- 명령 실행 결과 실시간 출력

### 후보 1: SSE (Server-Sent Events)

**개요**
- HTTP 기반 단방향 스트림. 서버 → 클라이언트 이벤트 전송.
- Next.js App Router에서 `Response` + `ReadableStream`으로 구현.

**장점**
- HTTP 표준 — 추가 인프라 없음. Next.js API Route에서 그대로 동작.
- Cloudflare Tunnel 통과 — WebSocket보다 호환성 높음.
- 자동 재연결 내장 (EventSource API).
- 단순: EventSource API로 클라이언트 구현.
- 대시보드 메트릭 스트리밍에 완벽히 적합 (서버 → 클라이언트 단방향).

**단점**
- 단방향만 지원 (클라이언트 → 서버는 별도 HTTP 요청 필요).
- HTTP/1.1에서 도메인당 최대 6개 연결 제한 (HTTP/2에서 해소).
- 바이너리 데이터 비효율 (텍스트 기반).

### 후보 2: WebSocket (ws 라이브러리)

**개요**
- 전이중(full-duplex) 양방향 통신. ws 라이브러리로 Node.js 서버 구현.
- Next.js에서 사용하려면 Custom Server 필요 (App Router 기본 미지원).

**장점**
- 양방향 통신 — 클라이언트에서 서버로도 실시간 메시지 전송 가능.
- 낮은 지연 시간.
- 바이너리 데이터 효율적 전송.

**단점**
- Next.js App Router에서 기본 지원 안 됨 → Custom Server 필요 (server.js 추가, Next.js 최적화 일부 포기).
- Cloudflare Tunnel WebSocket 지원: 가능하지만 설정 필요.
- 양방향이 필요한 케이스가 현재 대시보드에서는 제한적.
- PM2로 재시작 시 모든 WebSocket 연결 끊김 → 클라이언트 재연결 로직 필수.

### 후보 3: Socket.io

**개요**
- WebSocket 위에 추상화 계층 추가. 룸, 이벤트, 자동 재연결, 폴백(long-polling) 지원.

**장점**
- 자동 재연결 + WebSocket 불가 시 long-polling 폴백.
- 룸/네임스페이스로 논리적 채널 분리.
- 사용하기 쉬운 API.

**단점**
- 번들 크기: 클라이언트 ~30KB.
- Next.js App Router와 통합이 복잡 (Custom Server 필요).
- SSE 대비 설정 복잡도 높음.
- 현재 필요한 기능(단방향 스트림)에 오버스펙.
- Socket.io 서버를 별도 포트에서 운영해야 하는 경우도 있음.

| 기능 (20%) | 3 | 5 | 5 |
| 생태계 (15%) | 4 | 4 | 5 |
| 1인 적합도 (20%) | 5 | 3 | 3 |

**가중 점수**

| 후보 | 최종 점수 |
|------|---------|
| SSE | 4×0.15 + 5×0.30 + 3×0.20 + 4×0.15 + 5×0.20 = **4.30** |
| WebSocket | 5×0.15 + 3×0.30 + 5×0.20 + 4×0.15 + 3×0.20 = **3.75** |
| Socket.io | 4×0.15 + 3×0.30 + 5×0.20 + 5×0.15 + 3×0.20 = **3.80** |

### 카테고리 6 최종 추천

> **SSE (Server-Sent Events)** — 4.30점

**근거**
1. 메트릭 스트리밍, 로그 스트리밍은 모두 서버 → 클라이언트 단방향 — SSE로 완벽 해결.
2. Next.js App Router Route Handler에서 Custom Server 없이 바로 구현 가능.
3. Cloudflare Tunnel과의 호환성이 WebSocket보다 안정적.
4. 클라이언트에서 서버로 명령 전송(PM2 재시작 등)은 기존 REST API로 충분 — 양방향 WebSocket 불필요.
5. HTTP/2 환경(Cloudflare Tunnel)에서 다중 SSE 스트림 동시 유지 가능.

## 전체 추천 스택 요약표

| 카테고리 | 추천 | 차선택 | 탈락 | 핵심 근거 |
|---------|------|--------|------|---------|
| **데이터베이스** | SQLite (better-sqlite3) | PostgreSQL | LowDB | 제로 운영, 파일 백업, WSL2 최적 |
| **ORM** | Drizzle ORM | Prisma | Raw SQL | 경량, Edge 호환, SQL-like API |
| **UI 컴포넌트** | shadcn/ui | Headless UI | Radix Themes | Tailwind 통합, 소유 모델, 다크 테마 |
| **테이블** | TanStack Table | AG Grid CE | react-data-grid | 헤드리스+타입 안전, shadcn 통합 |
| **코드 에디터** | Monaco (SQL) + react-simple (설정) | CodeMirror 6 | - | 용도 분리로 번들 최적화 |
| **실시간 통신** | SSE | - | Socket.io | 단방향 스트림, Next.js 기본 지원, Cloudflare 호환 |

### 최종 추천 스택 전체 구성

```
데이터 계층
├── SQLite (better-sqlite3)         ← 영속 데이터 저장
├── Drizzle ORM + drizzle-kit       ← 타입 안전 쿼리 + 마이그레이션
└── In-memory (기존 유지)           ← 실시간 메트릭 (histrogram은 SQLite)

UI 계층
├── Tailwind CSS 4 (기존)           ← 기반 스타일
├── shadcn/ui                       ← 컴포넌트 (Button, Dialog, Table, Toast 등)
├── TanStack Table                  ← 데이터 테이블 (Table Editor, 감사 로그)
├── Monaco Editor (lazy)            ← SQL Editor, JSON 편집기
└── react-simple-code-editor        ← 환경변수 편집기 (경량)

실시간 계층
└── SSE (Next.js Route Handler)     ← 메트릭/로그 스트리밍

기존 유지
├── Next.js 16 App Router           ← 프레임워크
├── TypeScript                      ← 타입 시스템
└── Zod                             ← 입력 검증 (Phase 11에서 적용)
```

### 단계별 도입 순서

**Phase 11 (현재 → 즉시): Zod 입력 검증**
- 추가 패키지 없음 (이미 결정됨)

**Phase 12 (다음): 데이터 영속성 기반**
1. `better-sqlite3` + `drizzle-orm` + `drizzle-kit` 설치
2. 감사 로그 SQLite 마이그레이션 (현재 in-memory → DB)
3. 메트릭 히스토리 30분 보존 (DB)

**Phase 13: UI 컴포넌트 강화**
1. shadcn/ui 설치 및 기본 컴포넌트 추가 (Button, Dialog, Table, Toast)
2. 기존 직접 구현 컴포넌트를 shadcn/ui로 점진적 교체

**Phase 14: 실시간 강화**
1. SSE Route Handler 구현
2. 메트릭 카드 자동 갱신 (5초 간격)
3. 로그 실시간 스트리밍

**Phase 15: Table Editor + SQL Editor**
1. TanStack Table로 DB 테이블 뷰어 구현
2. Monaco Editor lazy import
3. SQL 쿼리 실행 API
